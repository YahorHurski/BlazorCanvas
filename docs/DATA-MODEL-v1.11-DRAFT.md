**Основано на:** обсуждении 2026-07-21 + документе «Модели данных для фигур»

## Зачем меняем

Сегодня фигура хранится как `(type, x1, y1, x2, y2)` — четыре целых числа, которые ВСЕГДА
являются габаритом. Форма при этом существует только в коде: `FigureShape.razor` вычисляет
вершины треугольника по формуле от габарита.

Это упирается в жёсткие потолки, каждый из которых блокирует заявленную функциональность:

| Чего нельзя | Почему |
|---|---|
| Подвинуть вершину треугольника | у фигуры 4 степени свободы, свободному треугольнику нужно 6 |
| Треугольник вниз / влево / вправо | форма — формула в коде, а не данные |
| Многоугольник с 1000 вершин | нужно 2000 чисел, есть 4 колонки |
| Повернуть фигуру | угол хранить негде |
| «На передний план» | порядок слоёв — это `id`, менять его нельзя |
| Undo удаления | новая строка получает новый `id`, слой и ссылки теряются |
| Цвет и толщина у фигуры | колонок нет |
| Зум выше 100% | координаты целые |
| Больше ~10 000 фигур | нельзя спросить «что видно на экране» |

**Ключевая проблема:** каждая новая возможность требует менять схему. Цель этой миграции —
чтобы она была **последней, которая трогает уже записанные данные**.

## Принцип

> **Положение фигуры и форма фигуры — разные вещи, и хранятся раздельно.**

```
x, y, rotation   → ГДЕ фигура стоит       (обычные колонки)
geometry jsonb   → КАКОЙ она формы,       в ЛОКАЛЬНЫХ координатах от (0,0)
```

Отсюда всё остальное. Перемещение — это `update figures set x = x + 20`, при любой сложности
фигуры: `geometry` не читается и не пишется. Та «дженерик-операция без разбора типа», ради
которой старая модель держалась за габарит, здесь получается сама и работает даже для 1000
вершин.

**Правило раскладки полей** (применяется везде, один раз сформулировано):

| Данные | Куда | Почему |
|---|---|---|
| Плоское, общее для всех фигур (положение, порядок, время) | **колонки** | база проверяет и индексирует, EF мапит типизированно |
| Разное по типам или со структурой внутри (форма, стиль) | **jsonb** | новый тип фигуры не требует новых полей |

## Схема

Четыре таблицы: `users` (без изменений), `canvases`, `figures`, `figure_types`.

```sql
-- ─────────────────────────────────────────────────────────────
-- Холст перестаёт быть WHERE-условием и становится сущностью.
-- Интерфейса создания новых холстов в v1.11 НЕТ: заводится ровно
-- один холст на пользователя. Таблица нужна, чтобы позже эта фича
-- не требовала прохода по всем строкам figures.
-- ─────────────────────────────────────────────────────────────
create table canvases (
    id         uuid        primary key,
    owner_id   integer     not null references users(id) on delete cascade,
    name       text        not null default 'Canvas',
    width      integer     not null default 1472,      -- значения v1.1
    height     integer     not null default 828,
    background text        not null default '#FFFFFF',
    created_at timestamptz not null default now()
);

create index ix_canvases_owner on canvases (owner_id);

-- ─────────────────────────────────────────────────────────────
-- Справочник допустимых типов. Список типов становится ДАННЫМИ,
-- а не частью устройства таблицы. Новый тип фигуры = INSERT,
-- а не ALTER TABLE. Сегодня 4 строки.
-- ─────────────────────────────────────────────────────────────
create table figure_types (
    name text primary key
);

insert into figure_types (name)
values ('line'), ('rectangle'), ('circle'), ('triangle');

-- ─────────────────────────────────────────────────────────────
create table figures (
    id        uuid    primary key,
    canvas_id uuid    not null references canvases(id)    on delete cascade,
    type      text    not null references figure_types(name),

    -- ПОЛОЖЕНИЕ: перемещение трогает только эти колонки
    x        numeric(12,3) not null,
    y        numeric(12,3) not null,
    rotation numeric(7,3)  not null default 0,

    -- ФОРМА: локальные координаты от (0,0). Источник истины.
    geometry jsonb not null,

    -- СТИЛЬ: проходит через валидатор C# перед записью
    style jsonb not null default '{}',

    -- ПОРЯДОК ОТРИСОВКИ: отвязан от id, уникален в пределах холста
    z numeric not null,

    -- ГАБАРИТ: производный кэш, чистая функция от geometry.
    -- НЕ включает обводку — см. «Габарит и обводка».
    -- Нужен клэмпу у края холста и выделению рамкой.
    -- Поддерживается приложением — см. «Почему не generated column».
    bbox_x double precision not null,
    bbox_y double precision not null,
    bbox_w double precision not null,
    bbox_h double precision not null,

    created_at timestamptz not null default now(),

    constraint z_unique_per_canvas unique (canvas_id, z),
    constraint style_is_object     check (jsonb_typeof(style) = 'object'),
    constraint geometry_is_object  check (jsonb_typeof(geometry) = 'object'),
    constraint bbox_is_positive    check (bbox_w >= 0 and bbox_h >= 0)
);

create index ix_figures_canvas_z on figures (canvas_id, z);
```

## Форматы геометрии

Локальные координаты, начало отсчёта — `(0,0)`, положение задаётся колонками `x, y`.

```json
line          {"points": [[0,0],[100,40]]}
rectangle     {"w": 200, "h": 100}
circle        {"r": 50}
triangle      {"points": [[50,0],[0,80],[100,80]]}
```

Будущие типы вписываются без единого изменения схемы:

```json
ellipse       {"rx": 80, "ry": 40}
многоугольник {"points": [[...], ... сколько угодно пар ...]}
правильный    {"n": 1000, "rx": 60, "ry": 60}
звезда        {"n": 5, "rx": 60, "ry": 60, "inner": 0.38}
стрелка       {"w": 100, "h": 40, "head": 0.3}
перо          {"points": [[...]]}
кривая безье  {"segments": [{"c1": [..], "c2": [..], "to": [..]}]}
```

### Почему линия и треугольник — списки точек, а прямоугольник и круг — нет

Правило: **форма хранится в самом общем представлении, которое понадобится ЭТОМУ ЖЕ типу.**

- **Треугольник → точки.** Планируется правка вершин. Если хранить `{"w","h"}` и считать
  вершины формулой, то в день появления этой фичи придётся конвертировать все существующие
  треугольники. С точками — ноль изменений данных.
- **Линия → точки.** Побочно чинит ландмайн №2 текущего ADR: сегодня линия — единственная
  фигура, у которой колонки не габарит, а реальные концы, из-за чего нормализация у неё
  особая (D-41), и ошибка в ней превращает диагональ в противоположную **молча**. С точками
  габарит становится честным габаритом для всех фигур без исключения, а опасная ветка в
  `Normalisation` исчезает.
- **Круг → `{"r"}`, а не `{"rx","ry"}`.** Круг никогда не станет эллипсом — эллипс будет
  ОТДЕЛЬНЫМ типом с отдельной кнопкой. То, что круг не растягивается, — не потеря
  функциональности, а определение круга, и так приложение работает уже сегодня (D-13).
- **Прямоугольник → `{"w","h"}`.** Прямоугольник останется прямоугольником; скруглённые углы,
  если появятся, добавляются ключом `"rx"` без конверсии старых строк.

**Важное ограничение, которое надо помнить:** jsonb убирает миграции *схемы*, но не миграции
*формата*. Смена формата уже записанного типа = проход по всем строкам. Отсюда две обязательные
привычки: формат типа продумывается до первой записи, а чтение всегда защитное — отсутствующий
ключ подставляет умолчание, а не роняет отрисовку.

## Стиль и проверка входных данных

Стиль — jsonb, но **в базу никогда не попадает то, что прислал клиент**. Присланный JSON
разбирается в типизированную запись C#, проверяется, и сериализуется обратно уже из неё.
Всё, что не легло в известные поля, до базы не доезжает.

```csharp
public sealed record FigureStyle(
    string Stroke      = "#000000",
    double StrokeWidth = 2,
    string Fill        = "#FFFFFF",
    double Opacity     = 1)
{
    private static readonly Regex Colour = new(@"^#[0-9A-Fa-f]{6}$");

    public FigureStyle Sanitised() => new(
        Colour.IsMatch(Stroke) ? Stroke : "#000000",
        Math.Clamp(StrokeWidth, 0.5, 64),
        Colour.IsMatch(Fill)   ? Fill   : "#FFFFFF",
        Math.Clamp(Opacity, 0, 1));
}
```

**Диапазон толщины обводки: `0.5 – 64`.** Нижняя граница — самая тонкая линия, которая ещё
видна глазом: в SVG толщина `0` не рисуется вообще, а `0.25` на обычном экране размывается почти
в невидимость. Верхняя граница держит обводку обводкой — см. следующий раздел.

**Почему белый список цвета, а не просто экранирование.** Стиль в итоге попадает в SVG-атрибуты.
Blazor экранирует значения атрибутов, и базовый случай этим закрыт — но достаточно один раз
отрендерить стиль через `MarkupString` или собрать строку `style="..."`, и защита исчезает.
Регулярка `^#[0-9A-Fa-f]{6}$` закрывает вопрос насовсем: там, где разрешены только
шестнадцатеричные цифры, инъекции не бывает.

**Тот же шлюз обязателен для `geometry`** — она опаснее, потому что оттуда координаты идут в
клэмп. В проекте это уже осознано: `DrawGesture` помечен границей доверия, координаты из
браузера считаются враждебными. Теперь такой шлюз должен быть **один на всё приложение**, а не
по копии в каждом месте записи.

## Габарит и обводка

**`bbox_*` — чистая функция от `geometry`. Обводка в него не входит.**

Обводка в SVG рисуется по центру контура: при толщине `W` она выходит на `W/2` за пределы формы
во все стороны. Значит визуальный размер фигуры всегда чуть больше её геометрии, и надо было
решить, что считать габаритом.

**Решение: габарит геометрический.** Иначе `bbox` оказался бы завязан на стиль — смена толщины
обводки требовала бы его пересчёта, а тест «пересчитать из geometry и сверить» перестал бы быть
однозначным. Там, где нужен визуальный размер (клэмп у края, рамка выделения), `W/2`
прибавляется при отрисовке.

**Отсюда верхняя граница толщины — 64.** При ней расхождение между записанным габаритом и
видимым размером не превышает 32 пикселей по каждой стороне, чего глаз не замечает. Если бы
максимумом была меньшая сторона холста (828), прямоугольник 10×10 визуально закрыл бы весь
холст, оставаясь по данным фигурой 10×10: клэмп считал бы его «внутри холста», а рамка выделения
обрисовывала бы область 10×10 посреди сплошной заливки.

Решение обратимое: это зажим в валидаторе C#, менять его можно одной строкой без миграции.

## Реестр форм — как добавляется новая фигура

Вся типоспецифичная логика собирается в один интерфейс. Сегодня она размазана по восьми местам
(`FigureType`, `FigureTypeNames.ToDbValue`, `Parse`, `Normalisation`, `MinSizeGuard`,
`DrawGesture`, `switch` в `FigureShape.razor`, whitelist-CHECK).

```csharp
public interface IShapeDefinition
{
    string Name { get; }                                  // 'hexagon'
    object  ParseGeometry(JsonElement json);              // + валидация
    bool    IsDrawable(object geometry);
    Bbox    BoundsOf(object geometry);                    // источник bbox-колонок
    object  FromGesture(Point press, Point cursor);       // как рисуется
    object  Resize(object geometry, double sx, double sy);
}
```

**Добавление 50-й фигуры после этой миграции:**

1. один класс в C#;
2. одна строка в `figure_types` (заполняется при старте приложения);
3. одна кнопка в тулбаре.

Ноль изменений схемы. Ноль миграций. Ноль трогания существующих данных.

## Почему bbox — обычные колонки, а не generated column

⚠️ **Это изменение относительно того, что обсуждалось в переписке.** Там предлагалось поручить
пересчёт габарита самой базе (`generated always as ... stored`), чтобы приложение не могло его
забыть.

**От этого варианта надо отказаться.** Чтобы база считала габарит из `geometry`, ей нужна
IMMUTABLE-функция, разбирающая формат каждого типа фигуры, — то есть **вся логика форм,
продублированная на PL/pgSQL**. Последствия:

1. Добавление новой фигуры потребовало бы правки функции в базе, то есть **миграции** — ровно
   то, что эта переделка устраняет. Обещание «новая фигура = только C#» было бы нарушено.
2. Проект уже обжигался на паре «одна и та же логика в двух местах, обязана совпадать»:
   `MinSizeGuard` против CHECK-констрейнтов (D-50), их совпадение приходится доказывать тестом
   на матрице из 32 случаев.

**Решение:** bbox считает приложение, в **одном** месте — `IShapeDefinition.BoundsOf`, вызываемом
из единственного пути сохранения.

Риск протухания кэша при этом невелик, и вот почему: **габарит читается клэмпом при каждом
перетаскивании** (D-24/D-36 — фигура не должна уходить за край холста). Неверный bbox проявится
сразу и заметно — фигура начнёт упираться в край не там, где нарисована. Это не тихий отказ.
Плюс тест: пройти по всем строкам, пересчитать габарит из `geometry`, сравнить с сохранённым.
Инфраструктура для этого уже есть (тесты против живого контейнера).

## Миграция существующих данных

**Требование:** ни одна фигура не теряется и ни одна не меняет вид на экране.

Все четыре существующих типа конвертируются точными формулами:

| Тип | Положение `x, y` | `geometry` |
|---|---|---|
| `rectangle` | `x1, y1` | `{"w": x2-x1, "h": y2-y1}` |
| `circle` | `x1, y1` | `{"r": (x2-x1)/2}` |
| `triangle` | `x1, y1` | `{"points": [[(x2-x1)/2, 0], [0, y2-y1], [x2-x1, y2-y1]]}` |
| `line` | `x1, min(y1,y2)` | `{"points": [[0, y1-y], [x2-x1, y2-y]]}` |

Формула треугольника — буквальная транскрипция текущего рендерера (вершина сверху по центру,
основание снизу). Линия сохраняет направление диагонали точно: обе точки переносятся как есть,
относительно нового начала отсчёта.

**Порядок шагов:**

1. `canvases` — по одному холсту на каждого существующего пользователя, `1472×828`.
2. `figure_types` — четыре строки.
3. Новая `figures`. Для каждой старой строки:
   - `id` — новый uuid;
   - `canvas_id` — холст владельца;
   - `x, y, geometry` — по таблице выше;
   - **`z` = старый `id`** — текущий порядок слоёв сохраняется побитово (сегодня z-order это и
     есть `id`, D-39). Уникальность выполняется сама собой: старые `id` уникальны глобально, а
     холст у каждого пользователя один;
   - `style` — текущий фиксированный стиль (`#000000` / 2 / `#FFFFFF` / 1), поэтому вид не
     меняется;
   - `bbox_*` — считается при конверсии;
   - `rotation` = 0.
4. Старая таблица удаляется.

**Оговорка о честности данных:** `created_at` у существующих фигур будет временем миграции, а
не реальным временем создания — старая схема этого не хранила (D-46 удалил колонку). Это
задокументированная неточность, и она единственная.

**Проверка:** тест, который заливает дамп v1.1, прогоняет миграцию и сверяет каждую фигуру —
координаты вершин после рендера и порядок слоёв. Тесты против живого контейнера в проекте уже
есть.

## Как будет происходить отрисовка

**Было** — рисуем прямоугольник, нажали (100,100), отпустили (300,200):

```
INSERT (user_id, 'rectangle', x1=100, y1=100, x2=300, y2=200)
<rect x="100" y="100" width="200" height="100" />
```

**Стало:**

```
id = Guid.NewGuid()                                    ← до обращения к базе
INSERT (id, canvas_id, 'rectangle',
        x=100, y=100, rotation=0,
        geometry='{"w":200,"h":100}',
        z=max(z)+1, style=<умолчания>, bbox=...)

<g transform="translate(100,100)">
  <rect width="200" height="100" />
</g>
```

Главное изменение в рендерере: **фигура заворачивается в `<g transform="translate(x,y) rotate(...)">`,
а форма рисуется в локальных координатах от нуля.**

Треугольник вниз — тот же тип, никаких новых формул:

```
geometry = {"points": [[0,0],[100,0],[50,80]]}
<g transform="translate(240,160)"><polygon points="0,0 100,0 50,80" /></g>
```

Перетаскивание на 20 пикселей вправо, при любой сложности фигуры:

```sql
update figures set x = x + 20 where id = @id;
```

**Для пользователя не меняется ничего.** Те же фигуры, те же места, тот же вид. Рисование
по-прежнему видно в других вкладках только результатом, на отпускании (вариант А, D-35/D-39
сохраняются).

## Что сознательно НЕ берём сейчас

Критерий: берём сейчас только то, что иначе потребует **прохода по уже записанным строкам**.
Всё остальное добавляется потом мгновенно и с честными значениями
(`ADD COLUMN ... DEFAULT` в PostgreSQL 11+ таблицу не переписывает).

| Что | Когда понадобится | Цена добавления потом |
|---|---|---|
| `version` (оптимистичная блокировка) | совместное редактирование | `ADD COLUMN DEFAULT 1` — значение `1` честное |
| `parent_id` (группы и слои) | группировка фигур | `ADD COLUMN` — у старых строк просто NULL |
| GiST-индекс по габариту | 100 000+ фигур | `CREATE INDEX` — данные не трогаются |
| `figure_history` / журнал операций | undo/redo | `CREATE TABLE` — стартует пустой |
| Мягкое удаление (`deleted_at`) | — | не берём: undo решается историей, а `deleted_at` не умеет отменять перемещение и правку вершин, зато ломает правило D-40 «no resurrection» |

**Взяли, хотя формально можно было отложить:** колонки времени и стиль — одна строка в
миграции, которую мы и так пишем, зато следующий милестоун становится чистым UI без правки
модели.

---

## Принимаемые издержки

Честный список того, что теряется:

1. **База больше не проверяет геометрию.** Уходят `circle_is_a_circle`, `box_is_a_box`,
   `line_is_a_line`. `{"r": -5}` запишется молча. Вся геометрическая корректность переезжает
   из «невозможно» в «код не ошибся» — отсюда требование единого шлюза валидации.
2. **Смена формата уже записанного типа = миграция данных.** jsonb снимает миграции схемы, но
   не эту. Отсюда правило «формат продумывается до первой записи».
3. **bbox — кэш, который теоретически может разойтись с геометрией.** Смягчено тем, что его
   читает клэмп на каждом перетаскивании, плюс тестом.
4. **Строка стала шире** — jsonb вместо четырёх интов. На порядок больше на фигуру; при
   реалистичных объёмах несущественно.
5. **Уходит целочисленность координат (D-20).** Она давала точные сравнения в CHECK; после
   перехода это уже не нужно, потому что круг хранится одним радиусом и круглый по построению.

**Потолок, который эта модель НЕ снимает:** при сотнях тысяч фигур узкое место — не база, а
Blazor Server и SVG DOM в браузере. Каждая фигура это DOM-элемент, который диффится и уходит по
SignalR. bbox-колонки делают возможным viewport-куллинг (грузить только видимое) — это
необходимое условие, но при миллионе фигур понадобится ещё и отказ от SVG-элементов в пользу
canvas/WebGL. Схема этому не помешает, но и не решит.

## Порядок слоёв: `z` уникален

`z` определяет **только порядок отрисовки** — кто окажется сверху при перекрытии. К перемещению
он отношения не имеет: фигуры двигаются независимо, каждая своей парой `x, y`.

**Решение: `unique (canvas_id, z)`.** Совпадение `z` не смешивало бы цвета — SVG рисует в
порядке документа, одна фигура просто легла бы поверх другой непрозрачно, — но какая именно
окажется сверху, было бы непредсказуемо. Строгий полный порядок это исключает.

**Следствие, которое надо реализовать:** новая фигура получает `z = max(z) + 1`. Если две
вкладки создают фигуру одновременно, обе посчитают одно и то же число, и второй INSERT упадёт с
нарушением уникальности. **Нужен повтор с пересчётом** — без него фигура иногда молча не
нарисуется. Ситуация редкая, код небольшой, но обязательный.

Перестановка слоёв уникальность не нарушает: вставка между соседями даёт `1.5`, затем `1.25` и
так далее — `numeric` в PostgreSQL точный, поэтому делить пополам можно сколько угодно раз.

## Открытые вопросы

Нет. Все вопросы этапа проектирования закрыты:

- Круг при непропорциональном ресайзе **остаётся кругом** (берётся одна сторона).
