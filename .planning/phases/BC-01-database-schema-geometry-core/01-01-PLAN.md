  <action>
    Create `docker-compose.yml` at the repository root with a single PostgreSQL service, per D-27.

    Pinned values â€” these are locked, do not substitute:
    - image `postgres:17` (not `postgres:latest`, not `alpine` â€” the ADR pins the major version)
    - host port `5432` mapped to container port `5432`
    - `POSTGRES_DB` = `canvas`
    - `POSTGRES_USER` = `postgres`
    - `POSTGRES_PASSWORD` = `postgres`
    - a **NAMED** volume (name it `canvas-pgdata`) mounted at `/var/lib/postgresql/data`, declared under a top-level `volumes:` key. An anonymous or bind-mount volume does NOT satisfy D-27 â€” the whole point is that figures survive `docker compose down`.

    Discretionary (yours to choose, per CONTEXT.md "Claude's Discretion"): service name, container name, `restart` policy, and the healthcheck. Add a healthcheck using `pg_isready -U postgres -d canvas` with a short interval â€” plan 01-03 needs a reliable "is the DB up yet" signal at app startup.

    Bind the host port publication to the loopback interface (`127.0.0.1:5432:5432`). This still publishes port 5432 exactly as D-27 requires; it just declines to expose a postgres/postgres superuser to the local network. See `<threat_model>` T-BC01-04.

    Also create a root `.gitignore` for .NET: ignore `bin/`, `obj/`, `.vs/`, `*.user`. Do not ignore `appsettings.Development.json` â€” it holds no real secret (D-08) and plan 01-03 reads it.
  </action>
