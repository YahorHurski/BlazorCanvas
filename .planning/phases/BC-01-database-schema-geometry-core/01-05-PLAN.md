<objective>
Close the two Critical geometry-clamp defects (01-REVIEW.md CR-01 and CR-02, re-confirmed in 01-VERIFICATION.md — one reproduced live against the database) that let the clamp/circle-encoding maths silently emit an off-canvas or teleported figure. Both defects sit in the exact functions TEST-01 exists to prove sound, in the input region (negative / out-of-canvas / oversized coordinates) that the existing 145-test suite never probes.

Purpose: The phase goal is "the three silent failure modes are guarded before any UI exists to hide them." Two of those guards are currently defeatable by ordinary off-canvas pointer input — which BC-02's drag handler will produce the moment it wires up raw pointer coordinates. This plan makes the two clamp primitives robust so the guarantee holds universally, not just for the non-adversarial coordinate range the current tests cover.

Output: A hardened `Movement.ClampDelta` (degenerate `lo > hi` box does not move), a hardened `CircleEncoding.ClampDrawRadius` (centre clamped into the canvas and radius floored at zero), and two named regression tests that fail on the current code and pass on the fix.

Scope fence: The fix is in the C# clamp maths ONLY. Do NOT add a canvas-bounds CHECK constraint — D-36 deliberately has none, and 01-CONTEXT.md's scope fence lists a bounds CHECK as out of bounds. The correct fix is making the clamp arithmetic robust, not adding a database constraint.
</objective>

  <name>Task 1: Guard ClampDelta against the inverted lo &gt; hi case (CR-02)</name>
  <files>src/BlazorCanvas/Geometry/Movement.cs, tests/BlazorCanvas.Tests/Geometry/ClampTests.cs</files>
  <read_first>
    - src/BlazorCanvas/Geometry/Movement.cs (the function being changed — ClampDelta on line 10, ClampMove on lines 12-28)

    - tests/BlazorCanvas.Tests/Geometry/ClampTests.cs (existing test patterns and the ShapeInvarianceCases member data — the new tests belong alongside these)
    - 01-REVIEW.md CR-02 (lines 116-136): the concrete before/after and the worked Box(0,0,2000,100) example
  </read_first>
  <action>
    Change `Movement.ClampDelta(int v, int lo, int hi)` so that when `lo` is greater than `hi` it returns 0, otherwise it keeps the existing `Math.Min(Math.Max(v, lo), hi)` behaviour. Rationale (per CR-02): `lo` exceeds `hi` exactly when a box is wider than 1280 or taller than 720 (because `lo = -bx1` and `hi = W - bx2`, so `lo > hi` reduces to `width > W`); in that degenerate case the unguarded Min/Max order silently returns `hi`, which is less than `lo`, producing a nonzero "clamped delta" for a zero-input delta and teleporting the figure. Returning 0 means an oversized box does not move — the correct behaviour for a box that cannot fit inside the canvas. Do NOT touch `ClampMove`'s bounding-box recomputation or its per-axis-independence structure; the only change is inside `ClampDelta`.

    Add regression tests to ClampTests.cs proving the identity that the existing suite never asserts:
    - `ClampMove_OversizedWidthBox_ZeroDelta_IsIdentity`: `Movement.ClampMove(new Box(0, 0, 2000, 100), 0, 0)` returns `new Box(0, 0, 2000, 100)` unchanged (width 2000 exceeds 1280 — the exact CR-02 example).
    - `ClampMove_OversizedHeightBox_ZeroDelta_IsIdentity`: `Movement.ClampMove(new Box(0, 0, 100, 900), 0, 0)` returns `new Box(0, 0, 100, 900)` unchanged (height 900 exceeds 720 — the y-axis mirror).
    - `ClampDelta_WhenLoGreaterThanHi_ReturnsZero`: `Movement.ClampDelta(0, 0, -720)` returns 0, and a nonzero delta on the same inverted range also returns 0 — a direct unit test of the guarded primitive.
    Keep the existing tests intact; they must all still pass (they cover only in-canvas boxes, where `lo <= hi` and the guard is a no-op).
  </action>

  <action>
    Harden `CircleEncoding.ClampDrawRadius(int cx, int cy, double distance)` per CR-01, doing both halves of the drafted fix:
    1. Before computing the cap, clamp the centre into the canvas: derive a clamped centre x from `Movement.ClampDelta(cx, 0, CanvasBounds.Width)` and a clamped centre y from `Movement.ClampDelta(cy, 0, CanvasBounds.Height)`, and use those clamped values in the four edge-distance terms (`clampedCx`, `clampedCy`, `Width - clampedCx`, `Height - clampedCy`). This stops any of the four terms from going negative for an off-canvas centre.
    2. Floor the final result at zero: wrap the whole `Math.Min(...)` cap in `Math.Max(0, ...)` so the function can never return a negative radius even in an unforeseen combination.
    Keep the existing `Math.Round(distance, MidpointRounding.AwayFromZero)` rounding (locked per D-24/D-29, recorded in 01-02-SUMMARY). The function still returns an int radius; the caller keeps its raw centre, so an off-canvas press with radius 0 becomes a degenerate box the guard rejects — a silent no-draw, which is the intended per-type rejection (D-50).

On the pre-fix code this test fails because `r` is -5 and the normalised box `(-10,355,0,365)` is a valid even-sided off-canvas square the guard accepts.
