<objective>
Turn `CONSTRAINT-schema` into a real, running PostgreSQL schema — with the three CHECK constraints actually present in the database, not merely intended.

Purpose: this is ROADMAP success criterion 2, and it is the plan most likely to fail *invisibly*. **EF Core will not emit the CHECK constraints or the `COMMENT ON TABLE` on its own** (D-42). If `HasCheckConstraint` is not called, everything still works: the migration applies, the tables appear, the app boots, `dotnet build` is green — and the database will cheerfully accept a circle that is an oval. Every geometric guarantee in this design rests on constraints that only exist because someone typed them.

So "the constraints exist in the live database" is treated here as a **separately verified outcome**. Build success proves nothing. The acceptance criteria query `pg_constraint`.

Output: `src/BlazorCanvas/Data/` (two entities + the DbContext), the initial migration, and startup migration application.

⚠ The authoritative DDL is `CONSTRAINT-schema` in `.planning/intel/constraints.md`. **Never implement from D-12's own DDL sketch** — it is stale, and still shows the `created_at` column that D-46 dropped.
</objective>

  <action>
    Create namespace `BlazorCanvas.Data` under `src/BlazorCanvas/Data/`.

    **`User`** — `Id` (int), `Username` (string), `Password` (string). Nothing else. `users` has exactly three columns.

    **`Figure`** — `Id` (int), `UserId` (int), `Type` (**`string`**), `X1`, `Y1`, `X2`, `Y2` (all `int`). Nothing else. `figures` has exactly seven columns.

    `Type` is a `string`, deliberately (D-46). Do **not** declare it as a C# enum here, and do **not** reach for `HasConversion<string>()` — EF's built-in enum-to-string conversion uses `Enum.ToString()`, which yields PascalCase (`Circle`), while the CHECKs test lowercase (`type <> 'circle'`). An int-mapped enum is worse still: it would make every CHECK a comparison against a value that can never occur, so all three would pass vacuously and the database would accept anything. A plain `string` column of type `text` has no such failure mode. The geometry core (plan 01-02) supplies the four literals via `FigureTypeNames.ToDbValue`.

    **`CanvasDbContext : DbContext`** with `DbSet<User> Users` and `DbSet<Figure> Figures`, and an `OnModelCreating` that configures ALL of the following **explicitly**. EF Core infers none of it:

    Tables and columns (snake_case, matching `CONSTRAINT-schema`):
    - `User` → table `users`; columns `id`, `username`, `password`. `Id` is the PK, an always-generated identity integer. `Username` → `text`, required, with a **UNIQUE** constraint (D-44). `Password` → `text`, required (D-08 — plaintext; see `<threat_model>` T-BC01-01).
    - `Figure` → table `figures`; columns `id`, `user_id`, `type`, `x1`, `y1`, `x2`, `y2`. `Id` is the PK, an always-generated identity integer — **and it IS the z-order** (D-39). `Type` → column type `text`, required. The four coordinates → `integer`, required (D-20).
    - FK `figures.user_id` → `users.id` with **ON DELETE CASCADE**.

    The four CHECK constraints, via `HasCheckConstraint` on the `figures` table builder — **transcribe the predicates verbatim from `CONSTRAINT-schema`**:
    - `circle_is_a_circle` — rejects a non-square, non-positive, or odd-sided circle. This is what makes the inscribed-square encoding (D-22) machine-checked rather than hoped-for.
    - `box_is_a_box` — rejects a zero-width or zero-height rectangle or triangle.
    - `line_is_a_line` — rejects a zero-length line. **A horizontal or vertical line is legal** and must pass.
    - `figures_type_is_known` — restricts `type` to the four lowercase literals.

    The index, via `HasIndex` on `UserId` with the database name **`ix_figures_user_id`** (D-39) — every page load filters on it.

    The table comment, via `HasComment` on the `figures` table builder. Use the text from `CONSTRAINT-schema` verbatim (in C# it is a single string; the SQL source doubles the apostrophe in "figure's" — that doubling is SQL escaping, not part of the text). It documents that the four integers are always the bounding box, that a circle is stored as its inscribed square with the recovery formulas, that it is *drawn* centre-out but *stored* as a square, and that a line is normalised by swapping the whole point pair, never by sorting axes.

    **Do not add:**
    - a `canvases` table (D-12) — the canvas is just the set of figures belonging to a user
    - a `created_at` column (D-46) — the sequential `id` is the z-order
    - any canvas-bounds CHECK (D-36) — D-24/D-29 already guarantee figures live inside the canvas; a bounds CHECK would be belt-and-braces on a rule the app never breaks
    - any style, colour, or stroke column (D-14)
  </action>
