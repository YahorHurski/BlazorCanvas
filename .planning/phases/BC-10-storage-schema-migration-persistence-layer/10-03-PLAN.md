Positions and `z` are `decimal` because the columns are `numeric` (D-61 ŌĆö exact decimal so accumulated
translations never drift). The four `bbox_*` are `double` because the columns are `double precision`
(D-67 ŌĆö a coarse cache). Geometry and style come back as JSON text rather than parsed records: the

Create `src/BlazorCanvas/Data/V11/FigureRepository.cs`,
`public sealed class FigureRepository(NpgsqlDataSource dataSource)`.

**Public surface ŌĆö five members, no more:**

1. `Task<IReadOnlyList<FigureRow>> LoadAsync(Guid canvasId, CancellationToken ct = default)` ŌĆö
   `SELECT id, canvas_id, type, x, y, rotation, geometry::text, style::text, z, bbox_x, bbox_y,
   bbox_w, bbox_h FROM v11.figures WHERE canvas_id = @canvas_id ORDER BY z`. Document that
   `ORDER BY z` is not cosmetic: `z` IS the stacking order (D-63), SVG paints in document order, and
   `UNIQUE (canvas_id, z)` is what makes the ordering total rather than merely usual.

2. `Task<FigureRow> InsertAsync(Guid canvasId, ValidatedFigureInput input, decimal x, decimal y,
   decimal rotation = 0m, CancellationToken ct = default)` ŌĆö the production draw path. Generates
   `Guid.NewGuid()` **before** issuing the statement and passes it in explicitly; the `id` column has
   no default and no identity. Claims `z` inside the statement itself, and retries on collision (see
   below). Returns the inserted `FigureRow`, including the `z` the database actually accepted.

3. `Task<bool> InsertWithIdAndZAsync(Guid id, Guid canvasId, ValidatedFigureInput input, decimal x,
   decimal y, decimal z, decimal rotation = 0m, CancellationToken ct = default)` ŌĆö the migration
   path. The id and `z` are supplied by the caller (10-04 derives both from the legacy row), the
   statement carries `ON CONFLICT (id) DO NOTHING`, and the return value is whether a row was
   actually inserted. Document that this method deliberately does **not** retry: a `z` collision here
   is a genuine data error ŌĆö two legacy figures claiming one layer ŌĆö and must surface as an exception,
   not be papered over by taking the next free slot and silently changing the stacking order.

4. `Task<int> MoveAsync(Guid canvasId, Guid figureId, decimal x, decimal y, CancellationToken ct =
   default)` ŌĆö exactly one statement, held in a `private const string MoveSql` whose text is exactly:
   `UPDATE v11.figures SET x = @x, y = @y WHERE id = @id AND canvas_id = @canvas_id`. Returns the
   affected-row count. Document that the SET list is deliberately two columns: this is MODEL-01 and
   ROADMAP criterion 2 made literal, it works identically at any vertex count, and it is why the
   `bbox_*` cache is stored in the figure's local frame. Document that the returned count is D-10's
   staleness guard ŌĆö zero means the figure is already gone, and the caller removes it rather than
   treating the call as failed ŌĆö and that this method never inserts, which is D-40's no-resurrection
   rule.

5. `Task<int> DeleteAsync(Guid canvasId, Guid figureId, CancellationToken ct = default)` ŌĆö
   `DELETE FROM v11.figures WHERE id = @id AND canvas_id = @canvas_id`, returning the affected-row
   count. Hard delete; D-69 rejected soft delete explicitly.

**The single insert core.** Both insert methods delegate to one `private async Task<FigureRow?>
InsertCoreAsync(...)` holding the only `INSERT INTO v11.figures` statement in the codebase. It binds
`bbox_x`, `bbox_y`, `bbox_w`, `bbox_h` from `input.Bounds.X`, `.Y`, `.W`, `.H` and from nowhere else,
and binds `type`, `geometry` and `style` from `input.Type`, `input.GeometryJson` and `input.StyleJson`
(cast to `jsonb` in the SQL with `@geometry::jsonb` and `@style::jsonb`). Document above it that this
is D-67's "exactly one place" and MODEL-07's "exactly one code path" ŌĆö a second insert statement
anywhere would reopen the staleness hole the whole design of `bbox_*` depends on being closed.

The `z` expression is the only part that differs between the two callers. Express it as two SQL
constants sharing the same column list and the same parameter binding helper: the auto form uses
`COALESCE((SELECT MAX(z) FROM v11.figures WHERE canvas_id = @canvas_id), 0) + 1` computed inside the
INSERT, and the explicit form binds `@z`. Both `RETURNING` the full column list so the method can
build a `FigureRow` without a second round trip.

**The retry.** `InsertAsync` wraps its attempt in a loop of at most 5 attempts. It catches
`PostgresException` where `SqlState == PostgresErrorCodes.UniqueViolation` and `ConstraintName ==
"z_unique_per_canvas"`, and retries; any other `PostgresException` propagates unchanged. Each attempt
uses its own command on its own connection taken from the data source, so a failed attempt never
leaves an aborted transaction for the next one to run inside. After the fifth failure, rethrow.
Document, in this order: that D-63 requires this and names the failure mode ("without it a figure
silently fails to appear"); that the retry recomputes `MAX(z)` rather than incrementing a cached
value, because the whole reason the first attempt failed is that the cached value was stale; that the
constraint-name check keeps the retry from swallowing a genuine primary-key collision; and that this
is a different mechanism from `Program.cs`'s `EnableRetryOnFailure`, which handles transient transport
failures, not a lost race.

**Hard rules for this file**, each worth a comment:
- Every SQL string is a `private const string`. No SQL is built by interpolation or concatenation, and
  no caller-supplied value ever appears anywhere except as a bound `NpgsqlParameter`.
- Every method's `WHERE` clause includes `canvas_id`. This is the ownership guard the v1.1 store
  applied with `user_id`; a method that finds a figure by id alone would let a caller reach another
  user's canvas, and the migrated ids are deliberately predictable.
- The class accepts geometry and style only as `ValidatedFigureInput`. There is no overload taking
  raw text, and there must never be one: the type is the enforcement of 09-06's rule that Phase 10
  writes only through `FigureInputGateway`.
- No method reads `geometry` in order to write a position, and no method dispatches on `type`.
- There is no `updated_at` column to maintain ŌĆö it was deferred to D-69 during Phase 10 planning
  (see the D-68 amendment). A move writes exactly two columns and no more, per ROADMAP criterion 2,
  with nothing left stale.

The whole point of this file is that the proof is **deterministic**, not a hopeful race. Do not spawn
threads and hope they collide; force the collision with two connections and an uncommitted
transaction.

**A. The forced collision.** On a fresh canvas holding one figure at `z = 1`:
1. Open connection A from the fixture data source and begin a transaction on it.
2. Inside that transaction, insert a figure claiming `z = 2` ŌĆö the value any concurrent writer will
   also compute ŌĆö and do **not** commit.
3. Start `repository.InsertAsync(canvasId, ŌĆ”)` on the shared data source as a background `Task`. It
   computes `MAX(z) = 1` (A's row is invisible, being uncommitted) and issues an INSERT for `z = 2`.
   That statement blocks on the unique index, waiting for A to resolve.

## Flagged assumptions

Phase-local numbering continues from `10-02-PLAN.md`. The cross-cutting assumptions PA-01ŌĆ”PA-11 are
recorded in `10-01-PLAN.md` and `10-02-PLAN.md` and apply here too.

- **PA-12 ŌĆö `z` is claimed inside the INSERT statement, not by a prior SELECT.** One round trip, and
  the window between reading `MAX(z)` and writing it is as small as PostgreSQL can make it. The
  collision is still possible ŌĆö that is why the retry exists ŌĆö but it is rarer than a read-then-write
  pair would make it. **Overturn by requiring a serialisable transaction or an advisory lock instead,
  accepting the extra round trip.**
- **PA-13 ŌĆö RESOLVED (user decision, Phase 10 planning): `updated_at` no longer exists.** This plan
  originally flagged that `MoveAsync` writing exactly `x` and `y` would leave `updated_at` stale.
  On review, D-68 had pulled `updated_at` in alongside `created_at` in its *heading* but never
  argued for it in its body, and no reader existed in v1.11 or anywhere in `src/`. Keeping it would
  have contradicted D-46's "a column that exists for no functional reason" rule and D-69's "take now
  only what would otherwise require a pass over rows already written". The column is **dropped from
  the v1.11 schema and deferred to D-69**; D-68 is retitled "`created_at` returns" and carries an
  amendment note. `MoveAsync` writes two columns and nothing is left stale. **Overturn by naming a
  reader for `updated_at`, which would restore it to the DDL plus either a trigger or a third SET
  column.**
- **PA-14 ŌĆö The retry cap is five attempts.** Enough to survive a realistic two- or three-tab race,
  small enough that a persistent conflict surfaces as an exception within a second rather than hanging
  a circuit. **Overturn by naming a different cap or a backoff policy.**
- **PA-15 ŌĆö `InsertAsync` uses a random `Guid.NewGuid()`, not a sequential uuid.** It matches the
  existing `Guid.NewGuid():N` usage in `DatabaseFixture` and D-62's plain "uuid". A time-ordered v7
  uuid would give better B-tree insert locality; at this project's row counts that is not a
  measurable difference and it would make new ids weakly time-revealing. **Overturn by requiring
  `Guid.CreateVersion7()`.**
