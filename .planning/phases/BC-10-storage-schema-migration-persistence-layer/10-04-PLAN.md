<objective>
Run the migration and prove it lossless. Build the routine that turns a v1.1 database into a v1.11
one — one canvas per user, every figure converted, `z` from the old id, today's fixed style on every
row — and then prove, against the committed v1.1 dump restored into a scratch database, that every one
of the 795 figures comes back with the same rendered vertices and the same stacking order it had
before.

Purpose: ROADMAP criterion 3 says "proven, not assumed", and that phrase is the whole point of this
plan. MIGR-01's promise is that a user who had figures before the upgrade opens the canvas afterwards
and sees the identical picture; the only way to know that is to replay it.

Output: one production file and two test files. Nothing is wired into the running application, and the
old `public.figures` table is read but never altered.
</objective>

Create `src/BlazorCanvas/Data/V11/V11DataMigration.cs`, namespace `BlazorCanvas.Data.V11`.

First a small result record:

`public sealed record V11MigrationReport(int UsersSeen, int CanvasesCreated, int FiguresSeen, int
FiguresInserted, int FiguresAlreadyPresent);`

Document that `FiguresSeen` and `FiguresInserted + FiguresAlreadyPresent` must always be equal, and
that the caller may assert it — a discrepancy means a row went missing, which is the failure MIGR-01
forbids.

Then `public static class V11DataMigration` with one entry point:

`public static async Task<V11MigrationReport> RunAsync(NpgsqlDataSource dataSource, ShapeRegistry
registry, CancellationToken ct = default)`

Sequence, in this exact order — it is the draft's numbered step order minus its step 4, which the
ROADMAP assigns to Phase 11:

1. **Schema.** `await V11Schema.ApplyAsync(connection, ct)` and
   `await V11Schema.SeedFigureTypesAsync(connection, registry, ct)`. Idempotent by construction.
2. **Open one connection and one transaction** for everything that follows. Document that the whole
   data migration is one transaction on purpose: a failure at figure 700 of 795 must leave the
   database un-migrated, not half-migrated, because a half-migrated database is a state no test
   covers and no user can recover from.
3. **Canvases.** `SELECT id FROM public.users ORDER BY id`. For each user id, insert into
   `v11.canvases`: `id = V11DeterministicId.ForCanvas(userId)`, `owner_id = userId`,
   `name = 'Canvas'`, `width = 1472`, `height = 828`, `background = '#FFFFFF'`, with
   `ON CONFLICT (id) DO NOTHING`. Count the rows actually inserted. Document that every user gets a
   canvas including the ones who own no figures — 173 of the 708 in the fixture — because a user
   without a canvas is a state the application has never been tested against, and deciding they do not
   need one is a product decision nobody made.
4. **Figures.** `SELECT id, user_id, type, x1, y1, x2, y2 FROM public.figures ORDER BY id`. Read the
   whole set first (795 rows is nothing) so the reader is not held open across writes. For each row,
   in ascending id order:
   a. `var converted = LegacyFigureConversion.Convert(row)` — throws on anything it does not
      understand, and that throw must propagate.
   b. `var definition = registry.Get(row.Type)`; `var geometryJson = definition.ToJson(
      converted.Geometry)`.
   c. `gateway.TryValidate(row.Type, geometryJson, styleJson: null, out var input)` — using a
      `FigureInputGateway` built over the supplied registry. If it returns false, throw an
      `InvalidOperationException` naming the legacy id and the type but **not** the coordinates.
      Document why the round trip through JSON and back is deliberate rather than wasteful: it routes
      the migration through the same single trust boundary as every other write (09-06's recorded
      requirement for this phase), so there is one validation path rather than two obliged to agree.
      Document also that passing `styleJson: null` is what makes MODEL-06 true — the gateway
      substitutes `StyleGateway`'s own defaults, which are exactly today's fixed appearance
      (`#000000`, `2`, `#FFFFFF`, `1`), so the migration never hand-writes a style literal that could
      drift from the renderer's.
   d. `await repository.InsertWithIdAndZAsync(id: V11DeterministicId.ForFigure(row.Id),
      canvasId: V11DeterministicId.ForCanvas(row.UserId), input, converted.X, converted.Y,
      z: row.Id, rotation: 0m, ...)` on the migration's connection and transaction. Count inserted
      versus already-present. Document that `z` is the old `figures.id` verbatim: under v1.1 the id
      WAS the z-order (D-39), the old ids are globally unique, and each user has exactly one canvas,
      so `UNIQUE (canvas_id, z)` is satisfied with no renumbering and the stacking order survives bit
      for bit.
5. **Commit**, then assert internally that `FiguresSeen == FiguresInserted + FiguresAlreadyPresent`
   and throw if not. Return the report.

`FigureRepository` and `FigureInputGateway` are constructed inside `RunAsync` from the supplied
`dataSource` and `registry` — this method is the composition root for the migration. Add an overload
taking an already-open `NpgsqlConnection` so the replay harness can drive it against a scratch
database, and have the data-source overload delegate to it.

**Hard rules for this file**, each worth a comment:
- No `INSERT INTO v11.figures` statement of its own. Every figure goes through
  `FigureRepository.InsertWithIdAndZAsync`, because that is the single statement that writes the
  `bbox_*` cache and adding a second would reopen exactly the staleness hole D-67 designed around.
- No `try`/`catch` that swallows a conversion or validation failure. There is no "skip and continue"
  path, no default figure, no placeholder. The only correct response to a row it cannot handle is to
  abort.
- No `UPDATE`, `DELETE`, `DROP`, `ALTER` or `TRUNCATE` against anything in the `public` schema. The
  old tables are read-only to this code; Phase 11 is what retires them.
- No exception message contains a figure's coordinates. Legacy ids and type names are fine; user
  drawing data is not.
- Nothing from D-69's deferred list is written, and `rotation` is `0m` for every row.

## Flagged assumptions

Phase-local numbering continues from `10-03-PLAN.md`. The cross-cutting assumptions PA-01…PA-15 are
recorded in the earlier plans and apply here too.

- **PA-16 — The migration performs the draft's steps 1–3 and not its step 4.** The draft's
  `## Миграция существующих данных` ends by dropping the old table; the ROADMAP puts that in Phase 11
  and requires the old table, the old store and the existing tests to stay green throughout this
  phase. Where the two disagree the ROADMAP wins, and group K asserts the old table survives.
  **Nothing to overturn unless you want the cutover pulled forward.**
- **PA-17 — The dump is executed through the Npgsql driver with `psql` meta-command lines stripped,
  not by shelling out to `psql` or `docker exec`.** It removes an external-binary dependency from the
  test suite and works identically on any machine that can already reach the container. The stripped
  lines are `\restrict` and `\unrestrict`, which carry no schema or data. **Overturn by requiring a
  `psql` invocation, accepting the dependency.**
- **PA-18 — The replay uses a per-run scratch database rather than a transaction against the existing
  test database.** A dump recreates whole tables, which cannot be done inside a transaction against a
  database whose `public.figures` already exists and is in use by the rest of the suite.
- **PA-19 — Expected vertices in group C are transcribed literals, not values recomputed by the same
  formula the migration uses.** A test that recomputes proves the code agrees with itself and nothing
  more. The transcription source is `v1.1-pre-rewrite-MANIFEST.md` plus the v1.1 rendering rules in
  `FigureShape.razor`.
- **PA-20 — bbox comparisons are exact, not tolerance-based.** Both sides are the same `double`
  produced by the same function from the same parsed numbers, and `GeometryJson` writes shortest
  round-trippable representations, so exact equality must hold. If it does not, that is a real
  round-trip defect to surface, not something a tolerance should hide.
