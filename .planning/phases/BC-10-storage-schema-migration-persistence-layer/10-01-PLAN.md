<objective>
Land the four-table storage model in the live database as an additive, isolated schema, and prove
against PostgreSQL's own catalogs that it is exactly the shape D-59…D-68 specify. Position and shape
become separate columns (MODEL-01), the canvas becomes an entity with its own size (MODEL-02), and the
list of legal figure types becomes rows rather than a constraint (MODEL-03).

Purpose: everything else in Phase 10 writes to these tables. Getting the column types wrong here —
`double` instead of `numeric`, an absolute instead of local `bbox_*`, a CHECK instead of a lookup
table — is the one mistake this milestone exists to never make again, because it would need another
pass over rows already written.

Output: one production file (`V11Schema.cs`), one corrected doc comment, an additive extension to the
test fixture, five precision-only edits to the existing schema test, and a new live-schema assertion
suite. The running application is not touched and does not know these tables exist.
</objective>

## Flagged assumptions

Phase-local numbering. Every one is a design call the source artifacts did not settle outright; each
names the reading the plan proceeds on and what to say to overturn it.

- **PA-01 — `bbox_*` holds the LOCAL extent, not the position-offset one.** `Bbox.cs` currently
  claims Phase 10 stores `x + X, y + Y`. That claim is incompatible with MODEL-01 ("moving a figure
  writes only `x, y`") and with ROADMAP criterion 2, because an absolute cache must be rewritten on
  every drag; it also contradicts Phase 9's own `v1.1-pre-rewrite-MANIFEST.md`, whose expected-values
  table is explicitly headed "Local bbox". The plan resolves toward LOCAL and corrects the comment.
  **If you want an absolute cache, criterion 2's "writes only x and y" has to change with it.**
- **PA-02 — The new tables live in PostgreSQL schema `v11` until Phase 11 moves them to `public`.**
  Forced, not preferred: criterion 1 names the tables `canvases`, `figures`, `figure_types`, a second
  `public.figures` cannot exist, and the existing `SchemaShapeTests` asserts `public` holds exactly
  three tables and no `canvases`. The alternative is a temporary table name such as `figures_v11`,
  which would make criterion 1 literally false during the phase. **Overturn by choosing the temporary
  name instead.**
- **PA-03 — The DDL is applied by an idempotent applier, not an EF Core migration.** The old EF
  migration chain and `__EFMigrationsHistory` stay literally untouched, and the same applier serves
  the live database, the test database and 10-04's scratch replay database with one call. **Overturn
  by requiring a second EF `DbContext` with its own migrations history table.**
- **PA-04 — There is no EF entity model mirroring the v11 tables.** 10-02's repository is
  parameterised Npgsql SQL. One definition of the schema means no "two places obliged to agree"
  arrangement — the pattern D-50 cost this project a 32-case matrix test to police and that D-67
  names as the thing being removed. It also makes ROADMAP criterion 2's "writes only `x` and `y`"
  directly assertable on the actual statement text. **Overturn by requiring an EF context, accepting
  a model-versus-catalog agreement test.**
- **PA-05 — `ix_canvases_owner` is a plain index, not `UNIQUE (owner_id)`.** The draft DDL has it
  plain, and D-64 explicitly keeps the multi-canvas feature open. MODEL-02's "exactly one canvas per
  user" is therefore a migration invariant proven by test in 10-04, not a constraint that would have
  to be dropped the day the feature ships. **Overturn by adding the unique constraint and accepting
  a future migration to remove it.**
- **PA-06 — Five existing `SchemaShapeTests` queries gain a `public` schema filter.** Precision only:
  no assertion, expected value or test name changes. One of them
  (`FiguresUserId_ForeignKey_HasCascadeDeleteRule`) genuinely breaks without it, because it selects a
  scalar from a multi-row, unordered result once `v11.figures` has foreign keys. **This is the only
  pre-existing behaviour-bearing file this phase edits.**
- **PA-07 — The old `public.figures` table is NOT dropped in this phase.** The draft's migration step
  4 says the old table is dropped; the ROADMAP puts that in Phase 11 so the running application is
  never between two models for longer than one plan sequence. Where the two disagree, the ROADMAP
  wins. **Nothing to overturn unless you want the cutover pulled forward.**
- **PA-08 — Rotation is `0` for every row this phase writes, and `bbox_*` ignores rotation.** D-67
  makes the cache a pure function of `geometry`; a non-zero rotation would change the axis-aligned
  extent and make the cache wrong. v1.11 ships rotation possible and unused (D-59, STATE.md scope
  boundary). **If rotation is ever set to a non-zero value, the cache contract has to be revisited
  before that write path exists.**
