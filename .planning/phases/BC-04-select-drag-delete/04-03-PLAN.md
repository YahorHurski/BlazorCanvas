    **`private async Task OnWrapperPointerMove(PointerEventArgs e)`:**
    1. `if (!dragging) { return; }` — this is what keeps the drag machine and the draw machine from
       ever colliding. They are mutually exclusive by construction: `dragging` can only become true via
       a figure press while the pointer tool is armed, `drawing` only while a shape tool is armed, and a
       tool is one or the other. No locking, no coordination.
    2. THE BUTTONS GUARD: `if ((e.Buttons & 1) == 0) { await CommitDragAsync(); return; }`. `Buttons` is
       the bitmask of buttons currently HELD (bit 0 = primary), as distinct from `Button`, which names
       the button that raised the event. If the primary button is already up while we still believe a
       drag is running, the release happened where we could not see it — the user Alt-Tabbed away, or
       released outside the window and came back. Commit and end. This is D-37's second rule and it is
       ROADMAP criterion 4.
    3. Map the cursor through `CanvasCoordinates.FromPage(e.PageX, e.PageY)`. Compute
       `dx = cursor.X - dragPressX` and `dy = cursor.Y - dragPressY` — from the FIXED press point, every
       time. Never accumulate move-to-move.
    4. If `!dragMoved`, evaluate the threshold: `dragMoved = Math.Sqrt((double)(dx * dx + dy * dy)) >= 3;`
       Euclidean distance from the press point, ratified by the user on 2026-07-16 (04-UI-SPEC.md,
       Resolved Questions §3) and chosen to match the distance formula `DrawGesture.Build` already uses
       for the circle radius — one "distance" concept in the app. The threshold is radial: identical in
       every direction. Once `dragMoved` is true it stays true for the rest of the gesture; do not let it
       flip back if the cursor returns near the press point.
    5. If `dragMoved` is STILL false, return without touching `dragCurrentBox`. Below the threshold the
       figure must not move at all — not by a pixel, not temporarily. D-48 exists to stop a hand-wobble
       from "nudging the figure slightly"; rendering the nudge and undoing it on release would satisfy
       the letter of "no database write" while violating the reason the threshold exists. (04-RESEARCH.md's
       Pattern 3 example clamps unconditionally; this ordering is deliberate and supersedes it.)
    6. `dragCurrentBox = Movement.ClampMove(dragOriginalBox, dx, dy);`

    THE LANDMINE OF THIS PHASE. Call `Movement.ClampMove` with the ORIGINAL box and the TOTAL delta,
    every single call. Never clamp a coordinate on its own — clamping `X2` alone RESIZES the figure
    instead of moving it, and for a rectangle it fails SILENTLY (for a circle, since D-22's reversal
    stores it as its inscribed square, it fails loudly against the `circle_is_a_circle` CHECK the moment
    the row is written). `ClampMove` clamps the DELTA and then translates all four coordinates
    uniformly, which is the only correct shape. Its per-axis independence is the whole point: `dx'` is
    computed from x-terms only and never reads `y`, which is what makes a figure pinned against the
    right edge still slide freely up and down instead of sticking. Do not re-derive any of it, do not
    "improve" it, and do not add a bounds check, an if, or a second clamp around it.
    <!-- planner-discipline-allow: Math.Min -->
    <!-- planner-discipline-allow: Math.Max -->
    <!-- planner-discipline-allow: Math.Clamp -->
    (For the record: no clamping/min/max helper call belongs in this component. `Math.Sqrt` for the
    threshold is the one arithmetic call this file may make.)

    Do NOT pass `dragCurrentBox` back into `ClampMove` on the next move. An accumulating variant
    compounds rounding and makes many small moves against an edge behave differently from one large
    move — the clamp must always be evaluated against the figure's true original position and the total
    delta, exactly as D-36's formula is written. This is the same fixed-press-point shape
    `DrawGesture.Build` already uses.

    **`OnWrapperPointerUp` and `OnWrapperPointerLeave`** — both `private async Task`, both simply
    `if (dragging) { await CommitDragAsync(); }`. Nothing else, and no `Buttons` check in either: on a
    pointerup the released button is already gone from the bitmask, so the guard would read as
    "button up" and be redundant at best. `OnWrapperPointerLeave` is D-37's first rule — the cursor left
    the window, so commit at the current clamped position. It is coherent rather than a hack precisely
    because of the clamp: to leave the surface the cursor must cross the canvas boundary, by which point
    the figure is already pinned at the edge, so the commit drops it exactly where the user can already
    see it. Nothing jumps.

    **`private async Task CommitDragAsync()`** — order matters in every line:
    1. `dragging = false;` FIRST, synchronously. Then capture into locals: the figure id, `dragMoved`,
       and `dragCurrentBox`. All of this before the first `await`. Blazor Server dispatches events one at
       a time, but an `await` yields, so a pointerup arriving while a pointerleave-triggered commit is
       still awaiting the UPDATE would otherwise fire a second UPDATE for the same drag. This is exactly
       the guard `CommitAsync` already uses for the draw.
    2. If the captured id is null, or the captured `dragMoved` is false, return. A sub-3px gesture is a
       CLICK: the figure is already selected (from the press), and there is NO DATABASE WRITE. This
       return is ROADMAP criterion 2 and it is the entire reason the threshold exists.
    3. Apply the box to the in-memory figure — find it in `figures` and assign `X1`/`Y1`/`X2`/`Y2` from
       the captured box — BEFORE the await. This is not an optimisation. Blazor re-renders when an async
       handler yields at its first `await`, and by then `dragging` is already false, so a figure whose
       coordinates had not yet been updated would render at its OLD position for the duration of the
       round-trip and then jump to the new one. Applying first makes the drop visually seamless, and it
       is also what Phase 5's D-52 rollback is defined against: "restore locally" presupposes the local
       state was already moved.
    4. `var affected = await Figures.UpdateAsync(userId, figureId, box);` — ONE call, and this is the
       only UPDATE call site in the app. It happens on drop and nowhere else: no pointer-move writes.
       `userId` is the field the cookie claim populated in `OnInitializedAsync` — never from `e`, never
       from a parameter, never from the wire.
    5. `if (affected == 0)`: the figure no longer exists. Remove it from `figures` and clear `selectedId`
       if it names that figure. Silently. This is D-10: no message, no toast, no prompt, no merge, no
       exception, no retry. A zero-row result IS NOT AN ERROR — it is expected staleness, and treating
       it as a failure is the precise mistake Pitfall 4 names.
       <!-- planner-discipline-allow: affected == 0 -->
       (This branch is required to EXIST — the acceptance criterion below asserts it greps to 1, not 0.)
    6. Otherwise do nothing more. In particular do NOT clear `selectedId`: the figure STAYS SELECTED
       after the drop (D-48), which is what lets a user drag something and then immediately delete it.

    DO NOT BUILD THE BROADCAST HALF. D-10's local half belongs here; its "and broadcast a delete" half
    is Phase 5's DATA-03, as is every other notifier concern. Do not add a notifier, a message, a
    subscriber, a sender GUID, or a throttle in this plan.

    Do not add exception handling around the UPDATE. D-45/D-52 (Phase 5, DATA-04) own the retry policy,
    the rollback broadcast and the modal, and specify that only TRANSIENT failures retry — a zero-row
    UPDATE explicitly must never be retried. Handling written here would either swallow a real failure,
    leaving the screen showing a position the database does not hold, or duplicate Phase 5's work in the
