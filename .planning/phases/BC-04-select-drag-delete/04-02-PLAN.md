<objective>
Teach the two presentation components the three things they are missing: a figure must be able to
look selected, a figure must be able to claim a press, and the Delete button must be able to raise
a click.

Purpose: both components are pure presentation and neither knows what selection means — `Selected`
and `Selectable` are facts handed down by `Home.razor` (plan 04-03), and `OnPointerDown`/`OnDelete`
are facts handed back up. Keeping them dumb is what lets this plan run in parallel with the data
path and be verified by a build.

Output: `FigureShape` can render red, can claim a press, and can decline to; `Toolbar`'s Delete
button can fire. Nothing supplies any of these yet.
</objective>

    Add three parameters to `FigureShape`, in the plain non-required `[Parameter]` style (all three
    have safe defaults, so existing call sites that omit them keep compiling):
    `public bool Selected { get; set; }`, `public bool Selectable { get; set; }`, and
    `public EventCallback<PointerEventArgs> OnPointerDown { get; set; }`.

    Add one computed property beside `OpacityValue`, in the same one-line style:
    `private string StrokeColor => Selected ? "#B91C1C" : "#000000";`
    `#B91C1C` is the `Destructive` token already declared in `03-UI-SPEC.md` and activated by this
    phase — it is not a new colour, and no new hex value may be introduced (D-14 forbids figures
    having colours of their own; this is a selection signal, not a figure style).

    Replace the hardcoded black stroke literal in ALL FOUR shape branches with `stroke="@StrokeColor"`.
    All four, not just the filled ones: a selected line must go red too. Leave `stroke-width="2"`
    exactly as it is on every branch — D-31 rejected a thicker stroke as the highlight by name,
    because it reads as a different-looking figure rather than a selected one. Leave `fill="#FFFFFF"`,
    `fill-opacity` and `stroke-opacity` untouched.
    <!-- planner-discipline-allow: stroke="#000000" -->
    (The literal being replaced is the `stroke="#000000"` attribute; `#000000` itself survives exactly
    once, inside `StrokeColor`.)

    Add the press claim to the root element of all four branches:
    `@onpointerdown="HandlePointerDown" @onpointerdown:stopPropagation="Selectable"`.

    And the handler:
    `private Task HandlePointerDown(PointerEventArgs e) => Selectable ? OnPointerDown.InvokeAsync(e) : Task.CompletedTask;`

    THE MODIFIER IS BOUND TO A PARAMETER, AND THAT IS THE WHOLE DESIGN. `@onpointerdown:stopPropagation`
    accepts a C# `bool` expression, not only a literal — that fact is what resolves this phase's central
    tension without a single line of JavaScript:

    - `Selectable` true (plan 04-03 binds it to "the pointer tool is armed"): propagation stops, the
      canvas's own Phase 3 draw-start handler never sees the press, and the page selects/grabs the figure.
    - `Selectable` false (a shape tool is armed): propagation is NOT stopped, the press bubbles to the
      SVG, and Phase 3's draw-start handler fires exactly as it does today. This is what keeps D-30's
      locked rule alive — a small circle can still be drawn INSIDE a big rectangle. If you hardcode the
      modifier to `true`, drawing on top of any existing figure silently stops working, and Phase 3's
      success criterion 3 regresses without any test failing.

    The guard is duplicated deliberately: the modifier stops the bubbling and `HandlePointerDown`'s
    `Selectable ?` check stops the callback. Keep both. The modifier decides who receives the event;
    the check decides whether this component acts. Without the check, a shape-armed press would both
    draw AND notify the page.

    Do NOT add `:preventDefault` to anything here. Phase 3 solved the text-selection-drag problem with
    plain CSS (`user-select: none` on the canvas surface) and that fix is still in place; the modifier
    suppresses the browser's default action, which is a different problem, and adding it would be
    cargo-cult.

    Do NOT add `tabindex`, `@onkeydown`, `@onclick`, or any focus behaviour to any shape element.
    D-33 closed the keyboard-focus problem by deciding deletion is a toolbar button; giving figures
    focus would reopen exactly that problem.

    Note for the record, so it is not mistaken for an oversight later: the Blazor docs state the
    `stopPropagation` modifier's effect is limited to Blazor's own event scope and does not extend to
    the HTML DOM. That is fine here and only here — this app has no hand-authored JavaScript listener
    competing for these events, by locked constraint (D-37). It is not a general-purpose guarantee.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> circuit (pointer/click events on rendered components) | `PointerEventArgs` on every figure press and the Delete button's click arrive from the client and are attacker-controlled on a crafted circuit. Neither component makes a trust decision with them — both forward upward to `Home.razor` (plan 04-03), which is where the authorisation context (`userId` from the cookie claim) lives. |
| parent component -> `FigureShape` / `Toolbar` parameters | `Selected`, `Selectable`, `DeleteEnabled` are facts asserted by the parent. These components trust them completely and by design; they hold no state and enforce no policy. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-05 | Tampering | `FigureShape.HandlePointerDown` firing when a shape tool is armed | medium | mitigate | If a press were forwarded while `Selectable` is false, a shape-armed press would both draw a new figure AND grab the one underneath — D-30's locked "draw on top of an existing figure" rule breaks, silently, with no test failing. Mitigated by two independent guards: the bound `stopPropagation` modifier decides who receives the event, and `HandlePointerDown`'s `Selectable ?` ternary decides whether this component acts at all. Gated: the modifier greps to 4 bound occurrences and 0 hardcoded ones. |
| T-04-06 | Elevation of Privilege | the Delete button activating with no selection | medium | mitigate | The native `disabled` attribute blocks pointer and keyboard activation and removes the button from the tab order — a real attribute, not a CSS class. This is defence in depth only: it is client-side and a crafted circuit can invoke the callback regardless, which is why plan 04-03's `HandleDeleteAsync` opens with a server-side `selectedId` guard and plan 04-01's `DeleteAsync` filters on `user_id`. Neither this component nor its attribute is trusted for authorisation. Gated: the attribute greps to 1 and is unchanged. |
| T-04-07 | Spoofing / Tampering | non-Blazor JavaScript intercepting figure presses ahead of the `stopPropagation` modifier | low | accept | The Blazor docs state the modifier's effect is limited to Blazor's own event scope and does not extend to the HTML DOM, so a hand-authored DOM listener could observe presses this modifier believes it has stopped. Accepted because no application-authored JavaScript exists in this project by locked constraint (D-37, and the Out of Scope table names it load-bearing), so there is no listener to compete. Recorded rather than mitigated so the assumption stays visible if that constraint is ever revisited. |
| T-04-08 | Information Disclosure | a figure's selection state leaking to another user's screen | low | mitigate | `Selected` is a render-time parameter fed from a per-circuit field (plan 04-03) and is never serialised, persisted, or broadcast. D-15/D-31 lock selection as local UI state; this component holds no state of its own to leak. No schema column and no message kind (`CONSTRAINT-messages`: `draw`/`move`/`delete`/`rollback`) can carry it. |
| T-04-SC | Tampering | package manager installs | high | mitigate | This plan installs no NuGet, npm, pip or cargo package. It edits two existing `.razor` files and uses only `PointerEventArgs` and `EventCallback` from `Microsoft.AspNetCore.Components.Web`, already referenced and vetted in Phases 1-3. Gate: `PackageReference` counts in both csproj files unchanged. No `[ASSUMED]`/`[SUS]` package exists, so no legitimacy checkpoint is required. |

D-08 (plaintext passwords) is a deliberate, locked non-security decision and is deliberately absent
from this register.
