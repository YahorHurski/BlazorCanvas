<objective>
Make a drag write only the anchor. `FigureStore.UpdateAsync` — which today re-encodes and rewrites the
whole `geometry` column on every move — becomes `MoveAsync(userId, figureId, x, y)` whose UPDATE sets
exactly `x` and `y`. `Home.razor`'s drag stops translating a bounding box through the clamp and instead
adds the raw pointer delta to the figure's anchor, so figures leave the canvas. `Movement.cs` and its
test file go with it.

The renderer and the broadcast payload are still box-shaped at the end of this plan; both are fed from
a box DERIVED from the new anchor plus the figure's untouched geometry, which is exactly why the
geometry can no longer drift. 10-03 replaces the payload; 10-04 replaces the renderer.

Purpose: STOR-02's "a drag updates only x,y, for every shape" and STOR-04's "drag translates the anchor
only, with no bounds arithmetic" — the two changes the whole milestone exists to enable.
Output: an anchor-only store API, an anchor-only drag, no `Movement` class, and a reworked store suite.
</objective>

<planner_assumptions>
- **PA-3 — `UpdateAsync` is renamed, not merely narrowed.** The new name `MoveAsync` and the
  `(int x, int y)` signature are chosen so the API *cannot express* a geometry write. A narrower
  `UpdateAsync(userId, id, Box)` would still have to encode, and the "only x,y" rule would live in a
  comment instead of in the type.

  every pointer-move. That reference is read-only until commit.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Blazor Server circuit | Pointer deltas and the dragged figure's id originate in client-driven component state. |
| circuit → FigureStore | `userId` comes from the signed cookie's `user_id` claim (D-51) and is always a parameter; `figureId` is client-influenced. |
| FigureStore → PostgreSQL | Parameterised EF Core LINQ; no string-concatenated SQL. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-10-04 | Tampering | `FigureStore.MoveAsync` / `DeleteAsync` | high | mitigate | `figureId` is client-influenced, so a caller could name another user's real figure id. Mitigated by keeping the `f.UserId == userId` term in the same `Where` clause as the id term, and by carrying over the two cross-user tests as `MoveAsync_NeverTouchesAnotherUsersFigure` and `DeleteAsync_NeverDeletesAnotherUsersFigure`, both asserting a zero affected-row count and an unchanged victim figure. |
| T-10-05 | Tampering | pointer delta on the drag path | low | accept | With the clamp gone, a crafted delta can place a figure anywhere in the `int` domain. Accepted per STOR-04 and D-59 item 6; the row is scoped to the sender's own canvas and an off-canvas figure is the recorded accepted risk STOR-F1. `CanvasCoordinates.FromPage` (10-01) still guarantees the delta is a defined integer. |
| T-10-06 | Tampering | `geometry` jsonb column | medium | accept | D-59 item 9 deliberately leaves no DB CHECK on `geometry`; the server is the sole writer. This plan strengthens the boundary rather than weakening it: after it, no move path can write the column at all, so the only writer is the guarded draw commit. |
| T-10-07 | Denial of service | drag write volume | low | accept | A drag still issues exactly one UPDATE, on drop (D-09), regardless of pointer-move count; the 50 ms trailing-edge throttle (D-47) bounds broadcast volume and is unchanged. |
| T-10-SC | Tampering | package installs | low | accept | No NuGet, npm, pip or cargo package is added in this phase. The Package Legitimacy Gate does not apply. |
</threat_model>
