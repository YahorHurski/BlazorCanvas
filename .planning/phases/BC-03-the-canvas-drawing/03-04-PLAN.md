    `background: #DCE0E5`. Keep `margin: 0` — it is a locked constant (D-43) and removing it moves

    paint the document canvas outside its own element, so on a viewport narrower than 1280px or
    shorter than 768px the areas revealed by scrolling would render white and the borderless canvas
    edge would become invisible against them — and D-55 states plainly that this contrast is the ONLY
    thing that makes the canvas boundary visible; on white, "figures stop at the edge" reads as an
    inexplicable bug. Change nothing else in `app.css`.

      here is the same locked constant as `Toolbar.razor.css`'s `height: 48px` and plan 03-01's
      `CanvasCoordinates.ToolbarHeight`. `.canvas-area` must have no margin, no padding and no
      border: it is the element that carries the canvas down to document y = 48, and any box-model
      value on it moves the canvas without moving the arithmetic.
    - Add `.canvas-surface` with `display: block` and `background: #FFFFFF`. `display: block` is not
      cosmetic — an SVG is inline by default, and inline elements sit on a text baseline with
      descender space, which offsets the element and adds a phantom gap.
    - Add a shape-tool cursor rule: when the canvas carries the shape-armed class, `cursor: crosshair`
      over the entire surface, including over existing figures — a shape tool draws on top of them
      (D-30), so the cursor must not lie about that. With the pointer tool armed the canvas keeps
      `cursor: default`. Do not add `grab`/`grabbing` rules: nothing is selectable or draggable until
      Phase 4, and a grab cursor here would promise a behaviour that does not exist.

    stylistic: a border shifts the SVG interior by its own width, turning the mapping into
    `PageY - 48 - 1` — the classic "the shape appears slightly off from where I clicked" bug, which
    nothing reports and nobody notices until every figure is a pixel out. The canvas's only visible
    edge is the white-against-#DCE0E5 contrast.

    the server cannot know, and `canvasX = PageX` becomes false. (03-UI-SPEC permits centring on the
    login card only, which has no coordinate mapping to protect.)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cookie claim -> circuit state | `user_id` is read once from the authentication cookie into `Home.userId`. Everything downstream trusts that value; nothing upstream of it may be client-controlled. |
| circuit -> `FigureStore` -> PostgreSQL | `LoadAsync(userId)` is the only figure read in the app, and its `user_id` filter is the entire authorization boundary for canvas data. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege (IDOR) | `Home.razor` -> `FigureStore.LoadAsync` | high | mitigate | `userId` is assigned exactly once, in `OnInitializedAsync`, from `state.User.FindFirst("user_id")` on the cascading `AuthenticationState` — which is derived from the signed authentication cookie, not from any circuit message. The page adds no component parameter, route parameter, query-string read, or form field that could reach `userId`. Gated: `FindFirst("user_id")` greps to exactly 1 and `LoadAsync(userId)` to exactly 1. The cross-user isolation test from plan 03-02 covers the store side. |
| T-03-11 | Elevation of Privilege | unauthenticated access to `/` | high | mitigate | `@attribute [Authorize]` is retained verbatim (gated: greps to 1) and the cookie middleware's `LoginPath = "/login"` redirects anonymous visitors (D-51, plan 02-02). Removing the attribute would expose the page with a null claim and throw a `NullReferenceException` on the claim read rather than leaking data — but the attribute is what keeps it a redirect. |
| T-03-12 | Information Disclosure | figure data left in a prerendered response | low | accept | Prerendering causes `OnInitializedAsync` and therefore the figures SELECT to run twice, and the first pass embeds the user's own figures in the initial HTML response. Accepted: the response is already authenticated and scoped to that same user by `[Authorize]` plus the claim, so it discloses to a user only their own data. No change to the render mode, which plan 02-03 verified. |
| T-03-13 | Tampering | layout constants drifting apart | medium | mitigate | The 48px strip height, `calc(100vh - 48px)`, and `CanvasCoordinates.ToolbarHeight` are three expressions of one constant, and a border or margin on the canvas silently adds a fourth term. Gated by negative greps for `border`, `margin`, and `padding` in `Home.razor.css`, positive greps for `width="1280"`/`height="720"`, a negative grep for any scaling attribute on the SVG, and a behavior assertion that the SVG's measured bounding rect is exactly `x: 0, y: 48, width: 1280, height: 720`. |
| T-03-SC | Tampering | package manager installs | high | mitigate | This plan adds no NuGet, npm, pip, or cargo packages; it edits three existing files and references only wave-1 symbols. Gate: `PackageReference` counts in both csproj files unchanged. No `[ASSUMED]`/`[SUS]` package exists, so no legitimacy checkpoint is required. |

D-08 (plaintext passwords) is a deliberate, locked non-security decision and is deliberately absent
from this register.
</threat_model>
