  <action>
    Create `src/BlazorCanvas/Data/FigureStore.cs` in namespace `BlazorCanvas.Data` as a
    `public sealed class FigureStore` taking `IDbContextFactory<CanvasDbContext>` via primary
    constructor or a readonly field, exposing exactly two methods:

    `public async Task<List<Figure>> LoadAsync(int userId)`
    - `await using var db = await _factory.CreateDbContextAsync();`
    - Query: `db.Figures.AsNoTracking().Where(f => f.UserId == userId).OrderBy(f => f.Id).ToListAsync()`.
    - `AsNoTracking` because the returned entities outlive the context â€” the circuit holds them as
      plain view state for the life of the tab.
    - `OrderBy(f => f.Id)` is not cosmetic: `figures.id` IS the z-order (D-39), SVG paints in
      document order, and this ordering is the only thing that makes occlusion identical after an F5.
      There is no `created_at` column to order by (D-46).
    - No join, no canvas entity, no `Include` â€” "the canvas" is not a row, it is this WHERE clause
      (D-03, D-12).

    `public async Task<Figure> InsertAsync(int userId, FigureType type, Box box)`
    - `await using var db = await _factory.CreateDbContextAsync();`
    - Build `new Figure { UserId = userId, Type = FigureTypeNames.ToDbValue(type), X1 = box.X1, Y1 = box.Y1, X2 = box.X2, Y2 = box.Y2 }`.
    - `db.Figures.Add(figure); await db.SaveChangesAsync(); return figure;`
    - Return AFTER `SaveChangesAsync`, so EF Core has populated `figure.Id` from the database
      identity column. This ordering is D-39 and it is load-bearing for Phase 5: the id does not
      exist until the INSERT completes, so the `draw` broadcast can never be fired optimistically.
      Do not pre-assign an id, do not return before saving, and do not add an out-parameter.

    `userId` is a parameter, always. `FigureStore` must never read the claim, the `HttpContext`, or
    any ambient state to discover whose canvas it is â€” the caller (Home.razor, plan 03-04) reads it
    from the cookie claim and passes it in. That keeps exactly one place in the app where the
    identity of the canvas is decided.

    `FigureStore` must NOT normalise, clamp, or guard. It receives a Box that `DrawGesture` already
    clamped and normalised and that `MinSizeGuard` already accepted (plan 03-05). Adding a second
    copy of those rules here would create a second source of truth for the geometry, which is exactly
    what D-50's "mirrors the CHECKs exactly" forbids.

    Do not add an update or delete method â€” those are Phase 4 (FIG-03, FIG-04) and their zero-row
    affected-count guard (D-10) belongs with them.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> Blazor Server circuit (SignalR) | The circuit receives events from an authenticated but otherwise untrusted client. The identity of the canvas must never be derivable from anything the client sends. |
| circuit -> PostgreSQL (EF Core / Npgsql) | Every figure read and write crosses here. The `WHERE user_id = @id` clause in `FigureStore.LoadAsync` is the entire authorization boundary for figure data. |
| static SSR request -> PostgreSQL | `Login.razor`'s create-or-compare, retargeted in Task 1. Behavior must not change. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege (IDOR) | `FigureStore.LoadAsync` / `InsertAsync` | high | mitigate | `userId` is a required parameter supplied ONLY from the `user_id` cookie claim by the caller. `FigureStore` is forbidden from reading `HttpContext`, `ClaimsPrincipal`, or any ambient identity (gated: those identifiers grep to 0 in the file), so no code path exists in which a browser-supplied value could become the filter. `LoadAsync` always filters on `user_id`; there is no unfiltered "get figure by id" method to abuse. Proven by the cross-user isolation test in Task 3, which is a required acceptance criterion, not an optional check. |
| T-03-03 | Denial of Service | DbContext lifetime in the InteractiveServer circuit | medium | mitigate | The scoped context is removed from the container entirely (gated: `AddDbContext<CanvasDbContext>` greps to 0 in `Program.cs`). Every query and write creates and disposes its own context via `IDbContextFactory`, so a circuit cannot accumulate tracked entities for hours, and two overlapping awaited handlers cannot trigger `A second operation was started on this context instance` and kill the circuit. `LoadAsync` uses `AsNoTracking`. |
| T-03-04 | Information Disclosure | sequential `figures.id` | low | accept | Ids are sequential integers and therefore enumerable (D-39 locks this: the id IS the z-order and there is no `created_at`). Accepted because no method in `FigureStore` looks a figure up by id without also filtering on `user_id`, so an enumerated id discloses nothing. Revisit only if a by-id lookup is ever added (Phase 4 must keep the `user_id` filter in its UPDATE/DELETE predicates). |
| T-03-07 | Tampering | `Login.razor` migration | medium | mitigate | Task 1 retargets the context but is explicitly forbidden from altering the `EditForm`, the validation branches, `IsUniqueViolation`, the `user_id` claim, the static-SSR render mode, or the `IsPersistent` default. Gated by source assertions on `@rendermode` (0), `user_id` (1), and a passing `UsernameNormalizer` filter, so a silent auth regression cannot slip through. |
| T-03-SC | Tampering | package manager installs | high | mitigate | This plan adds no NuGet, npm, pip, or cargo packages â€” `IDbContextFactory` ships in `Microsoft.EntityFrameworkCore`, already present transitively via `Npgsql.EntityFrameworkCore.PostgreSQL` 10.0.3. The test-side factory is a hand-written nested class specifically so no DI or mocking package is introduced. Gate: `PackageReference` counts in both csproj files are unchanged (acceptance criteria in Tasks 1 and 3). No `[ASSUMED]`/`[SUS]` package exists, so no legitimacy checkpoint is required. |

D-08 (plaintext passwords) is a deliberate, locked non-security decision and is deliberately absent
from this register.
</threat_model>
