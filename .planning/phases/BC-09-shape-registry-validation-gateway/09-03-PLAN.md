## Planner Assumptions

- **PA-13 ŌĆö Seed before dumping, rather than capturing read-only.** The ROADMAP requires "a diagonal
  line in each direction"; the live database was verified at planning time to hold 176 down-and-right
  diagonals and zero of every other line orientation. A pure read-only capture would therefore bank a
  fixture that can never exercise the D-60 line conversion's direction preservation ŌĆö the single most
  dangerous formula in the migration. The seeded rows go in through the live CHECK constraints, so
  every one is genuinely v1.1-legal.
- **PA-14 ŌĆö The whole database is captured, not a curated subset.** The 631 users are largely test-suite
  artifacts, but they are free MIGR-02 coverage ("each existing user receives exactly one canvas"), and
  the capture is one-shot: taking everything is the choice that cannot be regretted.
- **PA-15 ŌĆö Passwords are redacted in the committed fixture.** See T-09-13. **Flagged for user
  override at the checkpoint** ŌĆö this is the one content decision that cannot be revisited after
  Phase 10 runs.
- **PA-16 ŌĆö The dump is assembled from parts rather than a single `pg_dump`.** Redaction is done by
  generating the user rows server-side with `format(%L)`, not by regex-editing dump text, because a
  value-level regex over generated SQL is fragile in exactly the way that would go unnoticed.
- **PA-17 ŌĆö `--inserts` rather than `COPY`.** The fixture is reviewed once by a human and then frozen;
  readability and diffability are worth more than restore speed for ~715 rows.
- **PA-18 ŌĆö Fixture path `tests/BlazorCanvas.Tests/Fixtures/`.** No fixture folder exists yet; this
  location keeps it next to the suite that consumes it and inside the project whose `<None Update>`
  copies it to the output directory.

The full cross-cutting assumption list (PA-01 ŌĆ” PA-08) is in `09-01-PLAN.md`.
