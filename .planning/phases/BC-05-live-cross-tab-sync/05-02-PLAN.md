<objective>
Two additions to `Program.cs`: register the notifier as a process-wide Singleton, and give the shared
DbContext factory D-52's bounded transient-retry policy.

Purpose: both are one-line seams with outsized consequences, and both fail in ways that produce no
error. A Scoped notifier compiles, boots, and makes every subsequent plan a silent no-op. A missing
retry policy means the first blip in the database connection reaches `Home.razor` as an immediate
failure and shows the user D-52's modal for something that would have succeeded on the second
attempt. Neither is visible in a code review of the files that *use* them, which is why they are their
own plan rather than a footnote in 05-03.

Output: a `Program.cs` where the notifier is a singleton and every database write retries transient
failures at most twice.
</objective>

  <action>
    Add `using BlazorCanvas.Sync;` to the using block at the top of `Program.cs`, keeping the existing
    alphabetical-ish grouping with the other `BlazorCanvas.*` usings.

    Add exactly one registration line, immediately after the existing
    `builder.Services.AddScoped<FigureStore>();`:

    `builder.Services.AddSingleton<CanvasSyncNotifier>();`

    SINGLETON, NOT SCOPED. This is the whole point of the line and it is the one mistake in this
    phase that produces no signal at all. In Blazor Server every open tab is its own circuit with its
    own DI scope. `AddScoped<CanvasSyncNotifier>()` would give each tab a private notifier instance:
    it compiles, the app boots, `Publish` runs, `Subscribe` runs, no exception is ever thrown — and no
    message ever crosses a tab boundary, because each tab is publishing into and subscribing to a
    different object. The feature would be a no-op that looks correct in code review. `AddSingleton`
    gives one instance per process, which is exactly what D-11's Mechanism section specifies ("a DI
    singleton notifier service") and what makes the `user_id` key (rule 7) the thing that separates
    users rather than the DI scope.

    Do not register it as `Transient` either — a transient notifier would hand out a fresh instance to
    every injection point, which is the same silent no-op, only worse.

    `FigureStore`'s existing `AddScoped` registration is correct and must not change. The two services
    have deliberately different lifetimes for deliberately different reasons: `FigureStore` is per-
    circuit because it wraps a per-circuit factory, and the notifier is process-wide because bridging
    circuits is its entire job.

    Add a short comment above the new line, in the style of the existing `AddDbContextFactory` comment
    block: record that the Singleton lifetime is load-bearing (D-11), that each tab is its own circuit
    with its own DI scope so a Scoped registration would give every tab a private notifier and cross-
    tab sync would silently never fire, and that the notifier is safe to share process-wide because it
    holds no per-user state outside its `user_id`-keyed buckets and takes no constructor dependency.
    Note in the comment that this is a deliberate departure from the official Microsoft notifier
    example, which registers Scoped for a single-circuit scenario.
  </action>

      exception, so nothing retries it and nothing needs to exclude it.
    - `.planning/phases/BC-05-live-cross-tab-sync/05-RESEARCH.md` — Pattern 4 (`EnableRetryOnFailure`,
      and why no hand-rolled classification is needed), its caveat about the reported `IsTransient`

      also makes `LoadAsync` retry — an accepted, deliberate side effect, not scope creep).

      `ExecuteUpdateAsync`/`ExecuteDeleteAsync` inherit the execution strategy automatically.

  <action>
    Extend the existing `AddDbContextFactory<CanvasDbContext>` registration so `UseNpgsql` takes its
    second, provider-options lambda parameter, configuring:

    `EnableRetryOnFailure(maxRetryCount: 2, maxRetryDelay: TimeSpan.FromMilliseconds(200), errorCodesToAdd: null)`

    Keep the first argument — `builder.Configuration.GetConnectionString("Canvas")` — exactly as it
    is. Do not inline a connection string and do not add a fallback; plan 01-06 deliberately removed a
    silent fallback from the design-time factory and that decision stands.

    The three values are not arbitrary and must not be "rounded up":
    - `maxRetryCount: 2` is D-52's "retry up to 2 additional times", literally. The Npgsql strategy
      counts *additional* attempts after the first, so 2 here means at most 3 total attempts — which
      is exactly what D-52 says. Raising it converts a database outage into a self-inflicted retry
      storm against a database that is already struggling.
    - `maxRetryDelay: TimeSpan.FromMilliseconds(200)` is D-52's "short delays" — it caps the backoff,
      so the worst case a user waits before the modal appears stays sub-second.
    - `errorCodesToAdd: null` means "add nothing to the provider's own transient list". This is the
      important one: the provider decides transient-vs-not via `NpgsqlException.IsTransient`, which is
      already false for CHECK-constraint violations (SQLSTATE class 23, integrity constraint
      violation) and true for connection-level failures. That is D-52's boundary exactly — "never
      retry validation errors, CHECK violations" — enforced by the provider rather than by app code.

    WRITE NO CLASSIFICATION CODE. Do not add a hand-written retry loop in `FigureStore`, do not switch
    on `PostgresException.SqlState`, and do not test `IsTransient` yourself anywhere. `FigureStore.cs`
    must not be edited by this task at all: since EF Core 7 a single, non-transaction-spanning
    `ExecuteUpdateAsync`/`ExecuteDeleteAsync` is wrapped by the configured execution strategy with no
    caller-side wrapping, so all four store methods become retry-aware from this one line. A second,
    hand-written classification would be a divergence-prone duplicate of the provider's list, and a
    hand-written loop wrapping an already-retried operation would multiply the attempt count.

    D-10'S ZERO-ROW CASE IS UNTOUCHED BY THIS. A zero-row `ExecuteUpdateAsync` throws nothing at all —
    it returns the integer 0. It therefore never reaches the retry strategy, needs no exclusion here,
    and stays what it has always been: expected staleness, not an error (D-52 says so explicitly, and
    D-45 repeats it). Nothing in this task should try to make the retry policy aware of it.

    ACCEPTED SIDE EFFECT, RECORD IT RATHER THAN UNDO IT: because this configures the one shared
    factory, the page-load `LoadAsync` query now also retries transient failures. That is beyond what
    Phase 5's success criteria describe, it is harmless, it strictly improves resilience, and it
    contradicts no locked decision (it sits squarely inside D-45's "the app stays alive" stance).
    05-RESEARCH.md Pitfall 5 calls for this to be documented explicitly rather than discovered later.
    Note it in the comment.

    Extend the existing comment block above `AddDbContextFactory` — do not delete or replace it. The
    current text explains why an `IDbContextFactory` is used instead of a scoped context; that
    reasoning is still true and still needed. Append to it: that the retry policy is D-52's, that the
    provider (not this app) decides what is transient via `NpgsqlException.IsTransient`, that this is
    why no exception-classification code exists anywhere in the app, that a zero-row UPDATE never
    reaches it because it is not an exception (D-10), and that `LoadAsync` inheriting the retry is a
    known and accepted side effect of configuring the shared factory.

    Finally, confirm the startup migration retry loop below still behaves. It stays exactly as
    written: its `catch (NpgsqlException) when (attempt < maxAttempts)` handles the container-still-
    starting case at boot, which is a different concern from a mid-session transient blip, and the two
    are complementary rather than duplicated. `Database.Migrate()` runs under the execution strategy
    and needs no change.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DI container → every user's circuit | The Singleton registration makes one `CanvasSyncNotifier` instance reachable from every authenticated user's circuit simultaneously. This registration is what turns 05-01's `userId` key from a convenience into the sole isolation control. |
| app → PostgreSQL | The retry policy governs how many times a failed write is re-attempted against the database, and therefore how much load a failing app can generate against a failing database. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-10 | Elevation of Privilege | `AddSingleton<CanvasSyncNotifier>()` | high | mitigate | The singleton is process-wide by design and therefore holds every user's subscriber buckets in one object. Isolation rests entirely on two controls, both outside this line: `Publish`'s `userId` key (T-05-01, proven by `Publish_ForDifferentUser_DoesNotInvokeHandler` in plan 05-01) and `Subscribe` being called only with the claim-derived `userId` (T-05-07, plan 05-03). No client-supplied value reaches either parameter. This plan's contribution is to make that dependency explicit in the `Program.cs` comment so the Singleton is never "hardened" into a Scoped registration by someone who reads the lifetime as the isolation mechanism. |
| T-05-09 | Denial of Service (self-inflicted) | `EnableRetryOnFailure` | medium | mitigate | `maxRetryCount: 2` with `maxRetryDelay: 200ms` bounds each failed write to at most 3 total attempts within a sub-second window — not an unbounded or exponentially-growing retry storm against an already-failing database. Enforced by the `maxRetryCount: [^2]` prohibition gate. |
| T-05-11 | Tampering | `errorCodesToAdd: null` / transient classification | medium | mitigate | Classification is delegated entirely to `NpgsqlException.IsTransient`, which excludes SQLSTATE class 23 (integrity constraint violation) — so a CHECK violation, which is the app writing a geometrically illegal row, is never retried and never masked by repetition. D-52 requires exactly this boundary. The "no `SqlState`/`IsTransient` anywhere in `src/`" gate prevents a second, drifting classifier from being added later. |
| T-05-SC | Tampering | package-manager installs | low | accept | No package is installed. `EnableRetryOnFailure` is an existing member of `Npgsql.EntityFrameworkCore.PostgreSQL` 10.0.3, referenced and vetted since Phase 1 and verified present this session via `dotnet list package` (05-RESEARCH.md § Standard Stack). 05-RESEARCH.md's Package Legitimacy Audit records the phase as installing nothing, so no `[ASSUMED]`/`[SUS]` entry exists and no legitimacy checkpoint is required. |
</threat_model>
