<objective>
Close the last success criterion: when a write ultimately fails, no screen is left lying. The acting
tab restores the figure, broadcasts `rollback` so every other tab does too, and shows one modal that
reloads from PostgreSQL on OK. The circuit stays alive.

Purpose: this is the half of the phase that only exists because of the other half. Because the glide
broadcasts are optimistic (plan 05-03, D-47's guarantee), a failed drop leaves every other tab showing
a position Postgres never accepted. D-52 calls the rollback FORCED, not defensive — without it, every
open screen is lying and only F5 fixes it. Kept separate from 05-03 so the error path is designed as
its own concern rather than bolted onto three handlers while they are being written.

Output: the phase's one new piece of markup, its one string, and the try/catch on all three write
paths.
</objective>

  <action>
    THE DRAG PATH — in `CommitDragAsync`, wrap the existing `await Figures.UpdateAsync(...)` call and
    its existing `if (affected == 0)` branch in a `try` block. Everything before it stays outside and
    unchanged: `dragging = false;` first, the `figureId`/`moved`/`box` capture, the click guard, the
    local figure mutation, and the unconditional `move` publish (which must stay ABOVE the try — it is
    D-47's trailing edge and it is deliberately optimistic).

    In the `catch`, in this order (D-52's steps 1 and 2, plus 3):
    1. Restore the local figure's four coordinates from `dragOriginalBox`, guarding for `figure` being
       null.
    2. `Notifier.Publish(userId, SyncMessage.Rollback(figureId.Value, dragOriginalBox, _sender));`
    3. `showSaveFailedModal = true;`

    THE ROLLBACK IS FORCED, NOT DEFENSIVE, AND THIS IS THE WHOLE POINT OF THE PLAN. By the time this
    catch runs, the glide's `move` broadcasts have ALREADY gone out — including the unconditional
    final one, sent a few lines above, before the write was even attempted. So every other tab is
    already showing the figure at its new position while the database still holds the old one. Showing
    a message here and leaving it at that would mean EVERY OPEN SCREEN IS LYING, with only F5 to fix
    it. The rollback broadcast is what makes all screens agree with the database again. Do not skip it
    on the grounds that "the other tabs will find out eventually" — they will not; there is no
    periodic re-read anywhere in this app.

    THE PAYLOAD IS `dragOriginalBox`, NOT `box`. `box` is the position that just FAILED to persist;
    broadcasting it would tell every other tab to keep the lie. `dragOriginalBox` is the pre-drag
    position, retained unmutated for the entire drag precisely so this is possible — Phase 4 was told
    to keep it for this exact line, and its comment in the field declaration says so.

    RETRY NOTHING HERE. Do not add a retry loop, a delay, or a second attempt. `EnableRetryOnFailure`
    (plan 05-02) already retried up to 2 additional times if the failure was transient; if execution
    reaches this catch, either those attempts are exhausted or the failure was never transient
    (a validation error, a CHECK violation) and D-52 forbids retrying it — it would fail identically
    every time. A hand-written retry here would also multiply the provider's attempt count.

    THE ZERO-ROW BRANCH IS NOT A FAILURE AND MUST NOT REACH THE MODAL. The `if (affected == 0)` branch
    stays inside the try (a zero-row result throws nothing — it returns the integer 0, so it never
    triggers the catch) and keeps the exact shape 05-03 left it in: remove locally, clear the
    selection if it matched, publish a `delete`. Do not set `showSaveFailedModal` in it, do not retry
    it, and do not add a message. D-10 and D-52 are both explicit: a zero-row UPDATE is expected
    staleness, not an error — the figure vanishing IS the feedback (05-UI-SPEC.md § Silent Ghost
    Removal). Blurring this line is the specific mistake that would tell the user something went wrong
    when the app worked exactly as designed.

    THE DRAW PATH — in `CommitAsync`, wrap the existing `await Figures.InsertAsync(...)`, the
    `figures.Add(figure);` and the `draw` publish in a try. In the catch, set
    `showSaveFailedModal = true;` and nothing else.

    NO ROLLBACK BROADCAST FOR DRAW, and the asymmetry is not an oversight. `draw` is only broadcast
    AFTER a confirmed insert (D-39 — the id does not exist until then), so if the insert throws,
    nothing was ever announced to any other tab and there is nothing to correct. There is also nothing
    to restore locally: `figures.Add` is inside the try and never ran. The reload on OK is what
    reconciles this tab.

    THE DELETE PATH — in `HandleDeleteAsync`, wrap the existing `await Figures.DeleteAsync(...)`, the
    `selectedId` clear, the `figures.RemoveAll(...)` and the `delete` publish in a try. In the catch,
    set `showSaveFailedModal = true;` and nothing else.

    NO ROLLBACK BROADCAST FOR DELETE either, for the same reason — plan 05-03 already restructured
    this method to await first and only mutate and broadcast after a confirmed success, so a throw
    leaves the figure still in `figures`, still on screen, and still unannounced. Nothing to undo.
    This is exactly why 05-03 flipped that order.

    All three write paths get the SAME modal and the SAME reload-from-database. The modal's text is
    generic, not drag-specific, and D-45 is the app's only general error-handling stance — it is not
    drag-exclusive. Only the drag additionally gets a `rollback` broadcast, because only the drag had
    already optimistically told other tabs something that turned out to be false. (This resolves
    05-RESEARCH.md Open Question 1 along its own recommendation.)

    CATCH `Exception` AT THESE THREE SITES — a deliberate, recorded engineering call, not laziness.
    05-RESEARCH.md Assumptions Log A2 establishes that whether the escaping type is a raw
    `NpgsqlException`/`PostgresException` or a wrapped `DbUpdateException` for
    `ExecuteUpdateAsync`/`ExecuteDeleteAsync` specifically is not settled by any authoritative source,
    because the `DbUpdateException` wrapping happens in the `SaveChanges`/change-tracker pipeline that
    the bulk-update APIs explicitly bypass. Guessing wrong and narrowing to the wrong type would let a
    real failure of the other type go uncaught and crash the circuit — the exact D-45 violation this
    plan exists to close. A broad catch is acceptable here specifically because (a) these are three
    named, narrowly-scoped call sites rather than a global handler, and (b) this catch's only job is
    to answer "did the write ultimately fail", never "why". Note this reasoning in the code comment so
