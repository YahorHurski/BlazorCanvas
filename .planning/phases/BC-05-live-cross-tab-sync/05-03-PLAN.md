<objective>
Wire `Home.razor` to the notifier: subscribe on init, unsubscribe on dispose, filter your own echo,
discard everything while dragging, apply the four kinds idempotently with `move` update-only — and on
the sending side, broadcast the draw, the throttled glide, the guaranteed final position, and the
delete (including the zero-row delete that stops a ghost living forever).

Purpose: this is where the phase becomes real, and it is where seven of `CONSTRAINT-sync-core`'s nine
irreducible rules live (rules 1 and 7 are structural and already landed in 05-01/05-02). Every rule
here exists because its absence produced a specific bug — none is optional. The plan deliberately
stops short of the failure path: `try`/`catch`, the rollback broadcast and the modal are plan 05-04,
so this plan can be reviewed as one question — "does the right message go out, and is the right
message ignored?" — without a second, orthogonal concern competing for attention.

Output: two tabs of the same user that mirror each other live, including a real-time drag glide.
</objective>

  <action>
    DIRECTIVES. Add to the existing block at the top of `Home.razor`: `@using BlazorCanvas.Sync`,
    `@inject CanvasSyncNotifier Notifier`, and `@implements IDisposable`. Keep the existing
    `@inject FigureStore Figures` and every other directive exactly as they are.

    FIELDS. Add three private fields to the `@code` block, next to the existing drag state:

    - `private readonly Guid _sender = Guid.NewGuid();` — one GUID per circuit, created once at field
      initialisation. This is D-53's `sender`: it exists solely for the echo filter. It must be
      `readonly` and must be initialised at the field, not inside a lifecycle method and never per
      message — a per-message GUID would make `msg.Sender == _sender` never true, so every tab would
      apply its own broadcasts and stomp its own in-progress drag.
    - `private IDisposable? _subscription;` — the handle returned by `Subscribe`.
    - `private long _lastBroadcastTicks = long.MinValue;` — the throttle gate's state, used by Task 2.
      Seed it with `long.MinValue` so the very first pointer-move of the very first drag broadcasts
      immediately instead of waiting out a 50ms window.

    SUBSCRIBE. In the existing `OnInitializedAsync`, after the existing `figures = await
    Figures.LoadAsync(userId);` line, add:
    `_subscription = Notifier.Subscribe(userId, HandleRemoteMessage);`

    ORDER MATTERS AND IT IS A SECURITY PROPERTY. The `userId` assignment from the `user_id` cookie
    claim must stay above this line (it already is). Subscribing before `userId` is read would
    subscribe with the field's default `0`, which would put every tab of every user into one shared
    bucket and broadcast every user's figures to every other user (T-05-07). `userId` comes from the
    claim (D-51) and from nowhere else — never from a component parameter, a query string, or a form
    value.

    Subscribing after `LoadAsync` is also deliberate: it means a message cannot be applied to a
    `figures` list that has not been populated yet.

    DISPOSE. Add `public void Dispose() => _subscription?.Dispose();`

    This is D-11 irreducible rule 1 and it is not optional: without it, a closed tab's delegate stays
    in the notifier's bucket forever, so the subscriber set grows without bound and every subsequent
    publish invokes a handler on a torn-down circuit. The null-conditional matters because
    `OnInitializedAsync` is `async` — a component whose initialisation threw or was cancelled before
    the subscribe line can still be disposed.

    THE CALLBACK. Add `private void HandleRemoteMessage(SyncMessage msg)`.

    It is invoked on the PUBLISHER's circuit thread — some other tab's thread — so it must not touch
    this component's state directly. Its whole body is a fire-and-forget `_ = InvokeAsync(...)` (D-11
    irreducible rule 2: touching a component from a foreign thread is undefined). Discard the returned
    task with `_ =`: `HandleRemoteMessage` matches `Action<SyncMessage>` and cannot be awaited by the
    notifier, which must never block on another tab's render.

    Inside the `InvokeAsync` lambda, in this order:

    1. Echo filter — if the message's `Sender` equals `_sender`, return. This is D-11 rule 3: the
       acting tab receives its own broadcast (the notifier deliberately delivers to every subscriber
       of the user, including the sender — it cannot tell tabs apart). Without this a tab re-applies
       its own move mid-drag and fights itself.
    2. The mid-drag discard — if `dragging`, return.
    3. `ApplyMessage(msg);`
    4. `StateHasChanged();`

    STEP 2 IS A BLANKET RULE AND THIS IS THE SINGLE EASIEST THING TO GET WRONG IN THIS PHASE. It is
    one boolean check on the existing `dragging` field and nothing else. Do NOT add a second condition
    comparing the incoming message's figure id against the figure this tab happens to be dragging.
    D-54 decides the blanket rule explicitly and lists the narrow per-figure filter under *Rejected*;
    D-11's own checklist summarised D-54 backwards for a while, which is exactly why an implementer
    reading the checklist rather than the decision builds the rejected version. It is safe because of
    the one-mouse premise: while you are dragging, nothing else is happening. The accepted cost — a
    message arriving from a second device mid-drag is lost until F5 — is recorded deliberately in D-54
    and is not a bug to fix.
    <!-- planner-discipline-allow: dragFigureId -->
    (For the record, the field name that must NOT appear anywhere inside `HandleRemoteMessage` is
    `dragFigureId`. Its absence from that method is the executable form of D-54.)

    Wrap the lambda's body in `try { ... } catch (ObjectDisposedException) { }` — swallow it silently.
    This is defence-in-depth alongside `Dispose()`, not instead of it: `Dispose()` guarantees no
    FUTURE publish reaches you, but it cannot retroactively cancel a publish that grabbed your handler
    reference microseconds earlier and is about to call `InvokeAsync` on a circuit that has since been
    torn down. That race is small, real, and inherent to any multi-circuit pub/sub. The tab is gone;
    there is nothing to update; dropping the message is correct.

    THE APPLY. Add `private void ApplyMessage(SyncMessage msg)` — a `switch` on `msg.Kind` over
    exactly the four D-53 kinds, called only from inside the `InvokeAsync` lambda above.

    - `"draw"` — if no figure with `msg.Id` is already in `figures`, add a new `Figure` built from the
      message (`Id`, `UserId = userId`, `Type = msg.Type!`, and the four coordinates). The existence
      check is what makes `draw` idempotent, matching D-53's "insert or update by id". `draw` is the
      ONLY kind that may create a figure.
    - `"move"` and `"rollback"` — one shared case; D-53 says `rollback` is applied update-only exactly
      like `move`. Find the figure with `msg.Id` in `figures`; **if it is not there, return — do not
      add it.** Otherwise assign the four coordinates onto the existing figure.

      THIS `return` IS D-40 AND IT IS THE POINT OF THE WHOLE ENTRY. D-11 originally specified an
      "idempotent upsert" here, and that was a bug: an upsert inserts when the row is absent, so a
      stale tab that missed a delete and then drags the ghost would have its glide broadcasts
      *resurrect* the figure in the tab that had correctly deleted it. Ignoring an unknown id kills
      resurrection at the root — a move can no longer conjure a figure into existence. There is no
      fallback, no "add it just in case", and no logging branch.
    - `"delete"` — remove every figure with `msg.Id` from `figures`, and if `selectedId` equals
      `msg.Id`, clear `selectedId`. Removing an unknown figure is a silent no-op, which is what makes
      this kind naturally idempotent (D-53). Clearing the selection is a *consequence* — it greys out
      the Delete button (D-58) — not a notification.

    Add no `default` branch that throws: a fifth kind does not exist (D-53 fixes the contract at four)
    and this switch is fed only by this app's own `SyncMessage` factories.

    RENDER NOTHING NEW. A figure that arrived via a broadcast is visually IDENTICAL to one drawn
    locally — no flash, no tint, no fade, no badge, no opacity difference (05-UI-SPEC.md § Remote-
    Origin Figures, UI-05-02). Do not touch the `@foreach`, do not add a parameter to `FigureShape`,
    and do not add a field tracking a figure's origin. The absence of an arrival cue is the decision,
    not an omission. Red is reserved for local selection only.

    Give each new method an XML-doc in this file's established style, citing the decisions:
    `HandleRemoteMessage` should record that it runs on the publisher's thread (hence `InvokeAsync`),
    that the discard is blanket per D-54 and why the narrow filter is wrong, and why
    `ObjectDisposedException` is swallowed. `ApplyMessage` should record that `move`/`rollback` are
    update-only per D-40 and what the ignored-unknown-id `return` prevents.

    THE THROTTLE — extend `OnWrapperPointerMove`, appending BELOW the existing final line

    Read `Environment.TickCount64` into a local, and if `dragMoved` is true AND the elapsed

    `Notifier.Publish(userId, SyncMessage.Move(dragFigureId!.Value, dragCurrentBox, _sender));`

    USE `Environment.TickCount64`, not `DateTime.UtcNow` — a monotonic millisecond counter is the
    textbook-correct instrument for elapsed time and cannot be skewed by a clock adjustment. This is
    engineering discretion; the ADR set is silent on the instrument and only pins the 50ms value
