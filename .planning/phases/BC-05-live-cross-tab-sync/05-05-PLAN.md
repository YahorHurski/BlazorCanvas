  <how-to-verify>
    SETUP
    1. `docker compose up -d` and wait for the container to be healthy.
    2. `dotnet run --project src/BlazorCanvas`.
    3. Open the app and log in. Open a SECOND TAB in the same browser at the same URL — it should be
       already authenticated with no second login (D-26). Put the two tabs side by side, ideally on
       two monitors. These are "tab A" and "tab B" below.

    CRITERION 1 — appear, disappear, and closing a tab
    4. In tab A, draw one of each shape: a line, a rectangle, a circle, a triangle. Each one appears
       in tab B as you finish it.
    5. In tab B, select a figure and click Delete. It disappears from BOTH tabs.
    6. Close tab B. In tab A, draw a figure, drag one, delete one. All three still work, and no error
       bar appears at the bottom of tab A. Re-open tab B for the rest of this script.
       (This is the leaked-subscriber / `ObjectDisposedException` gate.)

    CRITERION 2 — THE GLIDE. This is the sentence the whole project was built around.
    7. In tab A, press a figure and drag it slowly across the canvas over several seconds, WATCHING
       TAB B. The figure must MOVE in tab B while you are still dragging — continuously, in visible
       small steps. It must NOT sit still and then jump when you release.
       - Stepped, ~20fps motion is CORRECT and intended (UI-05-01). It is not a defect. Smooth
         easing would be the defect.
    8. Release. The figure in tab B lands in EXACTLY the same place as in tab A — it does not stop
       short by a few pixels and it does not drift or settle after you let go.
    9. Repeat with a circle: it must glide as a circle and never render as an oval.

    CRITERION 2 (continued) — EXACTLY ONE UPDATE PER DRAG, measured against the real database
    10. Turn on statement logging:
        `docker compose exec -T db psql -U postgres -d canvas -c "ALTER SYSTEM SET log_statement = 'all'; SELECT pg_reload_conf();"`
    11. Drag ONE figure slowly across the whole canvas for at least 3 seconds, then drop it.
    12. `docker compose logs db --since 1m | grep -c "UPDATE figures"` → this must return **1**.
        If it returns a large number, the glide is writing intermediate positions to Postgres — the
        ~100x write amplification D-11 forbids, and a failure of this criterion.
    13. Turn logging back off:
        `docker compose exec -T db psql -U postgres -d canvas -c "ALTER SYSTEM SET log_statement = 'none'; SELECT pg_reload_conf();"`

    CRITERION 3 — what must NOT cross
    14. In tab A, with a SHAPE tool armed, press and drag slowly WITHOUT releasing. Tab B shows
        NOTHING during the gesture. Release — only then does the finished figure appear in tab B.
        (Draw previews are never broadcast — D-35.)
    15. In tab A, click a figure to select it (red outline). In tab B that same figure stays BLACK.
        Selection is local and never travels (D-31).
    16. In tab A, drag a figure. Tab A's own figure follows the cursor smoothly and never stutters,
        jumps back, or fights itself. (That is the echo filter — a tab never reacts to its own
        broadcast.)
    17. Every figure in tab B looks IDENTICAL to a locally drawn one — black 2px outline, white fill.
        There is no flash, no tint, no fade-in, no badge, no opacity difference marking it as having
        arrived from elsewhere. **The absence of any arrival cue is a PASS, not a gap** (UI-05-02).

    CRITERION 3 (continued) — the blanket mid-drag discard (D-54). OPTIONAL — best effort only.
    18. This one is genuinely hard to observe by design: with one mouse you cannot act in a third tab
        while holding a drag, which is precisely the premise D-54 rests on. Its primary proof is the
        source gate in plan 05-03, which already passes.
        If you want to see it: open the app on your PHONE on the same network, logged in as the same
        user (trivially possible — there is no real auth). Hold a drag in tab A with the mouse and, on
        the phone, tap-draw a figure. Tab A must NOT show the phone's figure while your drag is in
        progress. After you drop, it is still missing — and it stays missing until F5.
        **That permanent loss is the CORRECT, decided behaviour** — D-54 records it as an accepted
        cost, deliberately traded for the simpler rule. It is not a bug. If you skip this step, say so.

    CRITERION 4 — the ghost, removed in silence
    19. Both tabs showing the same rectangle. In tab B, select it and click Delete — it disappears
        from both. Now press F5 in tab A ONLY, then in tab B delete a DIFFERENT figure while tab A is
        showing it, and immediately — in tab A — drag that same figure and drop it.
    20. In tab A the figure SILENTLY disappears at the drop. There is **no error, no toast, no
        banner, no modal, no prompt, no "this was deleted elsewhere" message, and no confirmation.**
        **The silence is the specification** (D-10, 05-UI-SPEC.md § Silent Ghost Removal) — a zero-row
        UPDATE is not an error, it is expected staleness, and the figure vanishing IS the feedback.
        Reporting the silence as missing feedback would mean filing the locked contract itself as a
        defect.
    21. If that figure was selected in tab A, the Delete button greys out as it vanishes.

    CRITERION 5 — the failed save, and the screens that must not lie
    22. Both tabs showing the same figure, well away from any edge. Stop the database:
        `docker compose stop db`
    23. In tab A, drag that figure well across the canvas — TAB B GLIDES IT along with you, because
        the broadcasts are in memory and do not touch the database. Drop it.
    24. Tab A shows a modal reading, exactly:
        *The change could not be saved. The canvas will be reloaded from the database.*
        It has one button, labelled **OK**, and nothing else — no ×, no Cancel.
    25. **TAB B SNAPS THE FIGURE BACK TO ITS ORIGINAL POSITION, AND SHOWS NO MODAL.** This is the
        criterion. If tab B keeps the new position, the rollback broadcast is missing and every open
        screen is lying about the database.
    26. In tab A, press Escape — the modal stays. Click the dimmed backdrop — the modal stays. Only OK
        closes it (UI-05-03).
    27. `docker compose start db`, wait a few seconds, then click OK. Tab A reloads from PostgreSQL and
        shows the figure at its ORIGINAL position. Both tabs now agree with the database.
    28. Neither tab ever showed a Blazor error bar, and both still work: draw a figure in tab B and it
        appears in tab A (D-45 — the app stays alive; the circuit did not crash).

    CRITERION 5 (continued) — the autofocus clause. CHECK THIS HONESTLY.
    29. Trigger the modal once more (stop the database, drop a drag). Without clicking anything, press
        **Enter**. If the canvas reloads, the OK button had focus and UI-05-04's autofocus clause
        holds — say so. If nothing happens, autofocus did not fire; say so plainly. The next task
        carries the decision. Restart the database afterwards.

    CROSS-USER ISOLATION — the one real security surface this phase adds
    30. Open a PRIVATE/incognito window and log in as a DIFFERENT username. Arrange it beside tab A.
    31. Draw and drag figures in tab A. The other user's window shows **nothing** — no figure appears,
        nothing moves, nothing flickers. Then draw in the other user's window; tab A shows nothing.
    32. Each canvas contains only its own owner's figures.

    REGRESSION — Phases 1-4 must still be intact
    33. All four shapes still draw: line/rectangle/triangle corner-to-corner, circle centre-out.
    34. With a shape armed, drag on top of an existing figure — it DRAWS a new figure rather than
        moving it. A small circle can still be drawn inside a big rectangle (D-30).
    35. Drag a figure into the right edge — it stops and still slides freely up and down (D-36).
    36. A click (<3px) still only selects and writes nothing; a drag still selects and stays selected
        after the drop (D-48).
    37. Press F5 in both tabs. Every figure is still there, in the same drawing order, with the same
        overlap (D-39). The definition of done: *everything survives a refresh.*

    THREE THINGS THAT ARE PASSES, NOT GAPS — do not report them as findings
    - Remote figures have no arrival cue of any kind (UI-05-02 — the contract's central clause).
    - Silent ghost removal has no user feedback whatsoever (D-10/D-40 — the silence is the spec).
    - The glide steps visibly at ~20fps rather than easing smoothly (UI-05-01 — correctness of the
      last frame was chosen over smoothness of the intermediate ones).

  <context>
    **Skip this checkpoint entirely — record "not applicable, autofocus works as specified" and move
    on — if step 29 above showed that pressing Enter reloaded the canvas.** This decision only exists
    if the locked mechanism turned out not to work.

    What was found: 05-UI-SPEC.md's § Save-Failure Modal asserts that "No JavaScript is required: the
    `autofocus` attribute on the button inside a native `<dialog>` is markup-only." That is reliably
    true for elements present when the document loads. This modal, however, is inserted by Blazor into
    an already-loaded document, and browsers do not consistently honour `autofocus` in that case.

    Two related facts, both good news, so the scope of this decision stays small:
    - The rest of UI-05-04 holds exactly as locked — native `<dialog>`, login-card tokens, `#1D4ED8`
      OK button, dimmed backdrop, 4px radius, centred, no reflow.
    - UI-05-03's "no Escape, no backdrop dismiss" comes FREE from the same no-JavaScript path: a
      `<dialog>` opened by the `open` attribute is non-modal, so the browser gives it no light-dismiss
      at all. The constraint that costs us autofocus is the same one that gives us forced dismissal.

    Why you are being asked rather than told: this is precisely the trade you have made three times
    already — D-33 (a toolbar Delete button instead of the Delete key), D-37 (no `setPointerCapture`),
    D-57 (no Escape to abandon a draw). Each time you accepted a worse interaction rather than write
    JavaScript. It would be wrong to assume the fourth answer.
  </context>

  <options>
    **option-a — Accept it. No autofocus; amend UI-05-04.**

    - Pros: Zero JavaScript, in letter and in spirit — consistent with D-33, D-37 and D-57's
      precedent. No code change at all. The OK button is still the only focusable element in the
      dialog and is one click away; the modal is unmissable and the flow is still forced.
    - Cons: Enter and Space do not work until the user clicks or Tabs to the button. UI-05-04's
      autofocus clause and its "the trap is structural" rationale would need amending in
      05-UI-SPEC.md. A keyboard-only user must press Tab once first.

    **option-b — Use Blazor's `ElementReference.FocusAsync()`.**

    - Pros: UI-05-04 is satisfied exactly as locked. We author no JavaScript file and write no JS —
      it is a first-party .NET API called from C#. PROJECT.md scopes the constraint to "JS *we write*
      to solve an app problem", and framework-shipped interop is explicitly out of that scope (the
      same clause that permits `blazor.web.js`). Roughly three lines in `Home.razor`.
    - Cons: It does reach the browser through the framework's JS interop channel underneath, so it is
      one step closer to the line than anything else in this app. It needs an `ElementReference` and
      an `OnAfterRenderAsync` guard, which is the first lifecycle-timing code in this component.

    **option-c — Keep the `autofocus` attribute anyway, as best-effort.**

    - Pros: No code change. Works on whichever browsers do honour it, degrading silently elsewhere.
    - Cons: The worst of both: the contract says focus is guaranteed while the behaviour varies by
      browser, so the app's stated design and its real behaviour disagree — exactly the kind of quiet
      divergence this decision log has been ruthless about. Adds an untestable claim.
  </options>
