<objective>
Write the two standing guards that make this storage model's silent failure modes loud: the `bbox_*`
cache is proven to agree with a fresh recompute from `geometry` for every row in the table, and hostile
`geometry` and `style` are proven to be refused at the database boundary rather than merely at the
gateway's return value.

Purpose: D-67 accepted a derived cache on the explicit condition that a test recomputes it, and
VALID-01/02/03 are only as good as the write path that honours them. Both of these fail silently by
nature — a stale bbox shows up as a figure stopping at the wrong edge, and a hostile payload that slips
through shows up as an SVG attribute — so neither is caught by anything else in the suite.

Output: two test files. No production code, no schema change, and nothing wired into the running
application.
</objective>

geometry text through `TryParseGeometry`; call `BoundsOf`; and assert all four stored values equal the
recomputed ones **exactly**, with a failure message naming the figure id and which of the four values

not only this test's rows, deliberately, so it is a standing invariant rather than a self-check; that
the comparison is exact because both sides are the same `double` from the same function over the same
parsed numbers, and a tolerance would hide a real round-trip defect; and that visiting order does not
matter because the invariant is per-row.

**C. Fail-first proof.** Inside a transaction that is always rolled back, take one of group A's rows
and `UPDATE v11.figures SET bbox_w = bbox_w + 1 WHERE id = @id`, then run the same comparison over
that single row and assert it now **fails** — capture the comparison in a small local function
returning a list of mismatches, so this test can assert the list is non-empty rather than expecting an
xUnit failure. Repeat for `bbox_x`. Roll back. Comment that without this, a green result from group B
would be indistinguishable from a comparison that never actually compared anything — this is what
makes the guard evidence.

**D. Zero-extent and constraint edges (MODEL-07).** Assert the horizontal line's stored `bbox_h` is
exactly `0` and the vertical line's `bbox_w` is exactly `0`, and that both pass group B's comparison.
Then, inside a rolled-back transaction, attempt a raw INSERT with `bbox_w = -1` and assert it is
refused with `PostgresException.SqlState` equal to `PostgresErrorCodes.CheckViolation` (`23514`) and
the constraint name `bbox_is_positive`. Assert also that all four `bbox_*` columns are `NOT NULL` by
attempting a raw INSERT omitting `bbox_h` and asserting SQLSTATE `23502`. Comment that `>= 0` rather
than `> 0` is deliberate: both zero-extent lines were legal in v1.1.

**E. Duplicates stay independent (TEST-03 adjacency).** Insert two figures with identical geometry at
identical positions and assert both rows exist with distinct ids, that both carry the same bbox values,
and that group B's comparison holds for each independently. Comment that nothing is deduplicated,
interned or shared — two identical figures are two figures.

**F. The cache survives a move.** Move one of group A's figures and re-run group B's comparison for
that row. Assert it still agrees. Comment that this is only true because the stored extent is local:
an absolute cache would have gone stale the instant `MoveAsync` ran, which is precisely why MODEL-01's
"a move writes only `x`, `y`" and D-67's "a pure function of geometry" are the same decision seen from
two sides.

recorded in the earlier plans and apply here too.

- **PA-21 — TEST-03's three guards are split across two plans.** The bbox agreement guard and the
  hostile-input guard are here; the `z`-collision guard is `ZCollisionRetryTests` in 10-03, delivered
  alongside the retry mechanism it proves. Splitting them keeps the proof next to the code it
  defends and lets 10-03 and this plan run in different waves. All three exist and pass, which is what

- **PA-22 — The bbox guard runs over the whole table, including rows other tests left behind.**
  That makes it a standing schema invariant rather than a self-check, and it means a future write path
  that forgets the cache is caught by an existing test rather than needing a new one. The cost is that

  message names the offending row.
- **PA-23 — Group E asserts a weakness on purpose.** Pinning "the database accepts `{"r":-5}`" looks
  like asserting a bug. It is asserting D-60's accepted cost, so that the day someone removes the
  gateway the failure explains what was surrendered rather than appearing as an unrelated red test.
  **Overturn by dropping the assertion and relying on the decision log alone.**
- **PA-24 — TEST-02's retirements are not in this phase.** The inscribed-square round-trip, the
  line-normalisation landmine test and the 32-case CHECK-mirror matrix still guard the old table,
  which is still live and still in use by the running application. Retiring them belongs with the
