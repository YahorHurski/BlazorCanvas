Dispatch on the type literal with an ordinal, case-sensitive comparison against the four v1.1
literals, and apply exactly these formulas — nothing more:

- `rectangle` — position `(x1, y1)`; geometry `new RectangleGeometry(x2 - x1, y2 - y1)`.
- `circle` — position `(x1, y1)`; geometry `new CircleGeometry((x2 - x1) / 2.0)`. The v1.1
  `circle_is_a_circle` constraint guaranteed `(x2 - x1)` is positive and even, so the radius is a
  whole number; do not round it, and do not add a guard that would change the value.
- `triangle` — position `(x1, y1)`; geometry `new TriangleGeometry([ new LocalPoint((x2 - x1) / 2.0,
  0), new LocalPoint(0, y2 - y1), new LocalPoint(x2 - x1, y2 - y1) ])`. Comment that this is a literal
  transcription of the v1.1 renderer: apex at the top, horizontally centred; base along the bottom
  edge, left vertex then right vertex.
- `line` — let `y = Math.Min(y1, y2)`; position `(x1, y)`; geometry `new LineGeometry([ new
  LocalPoint(0, y1 - y), new LocalPoint(x2 - x1, y2 - y) ])`. Comment that both points are carried
  across in their original order relative to the new origin, so the diagonal direction lives in the
  data; this is the v1.1 normalisation landmine (D-41) staying defused (D-60), and any code that
  sorted, swapped or canonicalised this pair would silently produce the opposite diagonal.

For any other type literal — including a differently-cased one, empty, whitespace or null — throw
`ArgumentOutOfRangeException` naming the parameter but **not** echoing the value, so an unexpected
literal fails the whole migration loudly instead of producing a guessed figure. Add a comment saying
that a migration that skips a row it does not understand is exactly the silent data loss MIGR-01
forbids.

Add one further guard with the same posture: if the row's coordinates could not have been produced by
the v1.1 CHECK constraints for its type — a rectangle or triangle with `x2 <= x1` or `y2 <= y1`, a
circle with a non-positive or odd side or unequal sides, a line with `x2 < x1` or with both
coordinates equal — throw `InvalidOperationException`. Comment that these rows cannot exist in a
database the v1.1 constraints policed, so encountering one means the input is not a v1.1 database and
the migration must stop rather than invent an answer.

**Hard rules for this file**, each worth a comment:
- No rounding, no `Math.Round`, no clamping, no `CanvasBounds` reference. Position is copied, extents
  are subtracted, and that is all.
- No sorting, swapping, `Math.Min`/`Math.Max` over a point pair, or any other canonicalisation. The
  only `Math.Min` in the file is the line's origin, and it computes an origin — it does not reorder
  anything.
- No JSON. `Convert` returns typed records; serialisation belongs to `IShapeDefinition.ToJson`, which
  10-04 reaches through `FigureInputGateway`. Producing JSON here would create a second write path
  around the Phase 9 trust boundary.
- No `z`, no style, no bbox. Those are the caller's business.

Create `src/BlazorCanvas/Data/V11/V11DeterministicId.cs`, namespace `BlazorCanvas.Data.V11`,
`public static class V11DeterministicId`.

The purpose, stated in a file-level comment: D-62 says the old-id-to-uuid mapping is computed once and
deterministically. Deriving the uuid from the legacy id rather than generating a random one makes the
whole migration re-runnable — the second run computes the same ids, so its inserts conflict and become
no-ops instead of duplicating every figure — and lets the replay test in 10-04 name expected ids
instead of discovering them.

Two members:

- `public static Guid ForCanvas(int legacyUserId)`
- `public static Guid ForFigure(int legacyFigureId)`

Both build an RFC 9562 **version 8** UUID — the version reserved for custom, vendor-defined layouts —
from a fixed 16-byte template:

- Bytes 0–5: a per-kind constant prefix. Use two distinct, hard-coded 6-byte constants, one for
  canvases and one for figures, declared as named `static readonly byte[]` fields with a comment that
  they must never change, because changing either would re-key every already-migrated row.
- Byte 6: the high nibble is the version, set to `0x8`; the low nibble is part of the payload.
- Byte 8: the top two bits are the RFC variant, set to `0b10` (i.e. `(b & 0x3F) | 0x80`); the
  remaining bits are payload.
- Bytes 12–15: the legacy id as a big-endian 32-bit value.
- All remaining payload bytes: zero.

Then construct the `Guid` from those 16 bytes using the constructor overload that treats the input as
**big-endian** (`new Guid(bytes, bigEndian: true)`), so the textual form of the uuid reads in the same
byte order on every platform. Add a comment that the little-endian overload would produce a different
string for the same input and that the choice is therefore load-bearing for the replay test's expected
values.

Reject a negative `legacyUserId` or `legacyFigureId` with `ArgumentOutOfRangeException` — v1.1 ids
come from identity columns and are always positive, and a negative input means the caller has read the
wrong column.

Add a comment recording, explicitly, that these identifiers are **not** capability tokens: they are
predictable by design, and every read and write path filters by `canvas_id` so that predictability is
never the thing standing between one user and another user's figures. A future change that starts
treating a figure uuid as an unguessable secret must first replace this derivation.

Do not add a reverse function that extracts the legacy id back out of a uuid. It is derivable, and
somebody will eventually want it for a one-off audit query, but a supported reverse mapping invites
code that treats the legacy id as still meaningful after the migration, and D-62's whole point is
that it is not.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| v1.1 row → `LegacyFigureConversion.Convert` | Rows read from a real user database. The values are user data, and an exception message about them is an output channel. |
| `Convert` output → `IShapeDefinition.ToJson` → `FigureInputGateway` (10-04) | The conversion deliberately stops at typed records so the serialise-and-validate step stays the Phase 9 trust boundary rather than being duplicated here. |
| `V11DeterministicId` output → `figures.id` / `canvases.id` | These become primary keys and, in Phase 11, values broadcast to other tabs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-10-08 | Tampering | a conversion that silently alters a figure | high | mitigate | Every formula is asserted against the manifest's transcribed expected values, the diagonal-direction case is asserted separately in both directions, and the "no tidying" theory pins the tempting cases (off-canvas, minimal size, duplicate coordinates). Also authored as a `test`-tier prohibition. |
| T-10-09 | Tampering | an unrecognised row silently skipped | high | mitigate | An unknown type literal throws `ArgumentOutOfRangeException` and an impossible coordinate pair throws `InvalidOperationException`; neither returns a guessed figure. 10-04 lets the throw abort the whole migration transaction. |
| T-10-10 | Information Disclosure | user coordinates in an exception message | medium | mitigate | Exceptions name the parameter and never the value; group E asserts no input coordinate appears in any message. ## Flagged assumptions

Phase-local numbering continues from `10-01-PLAN.md`. The cross-cutting assumptions PA-01…PA-08 are
recorded there and apply here too.

- **PA-09 — Migrated ids are deterministic RFC 9562 version-8 uuids derived from the legacy id;
  figures created after the migration get a random `Guid.NewGuid()`.** D-62's accepted cost says the
  mapping is "computed once, deterministically". Deriving rather than generating is what makes the
  migration re-runnable and what lets 10-04 assert that a second run changes nothing. The mixed scheme
  is deliberate: only migrated rows need determinism. **Overturn by requiring random uuids plus a
  persisted legacy-id-to-uuid mapping table.**
- **PA-10 — `Convert` throws on a row the v1.1 constraints could not have produced.** The alternative
  readings are "skip it" (silent data loss, which MIGR-01 forbids) and "coerce it" (a figure the user
  never drew). Failing loudly is the only one compatible with "proven lossless, not assumed".
  **Overturn by naming what a non-v1.1 row should become.**
- **PA-11 — `Convert` returns typed records, not JSON.** Serialisation stays with
  `IShapeDefinition.ToJson` behind `FigureInputGateway`, so the migration writes through the same
  single trust boundary as every other write — the constraint 09-06's summary recorded for this phase.
  **Overturn by accepting a second serialisation path used only by the migration.**
