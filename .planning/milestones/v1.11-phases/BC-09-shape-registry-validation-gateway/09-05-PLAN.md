<objective>
Prove the three claims the ROADMAP makes about this phase that no single implementation file can
prove on its own:

1. **Equivalence** — the new registry reproduces v1.1 draw behaviour exactly, across the same
   invariant grid the existing suite already uses. This is the mechanical form of the milestone's
   governing invariant that any user-observable difference from v1.1 is a defect.
2. **Extensibility (ROADMAP success criterion 3)** — a fifth shape type the application does not ship
   registers and round-trips through the existing abstractions, with zero changes to any other type's
   code and no schema involved.
3. **Point-list primacy (ROADMAP success criterion 2)** — line and triangle geometry is genuinely a
   point list and genuinely not derived from a bounding box, proven by geometries that share a box but
   differ, and by a downward- and a sideways-pointing triangle built with no new formula.

Purpose: these are contract tests, not implementation. They are what a later phase runs into if it
quietly reintroduces the old model.

Output: four test files. **This plan creates and modifies zero production files** — that constraint
is itself the SHAPE-03 evidence and is enforced as an acceptance criterion.
</objective>

deliberately reproduces the v1.1 arithmetic, so any drift is a real behavioural change and must fail.

inexact case means the milestone's zero-user-visible-change invariant is at risk and needs a decision.

**Part A — the generic contract, over every registered definition.** Build a registry from

`registry.All`. For each definition: build a representative geometry through `FromGesture` with a

- `BoundsOf(reparsed)` equals `BoundsOf(original)` exactly on all four components,

**Part B — the primacy assertion.** This is the guard against a later phase re-deriving line or
triangle geometry from a bounding box. For each of line and triangle, construct two DIFFERENT
geometries of the same type that share an IDENTICAL `Bbox`, then assert their `ToJson` outputs
differ:
- line: points `(0,0)` then `(100,80)`, versus points `(0,80)` then `(100,0)` — both bound to
  `(0, 0, 100, 80)`;
- triangle: points `(50,0), (0,80), (100,80)` (apex up), versus points `(0,0), (100,0), (50,80)`
  (apex down) — both bound to `(0, 0, 100, 80)`.
Assert in each pair that `BoundsOf` is equal AND `ToJson` differs. Comment plainly what a failure
means: if these collapse to the same value, the bounding box has become the source of truth again and
the point list has been demoted — which is exactly the regression the assumption-delta decision
recorded in `09-01-PLAN.md` exists to prevent.

**Part C — ROADMAP success criterion 2, literally.** Using only the shipped `TriangleShape` obtained
from the registry as an `IShapeDefinition`:

- build a **sideways-pointing** triangle from `{"points":[[0,0],[80,50],[0,100]]}` and assert it

different code, which is the whole claim of success criterion 2.

**Part D — bounds are computed from geometry, not stored.** For a triangle, assert that mutating the

mutating in place) yields a different `BoundsOf` result, and that `BoundsOf` called on the same
geometry instance twice returns equal values. Together these pin that bounds are a pure function of
the geometry and are not cached or carried alongside it.

## Planner Assumptions

- **PA-25 — The fifth shape is a pentagon, lives in the test project, and nests its geometry record
  privately inside its own class.** Nesting is what makes "a new figure type costs exactly one class"
  literally true rather than "one class plus a record in a shared file". A pentagon was chosen because
  it is a point-list type structurally unlike all four shipped types, and because it is a real v1.2
  backlog item, so the proof is not artificial.
- **PA-26 — This plan changes zero production files, deliberately.** That is not incidental tidiness:
  it is the mechanical evidence for ROADMAP success criterion 3's "with zero changes to any other
  type's code", and it is enforced as an acceptance criterion rather than asserted in prose.
- **PA-27 — Equivalence comparisons use exact double equality, never an epsilon.** The new arithmetic
  is a deliberate transcription of the v1.1 arithmetic (PA-19), so exactness is achievable and any
  drift is a genuine behavioural change that must be surfaced as a decision, not absorbed by a
  tolerance.
- **PA-28 — The primacy assertion is "same box, different JSON", not a structural inspection.** It
  tests the property that matters (the box cannot reconstruct the shape) rather than the current
  implementation shape, so it stays valid across refactors and only fails on the actual regression.

The full cross-cutting assumption list (PA-01 … PA-08) is in `09-01-PLAN.md`.
