<assumption_delta_decision>
**Signal:** SHAPE-02 moves line and triangle geometry from bounding-box-derived to a point list, and
SHAPE-03 makes shape types registrable (a fifth type costs exactly one class). The detector's raw
`pluralization` hit on the word "also" inside the ROADMAP database-dump admonition is a false
positive and is not the basis for this record.

**Primary noun:** the **ordered point list** (`IReadOnlyList<LocalPoint>`) is the primary geometry
representation for line and triangle. It is the stored source of truth.

**Decision: `promote`.** The bounding box is demoted to a *derived* value — a computed output of
`IShapeDefinition.BoundsOf` — and is never the stored source of truth for a line or a triangle. The
old `BlazorCanvas.Geometry.Box` remains in the tree untouched for the still-running v1.1 code path,
but it is not the model any new code builds on and no `BlazorCanvas.Shapes` type references it.

**Rationale:** ROADMAP success criterion 2 states this explicitly ("Line and triangle geometry is
represented as a point list, **not derived from a bounding box**"), and SHAPE-02 gives the forward
reason: vertex editing later needs no data conversion. D-60 records the same reversal and notes it
also defuses landmine #2 (the D-41 line-normalisation arm whose failure silently produced the
opposite diagonal).

**Invariant test: accepted, and planned.** Plan 09-05 carries `PointListPrimacyTests` — a contract
test asserting, for every registered shape definition, that geometry round-trips through the typed
record/JSON path with vertex order preserved and that `BoundsOf` is computed *from* that geometry.
A future phase that reintroduces bounding-box-derived line or triangle geometry turns it red
immediately, so the regression cannot land silently.
</assumption_delta_decision>

adds W/2 per side at render time, which is why D-66 caps stroke width at 64); it is a derived cache,
never a source of truth; and Phase 10 stores `x + Bbox.X`, `y + Bbox.Y`, `Bbox.W`, `Bbox.H` in the
`bbox_*` columns.

`TriangleGeometry.cs` — `public sealed record TriangleGeometry(IReadOnlyList<LocalPoint> Points) :

`RectangleGeometry.cs` — `public sealed record RectangleGeometry(double W, double H) :

`CircleGeometry.cs` — `public sealed record CircleGeometry(double R) : IFigureGeometry;` Local origin
is the TOP-LEFT of the circle's bounding square, not the centre — this is what the D-60 migration
formula fixes (`x, y = x1, y1` with `{"r": (x2-x1)/2}`), so the centre is at local `(R, R)` and the
extent is (0, 0, 2R, 2R). State this on the type; getting it wrong moves every migrated circle by one
radius. A circle stores one radius and never `rx`/`ry` — an ellipse is a separate type with its own
button (D-60).

boundary.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser circuit → `IShapeDefinition.TryParseGeometry` | Client-supplied `geometry` JSON crosses here. Under D-60 the database has no geometric CHECK constraints any more, so this is the only place geometric correctness can be established. |
| browser circuit → `ShapeRegistry` lookup key | The `type` name accompanying a figure is client-supplied and is used as a dictionary key. |
| `IShapeDefinition.BoundsOf` → `figures.bbox_*` → Phase 11 edge clamp | A wrong or non-finite bound propagates into the clamp that keeps a figure on the canvas. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-09-01 | Spoofing | `ShapeRegistry.TryGet` / `Get` | medium | mitigate | Ordinal, case-sensitive lookup; null/empty/whitespace/unknown all reject with the out parameter set to null. No fallback or default definition exists, so a forged `type` cannot resolve to a shape that then re-serialises under a different name. Proven by `ShapeRegistryTests`. |
| T-09-02 | Tampering | `ShapeRegistry.Register` | medium | mitigate | A duplicate `Name` throws `ArgumentException` and the first registration wins. Without this, a later-registered definition could silently take over an existing type's parse and bounds rules. Proven by the duplicate-registration test asserting the surviving instance by reference. |
| T-09-03 | Tampering | `GeometryJson.TryReadFiniteDouble` | high | mitigate | NaN and both infinities are rejected at the single read primitive. | T-09-04 | Tampering | `GeometryJson.TryReadPointList` | medium | mitigate | Exact required-count check rejects empty, single-element and over-long arrays, and each element must be a 2-element array of finite numbers. | T-09-05 | Tampering | `GeometryJson` number writing | medium | mitigate | All numbers go through `Utf8JsonWriter`, which is culture-invariant by construction. ## Planner Assumptions

No CONTEXT.md, RESEARCH.md, SPEC.md or PATTERNS.md exists for this phase (the user chose to plan
without discuss-phase). The following design choices were made by the planner from ROADMAP success
criteria, REQUIREMENTS.md, `docs/DECISIONS.md`, `docs/DATA-MODEL-v1.11-DRAFT.md`, and the actual code
in `src/BlazorCanvas/Geometry/`. They are listed so they can be overridden at plan review.

- **PA-01 — New namespace `BlazorCanvas.Shapes` in a new folder, not additions to
  `BlazorCanvas.Geometry`.** ROADMAP requires the existing geometry classes to be untouched and green.
  Putting new code in a separate folder makes "untouched" mechanically checkable with
  `git diff --name-only` instead of a judgement call, and keeps the v1.1 and v1.11 models visibly
  distinct while both exist. *Alternative if overridden:* add the new types alongside the old ones in
  `BlazorCanvas.Geometry`.
- **PA-02 — `IShapeDefinition` members are typed with `IFigureGeometry`, not `object`.** The draft's
  interface sketch used `object` for geometry. A marker interface costs one empty file, declares no
  members (so a new shape still costs exactly one class), and lets the compiler catch a geometry/type
  mismatch that `object` would surface only at runtime.
- **PA-03 — `TryParseGeometry` (Try pattern) rather than a throwing `ParseGeometry`.** VALID-02
  requires rejection to be SILENT, matching v1.1's silent rejection of a degenerate draw. A bool
  return makes silence the default and removes the exception-message channel for attacker-controlled
  text.
- **PA-04 — `Resize` is NOT on the interface in Phase 9.** The draft's interface sketch lists
  `Resize(object geometry, double sx, double sy)`, but no ROADMAP success criterion, no phase
  requirement (SHAPE-01/02/03), and no v1.11 or v1.2 consumer uses it, and its circle semantics are
  under-determined in the draft ("one side is taken" — which side is not stated). Shipping an
  unrequired member with undetermined semantics would also make the SHAPE-03 "a new shape costs
  exactly one class" proof assert against dead surface. Adding a member to `IShapeDefinition` later is
  a compile-time break across exactly the four shipped implementations plus the test-only fifth — loud
  and cheap, never silent. **Flagged for user override:** if you want `Resize` in Phase 9, say so and
  the circle's non-proportional-resize rule needs one sentence of decision.
- **PA-05 — Two distinct point types (`LocalPoint` and `CanvasPoint`).** D-59's whole design is that
  position and shape are different things. Two record structs cost nothing and make mixing them a
  compile error rather than a figure rendered a radius away from where it belongs.
- **PA-06 — `ToJson` returns `string`, not `JsonNode`.** It makes the VALID-01 assertion ("the stored
  value never contains what the client sent") a plain string comparison, and Npgsql maps `string`
  directly onto a `jsonb` column in Phase 10.
- **PA-07 — Ordinal, case-sensitive registry lookup.** `figure_types.name` is exact lowercase; a
  case-insensitive lookup would let a client-supplied `"Circle"` resolve and then be stored as
  `"circle"`, quietly normalising input the gateway is supposed to reject.
- **PA-08 — Point-list records use `IReadOnlyList<LocalPoint>` and callers compare `Points`
  element-wise.** Record `==` on a list member is reference equality. A hand-rolled structural
  `Equals` would be a second source of truth for equality; comparing the sequence in tests is
  explicit and costs nothing.
