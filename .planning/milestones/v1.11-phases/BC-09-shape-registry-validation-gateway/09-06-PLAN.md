<objective>
Build the single application-wide validation gateway that D-66 requires: one choke point through
which every client-supplied `geometry` and `style` payload must pass before it can be stored or
rendered. It parses into typed records, validates, and re-serialises from those records, so what the
client sent is never what gets stored (VALID-01). It rejects geometry the database no longer rejects
(VALID-02), and it bounds style values so no unescaped value can reach an SVG attribute (VALID-03).

Purpose: D-60 deliberately surrendered the geometric CHECK constraints — `circle_is_a_circle`,
`box_is_a_box`, `line_is_a_line` are gone and `{"r": -5}` becomes writable. That is the accepted cost
of the storage model, and this gateway is what pays it. The draft is explicit that the gate must be
**one for the whole application**, not a copy at every write site.

Output: two production files and one test file. Nothing is wired in: Phase 10's persistence layer and
Phase 11's sync receiver are what will route through it. No database, no schema, no change to the
running application.
</objective>

`public sealed class FigureInputGateway` taking a `ShapeRegistry` through its constructor (throw
`ArgumentNullException` on null). No static state, no default registry — the owner supplies one, which
keeps the test-only fifth shape from ever being reachable here.

Expose one primary member:

`public bool TryValidate(string? typeName, string? geometryJson, string? styleJson, out
ValidatedFigureInput? result)`

Sequence, in this exact order — each step is a gate the next one depends on:

1. **Resolve the type.** `registry.TryGet(typeName, out var definition)`. On failure return false. A
   forged or unknown type never falls back to a default definition.
2. **Parse the geometry JSON text.** `JsonDocument.Parse` with
   `JsonDocumentOptions { MaxDepth = 32, AllowTrailingCommas = false, CommentHandling =
   JsonCommentHandling.Disallow }`, inside a try/catch for `JsonException` and `ArgumentException`
   that returns false. Null, empty and whitespace-only input return false without parsing. Dispose the
   document.
3. **Parse the geometry into a record.** `definition.TryParseGeometry(document.RootElement, out var
   geometry)`. On failure return false.
4. **Check drawability.** `definition.IsDrawable(geometry)`. On failure return false. This is the step
   that preserves v1.1's silent rejection of a degenerate draw (VALID-02) — a zero-length line, a
   zero-area rectangle, coincident or collinear triangle points.
5. **Only now compute bounds.** `definition.BoundsOf(geometry)`. Add a comment: bounds are computed
   after parse and drawability precisely so no hostile geometry can reach a bounds computation, which
   is ROADMAP success criterion 4's exact wording. As a defensive post-condition, return false if the
   resulting `Bbox` has a non-finite or negative `W` or `H` — this should be unreachable given step 3
   and 09-04's parse-time rejection of non-positive extents, and the comment should say so, but a
   silent NaN here would disable the Phase 11 edge clamp.
6. **Sanitise the style.** `StyleGateway.Parse(styleJson)`, which never throws and already returns a
   sanitised record. Style is never a rejection reason — D-66 clamps or replaces rather than rejecting,
   so the worst case is that a figure carries today's fixed appearance.
7. **Re-serialise from the records.** `definition.ToJson(geometry)` and `StyleGateway.ToJson(style)`.
   Comment that this is the line that makes VALID-01 true: the strings handed back are produced from
   the typed records, and the client's original text is never assigned to either field.
8. Build and return the `ValidatedFigureInput` with `definition.Name` as `Type` — the registry's own
   literal, not the caller's `typeName` string, so casing or whitespace in the input cannot reach the
   `figures.type` column.

Add a second convenience member for the draw path:

`public bool TryValidateGesture(string? typeName, CanvasPoint press, CanvasPoint cursor, string?
styleJson, out ValidatedFigureInput? result, out double x, out double y)`

which resolves the definition, calls `FromGesture`, checks `IsDrawable` on the produced geometry,
computes bounds, sanitises style, serialises from the records, and returns the placement's `X` and `Y`
alongside the validated input. It must route through the same private helper as `TryValidate` for
steps 4 through 8, so there is genuinely one validation path and not two that must agree — the project
has already paid once for a "same rule in two places, obliged to agree" arrangement (D-50, proven by a
32-case matrix test) and this milestone exists partly to remove that pattern.

**Hard rules for this file**, each worth a comment:
- No `throw` on invalid input. Every rejection is `return false`. The only exceptions this class may
  raise are `ArgumentNullException` for a null registry at construction.
- No logging, no `Console`, no exception message, and no returned string that contains any portion of
  `geometryJson`, `styleJson` or `typeName`. The application re-broadcasts figure payloads to other
  tabs, so an error path that quotes the input becomes a delivery channel.
- No `out string reason` parameter and no result enum describing why validation failed. That is a
  deliberate omission: a reason code is the first step toward a message that reaches a UI, and v1.1's
  behaviour is silent rejection.
- Set `result` to null on every rejection path.

## Flagged assumptions — unresolved edge-probe rows

The spec-less probe fallback classified the edge rows for **VALID-01, VALID-02 and VALID-03** as
`unclassified`. Per the protocol these stay **`unresolved`** and are never auto-backstopped, so they
are surfaced here as explicit flagged assumptions for the user rather than silently dropped. Each
names the reading the plan proceeds on:

- **FA-01 (VALID-01, unresolved).** The probe could not classify an edge shape for "what the client
  sent is never stored". The plan proceeds on the reading that this means *field-level* exclusion:
  the persisted `geometry` and `style` strings must be byte-identical to a fresh serialisation of the
  typed records, so any unknown key or extra value is absent. It does **not** attempt to define
  behaviour for a payload that is valid but semantically surprising (for example a rectangle whose
  extent exceeds the canvas) — local geometry is deliberately unbounded by canvas size under D-59,
  because position is a separate column. **If you intend the gateway to also bound local geometry to
  the canvas, say so — it is not currently planned.**
- **FA-02 (VALID-02, unresolved).** The probe could not classify an edge shape for "geometry the
  database no longer rejects is rejected by the application instead, preserving today's
  silent-rejection behaviour". The plan proceeds on the reading that the compatibility floor is
  exactly the set of geometries the v1.1 CHECK constraints accepted (group F pins this), and the
  ceiling adds only the collinear-triangle rejection (PA-22), which no v1.1 gesture could produce.
  **If you want the ceiling to be exactly the v1.1 constraint set with no additions, the
  collinear-triangle rule is the one line to remove.**
- **FA-03 (VALID-03, unresolved).** The probe could not classify an edge shape for the style bounds.
  The plan proceeds on the reading that the bounds are exactly D-66's — stroke width clamped to
  0.5-64, opacity clamped to 0-1, colours matched against `^#[0-9A-Fa-f]{6}$` — with out-of-range and
  unmatched values **replaced or clamped, never rejected**, so a hostile style can never cause a
  figure to be dropped. **If you would rather a hostile style reject the whole write, that is a
  different contract and would change the return of `TryValidate`.**

## Planner Assumptions

- **PA-29 — The gateway rejects geometry but never rejects style.** D-66 says style values are
  clamped or replaced; VALID-02 says geometry is rejected. Treating them asymmetrically is what the
  decisions say, and it means a hostile style degrades a figure to today's fixed appearance rather
  than losing the figure. See FA-03 for the override.
- **PA-30 — There is no failure reason, code or message.** VALID-02 requires v1.1's silent rejection.
  A reason code is the first step toward a message that reaches a UI, which would itself be a
  user-visible change. Enforced by a reflection test rather than by convention.
- **PA-31 — The stored `type` literal is the registry definition's `Name`, and a mis-cased input is
  rejected rather than normalised.** Silently repairing input is how a gateway starts accepting
  things it was built to refuse.
- **PA-32 — `TryValidateGesture` exists alongside `TryValidate`, sharing one private helper.** The
  draw path has no JSON to parse — it has a gesture — and forcing it to serialise-then-reparse just to
  reach the gateway would be a second code path in disguise. The shared helper plus the parity test is
  what keeps them from drifting.
- **PA-33 — Local geometry is not bounded by the canvas size.** Under D-59 position lives in `x`, `y`
  and shape lives in local coordinates, so a shape larger than the canvas is representable by design;
