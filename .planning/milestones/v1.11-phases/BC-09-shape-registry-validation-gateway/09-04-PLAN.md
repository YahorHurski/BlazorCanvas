<objective>
Implement the four `IShapeDefinition` types â€” `LineShape`, `TriangleShape`, `RectangleShape`,
`CircleShape` â€” and the `DefaultShapes` composition root that registers them. Each class owns its
type's parse, drawability, bounds, canonical serialisation and draw gesture, so that every
type-specific rule the v1.1 codebase spread across eight sites is reachable through one interface
(SHAPE-01). Line and triangle carry ordered point lists, so a downward- or sideways-pointing triangle
is expressed as different data, never as a new formula (SHAPE-02).

Purpose: this is the plan where the registry stops being a contract and becomes the working thing.
It is also where the v1.1 arithmetic is preserved: `FromGesture` reproduces the existing draw
behaviour exactly, so the milestone's "zero user-visible change" invariant survives the model swap.

Output: five production files and five test files, all new. Nothing is wired into the running
application â€” no `Home.razor`, no `FigureShape.razor`, no `Data/`, no schema, no migration.
</objective>

## Planner Assumptions

- **PA-19 â€” `FromGesture` reproduces v1.1 arithmetic exactly, including whole-unit rounding with
  midpoints away from zero and the circle's edge cap.** Coordinates are `numeric` in the new storage
  model, so nothing forces the rounding any more â€” but removing it would change what a gesture
  produces, and the milestone's governing invariant is that any user-observable difference from v1.1
  is a defect. Keeping the arithmetic identical also makes the equivalence proof in 09-05 exact
  rather than approximate. *Flagged for user override:* dropping the rounding is a one-line change if
  you would rather have sub-pixel-accurate gestures now.
- **PA-20 â€” A line's points are stored in DRAW order (press first), not in the v1.1 canonical swapped
  order.** Draw order is information-preserving, it is what "the point list is the source of truth"
  means, and an SVG line renders identically for either ordering â€” so it is not a user-visible
  change. The absolute endpoint set is provably identical to the v1.1 result; 09-05 asserts it.
- **PA-21 â€” Rectangle and circle reject a non-positive extent at PARSE time, not only at
  `IsDrawable`.** This is what makes `BoundsOf` total and lets the "never a negative `Bbox.W`/`Bbox.H`"
  invariant be stated unconditionally. Point coordinates, which may legitimately be negative, are not
  subject to this rule.
- **PA-22 â€” A triangle with three collinear points is not drawable.** ROADMAP success criterion 4
  names "degenerate/duplicate points" as input that must be rejected. This is no stricter than v1.1
  for anything the application can draw, because the v1.1 gesture only produced apex-top-centre
  isosceles triangles, which are collinear only when width or height is zero â€” cases v1.1 already
  rejected.
- **PA-23 â€” New shape code transcribes the v1.1 arithmetic rather than calling the v1.1 classes.**
  Phase 11 deletes those classes; a compile-time dependency would turn their removal into a rewrite.
  Equivalence is proven by the 09-05 grid test instead, which is a stronger guarantee than shared
  code because it survives the deletion. The one exception is the canvas-size constants class, which
  Phase 11 keeps and which BC-06 established must not be duplicated as literals.
- **PA-24 â€” `DefaultShapes.CreateRegistry()` returns a fresh registry every call.** 09-05 registers a
  test-only fifth shape; a static cached registry would leak it.

The full cross-cutting assumption list (PA-01 â€¦ PA-08) is in `09-01-PLAN.md`.
