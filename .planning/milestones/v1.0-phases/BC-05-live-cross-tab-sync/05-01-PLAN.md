<objective>
Build the two new types this phase's entire sync layer rests on: the D-53 message contract, and the
in-memory notifier that carries it between tabs. Prove the one guarantee that fails silently — that
a broadcast for one user never reaches another user's tab.

Purpose: this is the only genuinely new infrastructure in Phase 5. Everything after it —
`Program.cs`'s registration (05-02), `Home.razor`'s wiring (05-03), the failure path (05-04) — is
calling code. The plan is deliberately isolated: it touches no component, needs no database, needs no
browser, and can be proven entirely by an in-memory test run. Both files are new, so nothing here can
regress a shipped Phase 1-4 behaviour.

Output: `BlazorCanvas.Sync` — one record, one service, and the tests that prove the isolation and the
unsubscribe contract.
</objective>

    Create `src/BlazorCanvas/Sync/SyncMessage.cs` in namespace `BlazorCanvas.Sync`.

    Declare one `public sealed record SyncMessage` with the positional parameters, in this order:
    `string Kind`, `Guid Sender`, `int Id`, `string? Type`, `int? X1`, `int? Y1`, `int? X2`, `int? Y2`.
    A `sealed record` matches every other data-shape type in this codebase (`Figure`, `Box`) — D-22's
    stated preference is "one flat C# record, no class hierarchy, no null-checks", and D-53's four
    kinds are close enough in shape that four subtypes would be ceremony. The nullable geometry fields
    are what let one record cover all four kinds.

    `Sender` is a `Guid`, a value type — NOT a `string` and NOT `object`. The echo filter (D-11 rule 3)
    compares it with `==`, and a reference-typed sender would make that comparison a reference check
    that silently never matches, disabling the echo filter without any error.

    Add exactly four `public static` factory methods, one per D-53 kind, and no others:

    - `Draw(Figure f, Guid sender)` — `Kind` is the literal `"draw"`, and it carries `f.Id`, `f.Type`,
      and all four of `f.X1`/`f.Y1`/`f.X2`/`f.Y2`. This is the ONLY kind that may create a figure on
      the receiving side (D-53), and it takes a `Figure` rather than an id because it is only ever
      called after the INSERT has returned a database-assigned id (D-39).
    - `Move(int id, Box box, Guid sender)` — `Kind` is `"move"`, carrying the four box coordinates and
      passing **null for `Type`**. D-53: "`move` carries no `type` — a figure's type never changes, and
      the receiver already knows it." Take a `Box`, not four loose ints: the caller always has one
      (`dragCurrentBox`), and a `Box` cannot have its coordinates transposed at the call site.
    - `Delete(int id, Guid sender)` — `Kind` is `"delete"`, with null for `Type` and null for all four
      coordinates. D-53's payload for this kind is `{ kind, sender, id }` and nothing else.
    - `Rollback(int id, Box originalBox, Guid sender)` — `Kind` is `"rollback"`, carrying
      `originalBox`'s four coordinates and null `Type`. Its parameter is named `originalBox`, not
      `box`, because D-52 is explicit that the payload is the coordinates retained from *before* the
      drag, never the failed new position.

    The four `Kind` literals are lowercase and must be exactly `"draw"`, `"move"`, `"delete"`,
    `"rollback"`. The receiving switch (plan 05-03) matches on these strings.

    Do not add a fifth factory and do not add a `drop` kind. D-53 is explicit: there is no separate
    drop kind — a drag's final position is simply the last `move`, guaranteed sent by D-47's trailing
    edge, followed by silence.

    Do not add a timestamp, a sequence number, or a version field. Nothing in the ADR set reads one:
    ordering is guaranteed by the one-mouse premise (D-11), and z-order is `figures.id` (D-39).

    Give the record a class-level XML-doc in this codebase's established style (see `Figure.cs`):
    state that this is D-53's canonical contract and that D-53 supersedes D-11/D-22/D-40's partial
    descriptions; state that `Sender` exists solely for the echo filter; and state the one rule that
    is invisible from the type itself — that `move` and `rollback` are applied UPDATE-ONLY by the
    receiver (D-40), so an unknown id means the message is ignored entirely, never inserted. Add a
    short doc on `Move` recording why `Type` is null there.

  <action>
    Create `src/BlazorCanvas/Sync/CanvasSyncNotifier.cs` in namespace `BlazorCanvas.Sync`, declaring
    `public sealed class CanvasSyncNotifier`.

    STORAGE. One private readonly field:
    `ConcurrentDictionary<int, ConcurrentDictionary<Guid, Action<SyncMessage>>>` — the outer key is
    the `user_id` (D-11 rule 7), the inner key is a per-subscription `Guid`, and the value is the
    subscriber's handler. Initialise it inline with `new()`. The inner dictionary is keyed by a
    subscription id rather than being a list because unsubscribing must remove exactly one
    subscription in O(1) without needing delegate equality — two tabs of the same user can hand in
    handlers that compare equal, and removing "the first match" would silently unsubscribe the wrong
    tab.

    `public IDisposable Subscribe(int userId, Action<SyncMessage> handler)` — generate a
    `Guid.NewGuid()` subscription id, get-or-add the user's inner bucket via `GetOrAdd`, store the
    handler under the subscription id, and return an `IDisposable` whose disposal removes that exact
    subscription id from that exact bucket via `TryRemove`. Returning `IDisposable` rather than `void`
    is what makes D-11 rule 1 implementable at the call site: `Home.razor`'s `Dispose()` (plan 05-03)
    holds this handle and nothing else can unsubscribe it.

    `public void Publish(int userId, SyncMessage message)` — `TryGetValue` the user's bucket; if there
    is no bucket, return immediately (publishing to a user with no open tabs is normal, not an error).
    Otherwise snapshot the bucket's values with `.ToArray()` and invoke each handler with the message.
    Take the snapshot deliberately: `ConcurrentDictionary` tolerates mutation during enumeration
    without throwing, but the snapshot keeps `Publish`'s cost independent of concurrent `Dispose()`
    calls and gives it one obvious, readable shape.

    Publish is `void`, not `async Task`. The handler's job is to marshal onto its own circuit with
    `InvokeAsync` (D-11 rule 2, implemented in plan 05-03); the publisher must never await another
    tab's render. A slow or dead subscriber must not be able to stall the acting tab's drag.

    THE `userId` KEY IS THE ONLY ISOLATION. There is no second filter anywhere downstream: the
    receiving tab applies whatever it is handed. `TryGetValue(userId, ...)` is what stops user B's
    tab from seeing user A's figures (T-05-01). `userId` always arrives as a parameter from the
    caller's `user_id` cookie claim (D-51) and this service must never discover whose canvas it is on
    by any other means — the same discipline `FigureStore`'s class doc already states.

    Nested `private sealed class Subscription(Action onDispose) : IDisposable` — primary constructor,
    matching `FigureStore`'s style. Hold the action in a nullable field and implement
    `Dispose()` as `Interlocked.Exchange(ref _onDispose, null)?.Invoke();` so a double dispose
    unsubscribes once and is otherwise a silent no-op. Blazor can call a component's `Dispose()` more
    than once in teardown races; this makes that harmless rather than a second `TryRemove` against a
    reused subscription id.

    ADD NOTHING ELSE. Specifically, do NOT add:
    - any lock, `Monitor`, `Mutex`, or `SemaphoreSlim` — D-11's one-mouse premise deletes this whole
      category by name; the `ConcurrentDictionary` is already thread-safe and adding a lock here would
      be actively regressive against a locked decision;
    - a timer, `PeriodicTimer`, background task, or polling loop — D-11 rejects timer polling by name,
      and the 50ms throttle is the caller's inline gate (D-47, plan 05-03), not this service's job;
    - the echo filter, the mid-drag discard, or the apply logic — those are per-circuit state
      (D-11 rules 3/4/5) and belong to `Home.razor`. This service must deliver every message to every
      subscriber of that user *including the sender's own tab*; the sender's tab filters its own echo
      itself. Filtering here would break the design, because the notifier does not know which
      subscriber is which tab;
    - constructor dependencies of any kind. Taking none is what lets 05-02 register this as a
      Singleton with no captive-dependency risk, and lets the tests build it with `new` and no DI
      container (D-49).
    - subscriber capping, eviction, or an "is this circuit still alive" heartbeat — D-11's Rejected
      list declines `CircuitHandler` reconnect plumbing; `Dispose()` is the whole mechanism.

    Give the class an XML-doc in this file set's established style: state that it is the DI singleton
    of D-11's Mechanism section, that it is keyed by `user_id` per D-11 rule 7 (and why — matching the
    database and the cookie claim, rather than a second identity), that the `userId` key is the whole
    of the cross-user isolation, and that it deliberately implements only rules 1 and 7 of
    `CONSTRAINT-sync-core` while the other seven are per-circuit state owned by the caller. Document
    on `Publish` that it delivers to the sender's own tab too, and that this is intentional.
  </action>
