<objective>
Prove the phase goal: **the database itself refuses an illegal row.**

Purpose: ROADMAP success criterion 3 is a claim about PostgreSQL, not about C#. As `01-CONTEXT.md` puts it: *"a test that only exercises the C# guard does not prove this."* So these tests deliberately **bypass the geometry core** and INSERT raw coordinates through the DbContext, then assert the failure comes back as a PostgreSQL `check_violation` naming the specific constraint.

And then the decisive test of the whole phase: **the guard and the CHECKs agree.** D-50 says the per-type min-size guard mirrors the CHECK constraints *exactly*, so the app can never write a row the database would refuse. Plan 01-02 built the guard. Plan 01-03 built the CHECKs. Neither can prove the other right. This plan runs both over the same matrix of cases and asserts `IsDrawable(t, b)` is true **if and only if** the INSERT succeeds. If they disagree, this is where it surfaces — rather than in Phase 3, as a figure that silently vanishes when the user draws it.

Output: `tests/BlazorCanvas.Tests/Database/` — four test files, no production code.
</objective>

  <action>
    Create `CheckConstraintTests`. Every test **constructs a `Figure` entity with literal coordinates and calls `SaveChanges` directly.** It must **not** call `MinSizeGuard`, `Normalisation`, or any other geometry-core method first. The whole point is that the rejection comes from PostgreSQL. Routing through the guard would prove only that the guard agrees with itself.

    Assert on the PostgreSQL error, not merely on "an exception was thrown": unwrap the `DbUpdateException` to the inner `PostgresException`, assert its `SqlState` is `23514` (check_violation), and assert its `ConstraintName` is the specific constraint the case targets. Naming the constraint is what turns "something went wrong" into "`circle_is_a_circle` did its job".

    **`circle_is_a_circle` must reject** (D-22 — a circle is stored as the square it is inscribed in):
    - `('circle', 0,0, 10,8)` — not square (would render as an oval).
    - `('circle', 0,0, 9,9)` — square but **odd-sided**; the radius would not be an integer and `r = (x2-x1)/2` would truncate, drifting the circle a half-pixel per round-trip.
    - `('circle', 10,10, 10,10)` — zero radius.
    - `('circle', 10,10, 0,0)` — un-normalised / negative side.

    **`box_is_a_box` must reject:**
    - `('rectangle', 10,10, 90,10)` — zero height.
    - `('rectangle', 10,10, 10,90)` — zero width.
    - `('rectangle', 10,10, 10,10)` — zero area.
    - the same three cases for `('triangle', …)` — a triangle is derived from the same box (D-21).

    **`line_is_a_line` must reject:**
    - `('line', 10,10, 10,10)` — both endpoints identical, zero length.
    - `('line', 90,10, 10,10)` — un-normalised (x2 < x1); the CHECK requires `x2 >= x1`.

    **`line_is_a_line` must ACCEPT** — these are legal figures and a test that rejects them has misread D-50:
    - `('line', 10,10, 90,10)` — a **horizontal** line (zero height). It must INSERT successfully.
    - `('line', 10,10, 10,90)` — a **vertical** line (zero width). It must INSERT successfully.
    - `('line', 0,100, 100,0)` — the up-and-right diagonal (y2 < y1 — legal for a line). It must INSERT successfully.

    **`figures_type_is_known` must reject** an unknown type string such as `('oval', 0,0, 10,10)`, and must reject a PascalCase `('Circle', 0,0, 10,10)` — the literal an accidental `Enum.ToString()` mapping would produce (D-46).

    **Valid figures must INSERT successfully** — the suite must prove the CHECKs are not simply refusing everything: a well-formed circle `('circle', 0,0, 10,10)`, rectangle `('rectangle', 10,10, 90,90)`, triangle `('triangle', 10,10, 90,90)`.
  </action>

    Create `GuardMirrorsChecksTests`. **This is the decisive test of the phase.** D-50 says the per-type min-size guard mirrors the CHECK constraints *exactly*, so the app can never write a row the database would refuse. Plan 01-02 wrote the guard; plan 01-03 wrote the CHECKs; nothing so far has checked that they say the same thing.

At minimum, for each of the four types, include: a well-formed box; a zero-width box; a zero-height box; a zero-area box; an odd-sided square; a non-square box; a box with x2 < x1; and (for lines) an up-and-right diagonal with y2 < y1.

    For **every** case, run both halves and assert they agree:
    - the C# half: `MinSizeGuard.IsDrawable(type, box)`
    - the database half: attempt the INSERT (using `FigureTypeNames.ToDbValue(type)` for the `type` column) and record whether it succeeded or raised a `23514` check_violation

    Assert `IsDrawable(type, box)` is `true` **if and only if** the INSERT succeeded. Report any disagreement with the offending type, box, guard verdict and database verdict, so the failure message tells you which half is wrong.

    Two directions of failure, both fatal, and the test must catch both:
    - **Guard says yes, database says no** → the app writes a row PostgreSQL rejects. In Phase 3 the user draws a figure, it appears in the preview, and the save throws. This is the failure D-50 exists to prevent.
    - **Guard says no, database says yes** → the app silently discards a figure PostgreSQL would happily have stored. A user draws a perfectly legal horizontal line and nothing happens. Nothing errors. This is the failure D-23's retracted shared guard actually produced.

    If a disagreement is found, **resolve it in favour of `CONSTRAINT-schema`** — the DDL is authoritative over both the guard and the DbContext.

    **Type-literal round-trip (D-46):** for each of the four `FigureType` members, assert an INSERT with `type = FigureTypeNames.ToDbValue(member)` succeeds against the live `figures_type_is_known` CHECK. This closes the deliberate seam left in plan 01-03: the geometry core owns the enum and the literals; the DbContext owns the CHECK; this assertion proves they match, rather than trusting that they do.

Then, as a scripted step (this is a shell action, not an xUnit assertion): `docker compose down` — **without** `-v` — followed by `docker compose up -d`. Plan 01-01 proved the volume holds with a scratch table; this proves it holds for the real thing.
