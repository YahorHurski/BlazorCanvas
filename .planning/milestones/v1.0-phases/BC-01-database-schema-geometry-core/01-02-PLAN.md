<objective>
Write the pure C# geometry core and prove it with the three mandated tests. This is the maths the database's CHECK constraints mirror — and the three tests here are exactly the three *silent* failure modes the phase exists to guard.

Purpose: ROADMAP success criterion 4 lives entirely in this plan. Each of the three silent failure modes is a bug that produces a *plausible-looking wrong picture* rather than an error:
- **Line normalisation (D-41)** — sort a line's axes independently and (0,100)→(100,0) silently becomes (0,0)→(100,100): the **opposite diagonal**. The line still draws. It is just the wrong line.
- **The clamp (D-36)** — clamp `x2` independently of `x1` and the figure *resizes* while the user thinks they are moving it. For a rectangle this fails completely silently.
- **The circle round-trip (D-22)** — get the inscribed-square encoding wrong and the radius drifts a pixel per drag. Nobody notices until the tenth drag.

Output: `src/BlazorCanvas/Geometry/` (pure C#, no Blazor, no EF, no I/O — D-49) and the TEST-01 test suite.

Task order is interface-first: Task 1 defines the types every later task consumes; Task 2's `ClampMove` is what Task 3's round-trip test uses to prove translation preserves the radius.
</objective>

    Write these tests FIRST — this is **mandated test 1 of TEST-01**, and it must fail before `Movement` exists.

    **Per-axis independence (the headline assertion):**
    - A 100×100 box flush against the right edge, `(1180,300)→(1280,400)`, given `dx = +50, dy = -30`, moves to `(1180,270)→(1280,370)`. The x delta is clipped to 0; the y delta passes through **at its full −30**. `dx'` must never read `y`. A figure pinned to the right edge still slides freely up and down.
    - The mirror case: a box flush against the bottom edge given `dx = +40, dy = +60` moves 40 in x and 0 in y.

    **Inclusive bounds, 0..1280 × 0..720 (D-19):**
    - A box at `(1200,600)→(1280,720)` — already touching both maxima — with `dx = +1, dy = +1` does not move at all. X2 == 1280 and Y2 == 720 are **legal, reachable positions**, not overflow.
    - A box at `(1180,600)→(1280,700)` with `dy = +100` lands exactly at Y2 == 720.
    - A box at `(0,0)→(50,50)` with `dx = -1, dy = -1` does not move. X1 == 0 and Y1 == 0 are legal.
    - A box at `(10,10)→(60,60)` with `dx = -100` lands exactly at X1 == 0 (clipped from −100 to −10, not rejected).

    **Shape is invariant — a move is a move, never a resize:**
    - For every case above, and for a table of random deltas including ones far outside the canvas, `Width` and `Height` are identical before and after. Clamping `x2` independently of `x1` would break this and is exactly what must not happen.

    **A line whose Y1 > Y2 (legal after D-41 normalisation) clamps correctly:**
    - The line `(0,700)→(100,100)` (Y1 > Y2 — legal for a line) with `dy = +100` clamps against the *bounding box*, not against raw Y1/Y2: `by2 = 700`, so `dy'` is clipped to +20 and the line lands at `(0,720)→(100,120)`. This is why the formula computes min/max rather than trusting Y1 ≤ Y2.
  </behavior>

  <action>
    Create `src/BlazorCanvas/Geometry/Movement.cs` — pure C#, `System.*` usings only.

    `ClampDelta(int v, int lo, int hi)` → `Math.Min(Math.Max(v, lo), hi)`.

    `ClampMove(Box b, int dx, int dy)` → transcribe `CONSTRAINT-clamp` exactly, with `W = CanvasBounds.Width` (1280) and `H = CanvasBounds.Height` (720):
    1. Compute the bounding box: `bx1 = min(X1,X2)`, `by1 = min(Y1,Y2)`, `bx2 = max(X1,X2)`, `by2 = max(Y1,Y2)`. **Do not skip the min/max** — a normalised *line* may have Y1 > Y2 (D-41), and trusting Y1 ≤ Y2 here silently mis-clamps every down-and-right line.
    2. Clamp the **delta**: `dx' = ClampDelta(dx, -bx1, W - bx2)` and `dy' = ClampDelta(dy, -by1, H - by2)`.
    3. Translate all four coordinates uniformly: `X1 += dx'`, `Y1 += dy'`, `X2 += dx'`, `Y2 += dy'`. Return a new `Box`.

    Two invariants this shape of the code enforces for free, and which the tests pin down:
    - **Never clamp X2/Y2 independently of X1/Y1.** That resizes the figure instead of moving it. For a rectangle it fails *silently*; for a circle it now fails loudly against `circle_is_a_circle` (thanks to the D-22 reversal) — but relying on the CHECK to catch a bug the maths should not have is not the design.
    - **Per-axis independence.** `dx'` is computed from x-terms only and `dy'` from y-terms only. Neither expression may reference the other axis. This is what makes a figure *slide along* the wall instead of sticking to it, and it is the difference between the app feeling right and feeling broken.

    Bounds are **INCLUSIVE**: `W - bx2` permits `bx2` to reach exactly 1280, and `H - by2` permits exactly 720. An off-by-one that makes the domain 0..1279 is a real bug and the tests above will catch it.

    Note the ordering rule this belongs to (D-36, for Phases 4–5, not implemented here): **clamp → render → broadcast.** A raw, unclamped position is never broadcast. Nothing in this plan broadcasts anything; the note exists so the seam is not misused later.
  </action>
