<objective>
Close the third Critical defect (01-REVIEW.md CR-03, confirmed by direct code read in 01-VERIFICATION.md): `CanvasDbContextFactory.CreateDbContext` falls back to a hardcoded `Host=localhost;Port=5432;...` connection string whenever `ConnectionStrings:Canvas` cannot be found. On this machine port 5432 is the unrelated native `postgresql-x64-18` Windows service (the exact conflict that forced the D-27 port move to 5433), so `dotnet ef` invoked from the repository root — where neither optional appsettings file is on the current directory — would silently apply migration DDL to the wrong server.

Purpose: A design-time factory that cannot find its configuration must fail loudly, not invent one. This protects the schema/migrations tooling path (DATA-02) from silently corrupting an unintended database on the same host.

Output: A `CanvasDbContextFactory` that throws an actionable `InvalidOperationException` (telling the developer to run `dotnet ef` from `src/BlazorCanvas/` or set the `ConnectionStrings__Canvas` environment variable, and warning that port 5432 is a different server) instead of falling back to a hardcoded literal.

Scope fence: This is design-time-only. Do NOT change `Program.cs` — its separate runtime DI registration correctly reads `Port=5433` from `appsettings.Development.json` and must be left exactly as is. Do NOT change `docker-compose.yml` or `appsettings.*.json`.
</objective>
