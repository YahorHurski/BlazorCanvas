# Phase 4 — UI Design Contract

> Visual and interaction contract for **Select, Drag & Delete**. Generated by gsd-ui-researcher.
> There is no `CONTEXT.md` for this phase and this agent has no interactive question tool — every
> decision below is traced to a Phase 4 ROADMAP Success Criterion, a locked `docs/DECISIONS.md`
> entry (cited by D-XX), `03-UI-SPEC.md`'s inherited baseline, or `04-RESEARCH.md`. Where none of
> those settles a value, it was recorded rather than decided here — and then put to the user. All
> such values are now ratified under **Resolved Questions** (user sign-off 2026-07-16).

## Scope — this is a DELTA spec

Phase 4 adds three verbs (select, drag, delete) to the exact same screen `03-UI-SPEC.md` already
specifies (the 48px toolbar + the 1280×720 canvas). **This document does not restate that
baseline** — it is fully inherited by reference. Only what Phase 4 introduces or changes is
specified below:

1. The selected-figure red outline (new **state**, existing token).
2. Drag cursor affordance (new interaction states — `grab`/`grabbing`, confirmed 2026-07-16).
3. The Delete button's disabled→enabled *behavioral* transition (the visual states already exist
   in `03-UI-SPEC.md` and are already implemented in `Toolbar.razor`/`Toolbar.razor.css` — Phase 4
   only wires the click behavior; there is no new visual token).
4. Drag feedback while a figure is in motion (no new token — states explicitly what does *not*
   change).

Where this document is silent on something (spacing scale, typography, toolbar layout, canvas
chrome, color palette, login page, save-failure modal), **`03-UI-SPEC.md` remains the controlling
document.** Do not re-derive or restyle any of it here.

## Color

**No new color tokens.** This phase's one visual delta — the selected-figure outline — reuses the
`Destructive` token (`#B91C1C`) that `03-UI-SPEC.md` already declared and reserved in part for
exactly this purpose ("the selected figure's outline (2px stroke, D-58's 'red')"). No new hex value
is introduced.

| Role | Value | Usage (Phase 4 delta only) |
|------|-------|------|
| Destructive | `#B91C1C` (inherited, unchanged) | **Now activated:** the selected figure's 2px outline (ROADMAP SC1, D-31, D-58). Already also reserved for the Delete button's hover/focus icon tint (`03-UI-SPEC.md`, unchanged, now behaviorally live since Delete is clickable) and the login/save-failure error text (unrelated to this phase). |
| Accent | `#1D4ED8` (inherited, unchanged) | No new usage this phase. |

**AA contrast check (new, since this state did not exist before Phase 4):** `#B91C1C` on `#FFFFFF`
(the figure's own white fill, D-38) is a 3px-equivalent-width UI-component boundary — WCAG 1.4.11
non-text-contrast requires 3:1. `#B91C1C` vs `#FFFFFF` measures **5.9:1** — passes with margin. Against
the grey page background (`#DCE0E5`, visible where a figure sits near the canvas edge) it measures
**4.9:1** — also passes. No new supporting token is required.

## Selection State Spec (new)

- Exactly one figure may be selected at a time (D-31). The `Selected` figure renders with
  `stroke="#B91C1C"` at the same 2px width as the default; every other figure keeps
  `stroke="#000000"`. Fill, opacity, and all other rendering stay identical to the unselected
  state — only the stroke color token changes. (See `04-RESEARCH.md`'s `FigureShape.razor` code
  example: `StrokeColor => Selected ? "#B91C1C" : "#000000"`.)
- Clicking empty canvas (i.e. `OnPointerDown` firing with no figure hit) clears selection.
- A click on overlapping figures selects the topmost — resolved for free by native SVG hit-testing
  and DOM paint order (`figures.id` ascending, D-39); no z-order UI or resolver is built or needed.
- Selection state is rendered from a per-circuit C# field (`selectedId`) — never a database column,
  never a broadcast payload. **A user's other open tab or monitor shows no selection indication for
  a figure selected elsewhere** (D-15, D-31, FIG-02). This is not a bug to fix in a later phase; it
  is the locked design.

**Accessibility note — color alone as the selection signal (accepted, not open):** `docs/DECISIONS.md`
D-31 explicitly considered and **rejected** both alternative non-color encodings (a thicker stroke —
"reads as a different-looking figure rather than a selected one"; a dashed bounding box — "looks
like it implies resize handles" which the app has none of, D-04). The red-outline-alone signal is
therefore a **locked, deliberate accepted cost against WCAA 1.4.1 (Use of Color)**, not a gap this
spec can close by adding a redundant cue — doing so would contradict D-31's own rejected-alternatives
reasoning. Record this as a known, conscious tradeoff; do not "fix" it in the executor.

## Drag Interaction Spec (new)

- **What moves, and how it looks:** the dragged figure is the real figure, rendered live at its
  current clamped position on every `pointermove` — full opacity throughout (`fill-opacity`/
  `stroke-opacity` = `1`, same as any committed figure). **There is no separate translucent drag
  preview / ghost shape.** This differs deliberately from the *draw* gesture's 70%-opacity preview
  (D-35, `03-UI-SPEC.md` Canvas & Cursor Spec) — a drag moves something that already exists; a draw
  gesture builds something that does not yet exist. Conflating the two treatments would visually
  suggest the dragged figure isn't "real" yet, which is false (`04-RESEARCH.md` Anti-Patterns).
- **Edge behavior:** the figure's edge-clamp is per-axis independent (`Movement.ClampMove`, D-36) —
  visually this means a figure pinned against one edge keeps sliding freely along the perpendicular
  axis, never "sticking." No visual affordance (no highlight, no edge-glow) marks the pinned state;
  the clamp itself is the only feedback, consistent with `03-UI-SPEC.md`'s unobtrusive "drafting
  tool" direction.
- **Drag interruption (release outside window / Alt-Tab):** the figure simply stays exactly where
  it was last rendered — its clamped position at the moment of interruption. No animation, no
  snap-back, no flash. "Nothing jumps" (ROADMAP SC4) means literally no additional visual treatment
  is applied on commit; the already-rendered position **is** the committed position.
- **Stays selected after drop:** the red outline persists after a drag completes (D-48) — this is
  the same Selection State Spec above, simply not cleared by a successful drag.

## Cursor Spec — RESOLVED: Option 2 confirmed by user 2026-07-16

**Decision: implement Option 2 below** (`grab`/`grabbing`). This was not settled by any locked
decision, so it was put to the user and ratified — see Resolved Questions §1. Option 1 is retained
below only to record what was rejected and why the choice was real.

`docs/DECISIONS.md` does not mandate a `grab`/`grabbing` cursor affordance anywhere, and no ROADMAP
Phase 4 Success Criterion names one. `04-RESEARCH.md` (Open Question 3) flags this explicitly:
`03-UI-SPEC.md`'s Canvas & Cursor Spec sketched the *idea* (`.tool-pointer svg > * { cursor: grab; }`)
before this codebase's actual class names existed. Verified against the current on-disk CSS
(`Home.razor.css`), the real selectors are `.canvas-surface` (root) and `.canvas-surface.shape-armed`
(crosshair state) — there is no `.tool-pointer` class anywhere in the shipped code.

**Candidate options:**
1. **No cursor change at all when the pointer tool is armed** (cursor stays `default` everywhere,
   as it already is per the inherited `.canvas-surface` rule). Zero new CSS. Fully satisfies every
   ROADMAP Success Criterion (none of the five mentions cursor feedback).
2. **`grab`/`grabbing` affordance** (recommended by `04-RESEARCH.md` as "cosmetic only, safe to leave
   to implementation-time discretion"), scoped as:
   ```css
   .canvas-surface:not(.shape-armed) > :is(rect, line, circle, polygon):hover {
       cursor: grab;
   }
   .canvas-surface.is-dragging {
       cursor: grabbing;
   }
   ```
   This requires the executor to add one new toggled class (`is-dragging`) to the canvas root for
   the duration of an active drag, wired from the same `dragging` boolean already planned in
   `04-RESEARCH.md`'s architecture.

**Recommendation:** Option 2, because it matches ordinary drawing-tool conventions (Figma, etc.) and
costs one CSS rule plus one class toggle already implied by the drag state machine — but this is a
**recommendation, not a decision.** Neither option is traceable to a ROADMAP criterion or a D-XX
entry. Flagged for explicit user sign-off before the planner locks it in; if no answer is given, the
planner should treat Option 2 as the default per `04-RESEARCH.md`'s own framing ("safe to leave to
implementation-time discretion").

## Delete Button State Spec

**No new visual token.** `03-UI-SPEC.md`'s Toolbar Component Spec already fully specifies both
states, and both are **already implemented** in the shipped `Toolbar.razor` / `Toolbar.razor.css`
(read directly this session):

| State | Treatment (already shipped, unchanged) |
|-------|-------|
| Disabled (nothing selected) | native `disabled` attribute (`disabled="@(!DeleteEnabled)"`), icon color `#9CA3AF` (`disabled.grey`), `cursor: not-allowed`, no hover/focus treatment (native `disabled` already suppresses both) |
| Enabled, hover/focus (something selected) | icon color → `#B91C1C` (Destructive), background follows the normal `hover.tint` (`#CBD1D8`) |

**What Phase 4 actually changes:** today `DeleteEnabled` is always `false` (Phase 3 never sets a
selection) and the button has no `@onclick`. Phase 4's delta is purely behavioral — wire
`DeleteEnabled="selectedId.HasValue"` and add the `OnDelete` click handler (per `04-RESEARCH.md`'s
`HandleDeleteAsync`). **No CSS changes, no new markup, no new visual state** are required; the
existing disabled/enabled treatment already satisfies ROADMAP SC5 ("greyed out until something is
selected") as written and as shipped.

## Accessibility

Inherits `03-UI-SPEC.md`'s Accessibility section in full (focus-visible ring, disabled semantics,
`aria-pressed` on tool buttons, contrast requirements, 40×40 hit targets). This phase adds:

- **Delete button disabled state must be a real `disabled` attribute** — already true in the shipped
  `Toolbar.razor` (`disabled="@(!DeleteEnabled)"`), not merely a CSS class. This correctly removes it
  from the tab order and blocks both pointer and keyboard activation with zero JS, satisfying the
  behavioral (not merely cosmetic) framing of ROADMAP SC5.
- **Selected-figure signal is color-only** — see the Accessibility note under Selection State Spec
  above. This is a locked, conscious tradeoff (D-31's explicit rejection of alternative encodings),
  not a gap for this phase to close.
- **No new keyboard interaction is added, and none should be.** There is explicitly no Delete-key
  handler (D-33, ROADMAP SC5) — do not add `tabindex`/`@onkeydown` to the canvas or any figure as a
  side effect of this phase's work; that would reopen the exact focus-management problem D-33 closed.
- **Silent deletion of an already-gone figure** (a stale-tab zero-row DELETE) surfaces no message,
  matching D-10's established pattern — this is expected staleness, not an error state, and must not
  be treated as one (`04-RESEARCH.md` Pitfall 4).

## Resolved Questions

These were judgment calls this document could not settle from a ROADMAP Success Criterion, a locked
`docs/DECISIONS.md` entry, `03-UI-SPEC.md`'s baseline, or `04-RESEARCH.md`. None was silently
decided. **All three were put to the user and answered on 2026-07-16 (`/gsd-ui-phase 4`); each is
now CONFIRMED and carries no `[ASSUMED]` marker.** The planner must treat these as locked.

1. **Cursor affordance for the pointer tool (`grab`/`grabbing`).** **[CONFIRMED 2026-07-16: Option
   2]** — the `grab`/`grabbing` affordance, scoped exactly as the Cursor Spec section above
   specifies. The executor must add the toggled `is-dragging` class to the canvas root for the
   duration of an active drag, wired from the `dragging` boolean in `04-RESEARCH.md`'s architecture.
   Note the `.tool-pointer` selector sketched in `03-UI-SPEC.md` does **not** exist in shipped code;
   use the real `.canvas-surface` / `.shape-armed` selectors given above.

2. **No confirmation dialog on Delete.** **[CONFIRMED 2026-07-16: no confirmation, immediate delete
   on click]** — carried forward from `03-UI-SPEC.md`'s Open Question 3 and now explicitly ratified,
   not merely inherited. The user was shown the full cost before deciding: D-04 excludes undo/redo,
   so there is **no safety net** — a misclick permanently destroys a figure and its row. Accepted as
   consistent with the MinVP guiding principle. Do **not** add a confirmation step, and do not treat
   its absence as an oversight to be "fixed" during execution.

3. **3px click-vs-drag distance formula.** **[CONFIRMED 2026-07-16: Euclidean, `sqrt(dx²+dy²) >= 3`]**
   — D-48's text ("move less than 3 px") does not specify a formula; Euclidean was chosen over
   per-axis/Chebyshev to match the distance formula `DrawGesture.Build` already uses for the circle
   radius, keeping the app to a single consistent "distance" concept. The threshold is therefore
   radial — identical in every direction.
