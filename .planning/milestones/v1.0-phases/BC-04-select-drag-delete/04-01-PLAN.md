  <action>
    Add two public methods to `FigureStore`.

    `public async Task<int> UpdateAsync(int userId, int figureId, Box box)` — open a context from the
    factory, then a single query: filter `db.Figures` on `f.Id == figureId && f.UserId == userId`, and
    feed that filtered query straight into `ExecuteUpdateAsync`, whose setters assign the four
    coordinate properties from the box: `X1` from `box.X1`, `Y1` from `box.Y1`, `X2` from `box.X2`,
    `Y2` from `box.Y2`. Return the result of `ExecuteUpdateAsync` directly — it is the affected-row
    count. Set no other property: a move translates coordinates and nothing else. `Type` never
    changes on a move, and `UserId` must never be assignable from a write path at all.

    `public async Task<int> DeleteAsync(int userId, int figureId)` — same context, same
    `f.Id == figureId && f.UserId == userId` filter, into `ExecuteDeleteAsync`. Return its result.

    THE OWNER FILTER IS THE POINT. Both filters name BOTH columns. Filtering on the figure id alone
    is the one concrete way this phase could let user A move or delete user B's figure: `figures` has
    no row-level security policy, so this LINQ `Where` — which EF Core translates verbatim into the
    SQL `WHERE` — is the only thing standing between the two users. A foreign figure simply fails to
    match, 0 rows are affected, and the caller's existing zero-row branch (plan 04-03) handles that
    as ordinary staleness. The hole closes with no special case and no ownership `if` anywhere.
    `userId` always arrives as a parameter from the caller's cookie claim; this store must never
    discover whose canvas it is on by any other means, exactly as its class doc already states for
    `InsertAsync`.

    THESE TWO APIS, NOT TRACKED SAVES. Do not load the entity and mutate it, and do not call
    `SaveChangesAsync` in either method. `ExecuteUpdateAsync`/`ExecuteDeleteAsync` execute immediately,
    bypass the change tracker completely, return the affected-row count, and — this is the reason
    D-10 names them specifically — never throw when zero rows match. The tracked alternative throws
    `DbUpdateConcurrencyException` on a zero-row update even with no concurrency token configured,
    which is an unhandled crash of that tab's circuit. It would also force a round-trip load before
    every write. `InsertAsync`'s existing `SaveChangesAsync` stays exactly as it is — it needs the
    tracker to read back the database-assigned id (D-39).

    Do not add exception handling to either method. A zero-row result is not a failure and needs no
    handling; a genuine database failure is D-45/D-52 and belongs to Phase 5's DATA-04, which owns the
    retry policy and the modal. Swallowing it here would leave the screen showing something the
    database does not hold — the exact lie D-52 exists to prevent.
    <!-- planner-discipline-allow: try -->
    <!-- planner-discipline-allow: catch -->
    (For the record, the two forbidden statement keywords are `try` and `catch`.)

    Do not add a concurrency token, a `rowversion`/`xmin` column, or a retry loop. D-10 rejects
    concurrency tokens by name as "provably redundant" — the affected-row count already covers the only

    Give each method an XML-doc comment in this file's established style. `UpdateAsync`'s doc should

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| circuit -> `FigureStore.UpdateAsync`/`DeleteAsync` -> PostgreSQL | `figureId` originates from a rendered figure and reaches this store across the SignalR circuit; on a crafted circuit it is fully attacker-controlled. `userId` does not — it comes from the signed cookie claim. This asymmetry is the whole security design of the phase. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-01 | Elevation of Privilege (IDOR) | `UpdateAsync` / `DeleteAsync` | high | mitigate | Both queries filter `f.Id == figureId && f.UserId == userId`. EF Core translates the `Where` into the SQL `WHERE`, so ownership is enforced by PostgreSQL rather than by an application check that a later edit could drop. A foreign id matches 0 rows, which the caller already treats as benign staleness (D-10) — the hole closes with no error path. Gated three ways: the two-column filter greps to exactly 2; tests 4 and 7 attempt the attack with a real foreign id and assert both a 0 count and an untouched/present row; and the mutation check in Task 2's acceptance criteria proves the test fails when the guard is removed. |
| T-04-02 | Tampering | `ExecuteUpdateAsync` bypassing EF validation | medium | mitigate | These APIs bypass the change tracker, so no application-side entity validation runs on the written values. The three CHECK constraints (Phase 1) remain the backstop and are unaffected by the bypass — they are enforced by PostgreSQL on every write regardless of how the SQL was produced. Reachability is further limited by construction: a move only translates an already-legal box uniformly (`Movement.ClampMove`, wired in plan 04-03), and translation preserves size, squareness and even-sidedness. Test 2 pins exactly this for the circle, where D-22's reversal makes a violation loud. |
| T-04-03 | Tampering | a write path that could reassign `user_id` | high | mitigate | `UpdateAsync`'s setters are restricted to the four coordinate properties, so no write path in the app can move a figure onto another user's canvas. Gated: `SetProperty` greps to exactly 4. |
| T-04-04 | Denial of Service | unhandled database exception killing the circuit | medium | accept | Deliberately NOT handled here. D-45/D-52 (Phase 5, DATA-04) own the retry policy, the rollback broadcast and the modal, and specify that only transient failures retry — a zero-row result is explicitly not a failure and must never be retried. Handling added in this plan would either swallow a real failure or duplicate that phase's work in the wrong layer. Accepted for this phase and gated: statement-position `try`/`catch` greps to 0, so the gap stays visible to Phase 5 rather than being silently plugged. |
| T-04-SC | Tampering | package manager installs | high | mitigate | This plan installs no NuGet, npm, pip or cargo package. `ExecuteUpdateAsync`/`ExecuteDeleteAsync` ship in `Microsoft.EntityFrameworkCore` 10.0.10, already referenced and already vetted in Phases 1-3 (they have existed since EF Core 7). 04-RESEARCH.md's Package Legitimacy Audit records "not applicable — this phase installs no new packages". Gate: `PackageReference` counts in both csproj files unchanged. No `[ASSUMED]`/`[SUS]` package exists, so no legitimacy checkpoint is required. |

D-08 (plaintext passwords) is a deliberate, locked non-security decision and is deliberately absent
from this register.
</threat_model>
