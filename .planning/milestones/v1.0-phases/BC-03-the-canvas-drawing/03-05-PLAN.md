    `private async Task OnPointerMove(PointerEventArgs e)`:
    - If `!drawing`, return.
    - THE BUTTONS GUARD: if `(e.Buttons & 1) == 0` the primary button is already up — the user
      released it outside the window or Alt-Tabbed away mid-gesture — so `await CommitAsync()` and
      return. `Buttons` is a bitmask of the buttons currently held, and bit 0 is the primary one. This
      is D-37's second rule. It is written here rather than left to Phase 4 because D-57 makes the
      draw and the drag one consistent rule, and without it an Alt-Tab mid-draw leaves the preview
      welded to the cursor — a defect in this phase's own deliverable. Phase 4 reuses the same rule
      for the drag.
    - Otherwise map the cursor with `CanvasCoordinates.FromPage(e.PageX, e.PageY)` and recompute
      `previewBox = DrawGesture.Build(previewType, pressX, pressY, cursorX, cursorY)`. `Build` clamps,
      so the shape stops growing at the boundary while the cursor keeps moving (D-29), and the circle
      arm caps the radius so it can never render as an oval (D-36). Do not add a clamp, a bounds
      check, or a `Math.Min` here — every one of them already lives inside `Build` and a second copy
      would be a second source of truth.

    local state only and is never persisted and never broadcast (D-35, D-53) — do not add it to

    Do not call `StateHasChanged` by hand. Blazor re-renders after each event handler completes;
    `InvokeAsync(StateHasChanged)` is for Phase 5's notifier callbacks arriving on a foreign thread

    text-selection drag, which changes the cursor mid-gesture and can swallow pointer events. This is
    a CSS-only fix; do not reach for a JavaScript listener or a `preventDefault` on the pointer
    handlers to solve it.

  <action>
    Add to `src/BlazorCanvas/Components/Pages/Home.razor`:

    `private async Task OnPointerUp(PointerEventArgs e)` — if `drawing`, `await CommitAsync()`.
    Do NOT check `e.Buttons` here: on a pointerup the released button is already absent from the
    bitmask, so the guard from Task 1 would read as "button up" and the check would be either
    redundant or, if inverted, would swallow every normal release.

    `private async Task OnPointerLeave(PointerEventArgs e)` — if `drawing`, `await CommitAsync()`.
    This is D-57: leaving the surface mid-draw COMMITS the figure at its clamped preview position.
    It is deliberate that there is no way to abandon a draw — Escape would need a document-level key
    listener, which needs JavaScript, which this project does not have. Do not add a cancel path, a
    modifier-key escape, or a confirmation. If the user changes their mind, they draw it and then
    delete it. It is also coherent rather than jarring: by the time the cursor leaves the canvas the
    preview is already pinned at the edge by the clamp, so nothing jumps on commit.

    `private async Task CommitAsync()`:
    1. `drawing = false;` FIRST, and capture `var type = previewType; var box = previewBox;` into
       locals immediately after — all before the first `await`. A Blazor Server circuit dispatches
       events one at a time, but an `await` yields, so a `pointerup` arriving while a `pointerleave`
       commit is in flight would otherwise INSERT the same figure twice. Setting the flag
       synchronously at the top is the whole re-entrancy guard.
    2. `if (!MinSizeGuard.IsDrawable(type, box)) return;` — a rejected draw fails SILENTLY (D-50,
       D-32). No message, no toast, no log line the user can see, no exception. A zero-size drag
       simply creates nothing. Do not add feedback: D-32 records the accepted usability cost
       deliberately. The guard mirrors the three CHECK constraints exactly, so passing it means the
       database cannot refuse the row.
    3. `var figure = await Figures.InsertAsync(userId, type, box);` then `figures.Add(figure);` — in
       that order and no other. This is D-39: the id does not exist until the INSERT completes, so
       the figure joins the view carrying the id the database assigned. Do NOT add an optimistic
       placeholder to `figures` first and patch its id afterwards, and do NOT invent a client-side id.
       Phase 5's `draw` broadcast attaches at exactly this point, immediately after the append, and
       the reason the roadmap says to build the write path in this order NOW is that an optimistic
       insert makes that broadcast impossible to fire correctly later.
       `userId` comes from the field the claim populated in `OnInitializedAsync` — never from `e`, a
       parameter, or anything else on the wire.

    There is no Save button and there is never going to be one (D-09): the INSERT is the release.

    Do not add a try/catch around the INSERT in this plan. Database-failure handling is D-45/D-52 and
    is Phase 5's DATA-04, with a specific retry policy and a specific modal; a bare catch written here
    would either swallow the failure silently — leaving the screen showing a figure the database does
    not hold, the exact lie D-52 exists to prevent — or duplicate that phase's work in the wrong
    place.
  </action>

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege (IDOR) | `CommitAsync` -> `InsertAsync` | high | mitigate | `userId` is read from the `user_id` cookie claim in `OnInitializedAsync` (plan 03-04) and is the only value ever passed to `InsertAsync`. Nothing from `PointerEventArgs`, no component parameter, and no query string can reach it. Gated: `Figures.InsertAsync(userId` greps to exactly 1, so no other call site can exist. A user therefore cannot write a figure onto another user's canvas. |
| T-03-02 | Tampering | crafted pointer coordinates | medium | mitigate | Every coordinate crosses `CanvasCoordinates.FromPage` (which rounds a hostile `double` to an `int`) and then `DrawGesture.Build`, whose first step clamps all four values into `0..1280 x 0..720` via the Phase 1 clamp. Plan 03-01's grid test proves the inside-the-canvas invariant for out-of-range inputs on all four sides. Gated here: no `Math.Min`/`Math.Max`/`Math.Clamp` and no hand-written `48` exist in the component, so no code path bypasses `Build`. An out-of-canvas or oversized figure cannot be written. |
| T-03-14 | Tampering | crafted geometry that violates a CHECK constraint | high | mitigate | A row the database refuses would surface as an unhandled `DbUpdateException` on the circuit — a crash, and (once Phase 5 lands) a screen out of sync with the database. Mitigated structurally rather than by catching: `MinSizeGuard.IsDrawable` is a literal per-type transcription of the three CHECKs (proven equivalent across a 32-case matrix by Phase 1's `GuardMirrorsChecksTests`), and it gates every INSERT. `DrawGesture`'s circle arm can only emit square, even-sided, positive boxes, so `circle_is_a_circle` is unreachable. Gated: `MinSizeGuard.IsDrawable` greps to exactly 1 and sits before the insert line. |
| T-03-15 | Tampering / double-write | `CommitAsync` re-entrancy | medium | mitigate | A `pointerup` arriving while a `pointerleave` commit is awaiting the INSERT would write the same figure twice. `drawing = false` executes synchronously before the first `await`, and `type`/`box` are captured into locals at the same point. Gated by a line-order assertion: the flag assignment precedes the first `await` in the method. |
| T-03-16 | Denial of Service | pointermove flood over SignalR | low | accept | Every pointer-move is a SignalR round-trip and each triggers a `DrawGesture.Build` and a re-render. D-07 accepts this cost explicitly and by name. No INSERT occurs on move — only on commit — so a flood cannot amplify into database writes (D-11: broadcasting is not persisting). The 50 ms throttle of D-47 governs Phase 5's broadcasts, not this phase's previews. Blazor's own circuit limits bound the damage. |
| T-03-17 | Denial of Service | unhandled `DbUpdateException` killing the circuit | medium | accept | Deliberately NOT handled in this plan. D-45/D-52 (Phase 5, DATA-04) own the retry policy, the rollback broadcast and the modal, and specify that only transient failures retry. A bare try/catch added here would swallow the failure and leave the screen showing a figure the database does not hold — precisely the lie D-52 exists to prevent. Accepted for this phase and gated: `try`/`catch` greps to 0, so the gap stays visible to Phase 5 rather than being silently plugged. |
| T-03-SC | Tampering | package manager installs | high | mitigate | This plan adds no NuGet, npm, pip, or cargo packages; it edits two existing files and calls only symbols built in waves 1 and 2. Gate: `PackageReference` counts in both csproj files unchanged. No `[ASSUMED]`/`[SUS]` package exists, so no legitimacy checkpoint is required. |
