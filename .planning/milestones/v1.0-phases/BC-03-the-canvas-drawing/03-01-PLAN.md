<objective>
Build the two pure functions the canvas needs before any markup exists: the page-to-canvas
coordinate mapping (`canvasY = PageY - 48`) and the draw-gesture composition that turns a press
point plus a cursor point into a normalised, edge-clamped `Box`.

Purpose: success criteria 3 and 4 of this phase ("previews it live... commits on release",
"drawing stops at the canvas edge... a circle never renders as an oval... a zero-size drag creates
nothing silently, but a horizontal or vertical line still draws") are almost entirely geometry, not
markup. Extracted here, they are provable by automated test. Left inside a Razor component, they
would only ever be checkable by a human squinting at a screen.

Output: `CanvasCoordinates`, `DrawGesture`, and their tests. No Blazor dependency, no new packages.
</objective>

  <behavior>
    Rectangle / triangle (corner-to-corner):
    - `Build(Rectangle, 100, 100, 300, 200)` -> `Box(100, 100, 300, 200)`.
    - Dragged up-and-left: `Build(Rectangle, 300, 200, 100, 100)` -> `Box(100, 100, 300, 200)` (axes sorted).
    - Clamped at the far corner: `Build(Rectangle, 1200, 600, 5000, 5000)` -> `Box(1200, 600, 1280, 720)`.
    - Clamped at the origin: `Build(Rectangle, 100, 100, -500, -500)` -> `Box(0, 0, 100, 100)`.
    - `Build(Triangle, ...)` returns the same Box as `Build(Rectangle, ...)` for identical inputs.

    Line (THE landmine — D-41):
    - Up-and-right: `Build(Line, 0, 100, 100, 0)` -> `Box(0, 100, 100, 0)`. It must NOT be
      `Box(0, 0, 100, 100)` — that is the opposite diagonal, and it renders without erroring.
    - Same line drawn the other way: `Build(Line, 100, 0, 0, 100)` -> `Box(0, 100, 100, 0)` — the
      whole point pair swapped, producing an identical stored line.
    - Horizontal line is legal: `Build(Line, 10, 50, 200, 50)` -> `Box(10, 50, 200, 50)`, and
      `MinSizeGuard.IsDrawable(Line, thatBox)` is `true`.
    - Vertical line is legal: `Build(Line, 50, 10, 50, 200)` -> `Box(50, 10, 50, 200)`, and
      `MinSizeGuard.IsDrawable(Line, thatBox)` is `true`.
    - Zero-length: press == cursor -> `MinSizeGuard.IsDrawable(Line, box)` is `false`.

    Circle (centre-out — D-13, stored as its inscribed square — D-22):
    - `Build(Circle, 640, 360, 740, 360)` -> `Box(540, 260, 740, 460)`; `ToCentreRadius` of it
      returns `(640, 360, 100)`.
    - Never an oval, ever: for every case, `box.Width == box.Height`.
    - Draw-clamp near an edge: `Build(Circle, 10, 360, 210, 360)` -> radius caps at 10 ->
      `Box(0, 350, 20, 370)`. Still square. (This is D-36's known, accepted consequence: pressing
      near an edge forces a tiny circle.)
    - Zero radius: press == cursor -> `MinSizeGuard.IsDrawable(Circle, box)` is `false`.
    - Every circle Box has an even side, so the `circle_is_a_circle` CHECK can never reject it.

    Invariant across everything (proves criterion 4 without a browser):
    - For all four types, over a grid of press points and cursor points that includes coordinates far
      outside the canvas on all four sides, the returned Box satisfies
      `0 <= min(X1,X2)`, `max(X1,X2) <= 1280`, `0 <= min(Y1,Y2)`, `max(Y1,Y2) <= 720`.
      (Use min/max, not raw X1/Y1: a normalised line may legally have `Y1 > Y2`.)
  </behavior>

  <action>
    Create `src/BlazorCanvas/Geometry/DrawGesture.cs` in namespace `BlazorCanvas.Geometry` as
    `public static class DrawGesture` with a single public method:

    `public static Box Build(FigureType type, int pressX, int pressY, int cursorX, int cursorY)`

    Composition order, exactly:
    1. Clamp all four inputs to the canvas first, via `Movement.ClampDelta(v, 0, CanvasBounds.Width)`
       for the x values and `Movement.ClampDelta(v, 0, CanvasBounds.Height)` for the y values. This
       is D-29 ("drawing also stops at the canvas edge") and it is why the shape stops growing while
       the cursor keeps moving. Do not use `Math.Clamp` — `Movement.ClampDelta` is the app's clamp
       and carries the `lo > hi` guard added by plan 01-05.
    2. If `type` is `FigureType.Circle`: compute the drag distance from the clamped press point to
       the clamped cursor point with `Math.Sqrt` over `double` arithmetic (cast before multiplying so
       an int overflow is impossible), pass it to `CircleEncoding.ClampDrawRadius(pressX, pressY, distance)`,
       and return `CircleEncoding.FromCentreRadius(pressX, pressY, radius)`. The press point is the
       centre (D-13); the returned square is already normalised and already even-sided, so it must
       NOT be passed through `Normalisation.Normalise` afterwards.
    3. Otherwise return `Normalisation.Normalise(type, new Box(pressX, pressY, cursorX, cursorY))`
       using the clamped values — press point first, cursor second, so the line arm's whole-point-pair
       swap sees the gesture in the order it was actually drawn.

    `Build` must NOT call `MinSizeGuard.IsDrawable` and must NOT return a nullable. The guard is the
    caller's decision (plan 03-05 checks it before the INSERT and rejects silently, D-50); a preview
    still renders for a not-yet-drawable gesture, and folding the guard in here would make that
    impossible.

    `Build` must NOT call `Movement.ClampMove` — that is the Phase 4 move clamp, not the draw clamp.

    Do not re-implement normalisation, the circle encoding, or the clamp. Every one of them already
    exists and is already tested (plans 01-02 and 01-05); the whole value of this class is that it is
    a composition and nothing else.

    Then create `tests/BlazorCanvas.Tests/Geometry/DrawGestureTests.cs` in namespace
    `BlazorCanvas.Tests.Geometry` covering every case in the `<behavior>` block above. Use
    `[Theory]`/`[InlineData]` in the style of the existing `ClampTests`/`NormalisationTests`. For the
    invariant case, use a `MemberData` grid over the four `FigureType` values crossed with press and
    cursor points drawn from at least `{-500, 0, 5, 640, 1275, 1280, 5000}` on x and
    `{-500, 0, 5, 360, 715, 720, 5000}` on y.

    D-49 scope note for the executor: D-49 caps the test project at "the three silent failure modes"
    (clamp, circle round-trip, line normalisation). These tests are those same three modes exercised
    through the draw path — they are not a fourth subject, and they go in the existing single test
    project. Do not create a second test project and do not add bUnit or any other package.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> Blazor Server circuit (SignalR) | Pointer coordinates arrive from the client. Nothing on the wire is trustworthy: a crafted circuit message can carry any `PageX`/`PageY` double, including NaN, infinities, and values far outside the canvas. `DrawGesture` is the choke point that makes them safe. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-02 | Tampering | `DrawGesture.Build` | medium | mitigate | Every one of the four coordinates is passed through `Movement.ClampDelta(v, 0, CanvasBounds.Width/Height)` as step 1, before any type dispatch. The grid test asserts the inside-the-canvas invariant for out-of-range inputs on all four sides, so an out-of-canvas Box cannot reach the INSERT in plan 03-05 and cannot trip a CHECK constraint. |
| T-03-06 | Denial of Service | `DrawGesture.Build` circle arm | low | accept | `Math.Sqrt` over a hostile `double` could in principle yield NaN, and `(int)Math.Round(NaN)` is `int.MinValue`. Accepted because the inputs are already `int` by the time they reach `Build` — `CanvasCoordinates.FromPage` rounds to `int` at the boundary, and `int.MinValue` clamps to 0 through `ClampDelta` before the distance is computed. No unbounded allocation or loop exists in this class. |
| T-03-SC | Tampering | package manager installs | high | mitigate | This plan adds no NuGet, npm, pip, or cargo packages. `IDbContextFactory` and everything else used here already ships in the referenced assemblies. Gate: the `PackageReference` count in both csproj files is unchanged (asserted in both tasks' acceptance criteria). No `[ASSUMED]`/`[SUS]` package exists, so no legitimacy checkpoint is required. |

D-08 (plaintext passwords) is a deliberate, locked non-security decision and is deliberately absent
from this register.
</threat_model>
