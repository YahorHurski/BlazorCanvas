<objective>
Enlarge the fixed drawing canvas from the old 1280 x 720 surface to **1472 x 828** (16:9) per
CANV-03, which amends locked decisions **D-19** (canvas size â€” may grow, never shrink), **D-36**
(inclusive clamp bounds 0..1472 x 0..828), **D-18** (fixed 1:1 canvas) and **D-58** (the constants
roster). Enlarging keeps every stored figure valid, so there is **no database migration**.

The production change is tiny but lives at **three literal sites**, and the real risk is **silent
edge-coverage loss** in the test suite: many geometry tests hardcode the old boundary numbers, and a
blind find/replace would leave "edge" tests silently degraded into interior tests while the suite
still runs green. This plan makes the test update a **classify-each-hit-then-edit** operation with
acceptance criteria a lazy replace would FAIL.

Explicitly OUT of scope (do not touch): no database migration, no shrink path, no schema change, no
new figure types, no selection changes (Phase 7), auth/passwords untouched. `DrawGesture.cs` and
`CircleEncoding.cs` need **no edit** â€” they already read `CanvasBounds` and enlarge automatically.

Purpose: Deliver a larger canvas that fits a maximised window on a 1920 x 1080 monitor with no
scroll, keeps every stored figure exactly where it was, and leaves a test suite whose green is
meaningful (edge tests still assert the NEW edge).
Output: Updated `CanvasBounds.cs`, `Movement.cs`, `Home.razor`, and five re-pinned geometry test files.
</objective>
