## Goal

Add nine new figure types and a toolbar that can present them, **without breaking the "four integers
= the bounding box" storage model** (D-22) that makes the generic move/clamp/sync work for free.

## The architectural finding that makes this feasible (do not lose this)

> # 🛑 REWRITTEN BY v1.11 (2026-07-21) — and this plan got EASIER, not harder
>
> The premise below — *"every new shape stores as a 4-integer bounding box"* — is obsolete. The
> storage model was replaced before this milestone started: **shape now lives in `geometry jsonb`
> in local coordinates, position in `x, y, rotation`** (D-59…D-69).
>
> **What that changes for v1.2:**
>
> - The workarounds this plan invented to fit the 4-integer model are **no longer needed.**
>   Orientation no longer has to be smuggled into the type name: `rtriangle_left`/`rtriangle_right`
>   and the four arrow types can collapse to **one type each** carrying its own points or a
>   `rotation`. Ten buttons no longer imply ten type literals.
> - The "stretchable, not regular" compromise was adopted **specifically to avoid per-shape CHECK
>   constraints**. Those constraints no longer exist (D-60), so the choice is now a pure UX call,
>   not a schema-driven one. It can stay if it is still what you want.
> - The star's inner radius, the arrow's head proportion, an n-gon's side count — all fit in
>   `geometry`/`params` with **no schema change**. They no longer have to be hard-coded ratios.
> - **The list of "decisions to amend when v1.2 starts" is largely already done** by v1.11: D-05,
>   D-21, D-22, D-41, D-46, D-50 are all superseded. What remains for v1.2 is genuinely the
>   toolbar (D-16/D-30/D-33/D-58) plus each new shape's `geometry` format.
> - **Adding a figure type is now: one C# class, one row in `figure_types`, one toolbar button.**
>   No migration, at any count of new shapes.
>
> Re-read this section against `docs/DATA-MODEL-v1.11-DRAFT.md` before planning v1.2.

*(Historical — the v1.1-era reasoning.)* Every new shape stores as a **4-integer bounding box**,
exactly like today's figures. Therefore the generic MOVE (uniform translation, no type dispatch —
D-22's crown jewel), the generic CLAMP (D-36), and all of draw/drag/delete/select/sync work on the
new shapes **for free**. Shapes that differ only by **orientation** (right-triangle left/right; the
four arrows) encode the orientation in the **type name** (the D-46 `type is text` pattern), so the
box stays 4 ints and axis-sorting normalisation (D-41) is safe.

## KEY DESIGN DECISIONS (resolved 2026-07-20)

1. **STRETCHABLE, not regular** — for star/pentagon/hexagon/arrows/right-triangle (ellipse is
   stretchable by definition). Each shape fills whatever bbox is dragged; vertices are computed as
   **fixed fractions of the box**. Consequence, accepted: a non-proportional box gives a slightly
   distorted shape (draw a roughly-square box → a roughly-regular hexagon). **This is what avoids new
   per-shape aspect-ratio CHECK constraints** — the "regular" alternative would replicate the circle's
   whole machinery (fixed-aspect CHECK + draw-clamp + guard) per shape and add silent-failure surface
   (the D-50 bug class). Rejected in favour of simplicity.
2. ~~**Star inner radius = fixed RATIO ~0.38**~~ — ✅ **SETTLED AND DELIVERED BY v1.12** at
   **0.382** (1/φ², the exact pentagram value). The reasoning below still holds for any future
   ratio-based shape: a fixed ratio, not absolute pixels, which would look inconsistent across sizes.
3. **Circle KEPT + ellipse ADDED** — purely additive. No migration, no reversal of D-13/D-22.

## Toolbar redesign

- **10 buttons:** `pointer · line · rectangle · circle · ellipse · triangle · right-triangle ·
  pentagon · hexagon · arrow` — plus the existing **delete** and **logout** (still there). This
  supersedes the six-button toolbar (D-16/D-30/D-33) and touches D-56/D-58.
- **Dynamic split-button flyout** for the two multi-variant buttons (**arrow** and **right-triangle**):
  clicking the button opens a small set of **choice buttons**; picking one **replaces the parent
  button's icon** with the chosen variant and closes the flyout; re-click to reopen/change. Arrow's
  default icon is `arrow_right`.
- **Flyout opens DOWNWARD** over the canvas's top edge (the toolbar sits at page top, y 0–48, so "up"
  is off-screen).
- **Flyout close/hide logic REUSES the v1.1 deselect trigger set:** close on pick-a-choice, on
  starting a draw, on arming another tool, and on pressing toolbar grey space. Same mental model as
  selection deselect — build it as one shared notion if practical.
- **No JavaScript is required** for any of this (Blazor toggles a C# bool + conditional render) —
  though as of v1.1 hand-authored JS is *permitted* anyway, so use it if it's genuinely simpler.
